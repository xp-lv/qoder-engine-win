# 引擎增强需求：测试执行 + 自动部署 + 交互式需求澄清

## 1. 背景与问题

当前 lxp-eng-planning（吕小鹏精力规划系统）已完成全流程构建并部署验证，暴露出三个引擎级能力缺失：

| 缺失能力 | 实际后果 | 严重度 |
|---------|---------|--------|
| 无测试执行 | 代码通过静态校验但运行时出 bug（如 auth 中间件 path strip） | 高 |
| 无自动部署 | 部署需人工手动执行 npm install / prisma migrate / build / start | 中 |
| 无交互式需求澄清 | 需求接收者只能单次转换，不能追问用户 | 低（已有 awaiting_confirmation 兜底） |

## 2. 目标

为引擎新增两类自动化角色，补齐"测试→部署"闭环：

```
当前链路：
  ... → 代码生成者 → 校验者(静态审查) → 部署文档管理者 → 综合裁决者 → [人工部署]

目标链路：
  ... → 代码生成者 → 校验者(静态审查) → 测试执行者(运行验证) → 部署文档管理者 → 综合裁决者 → 部署执行者(自动部署)
```

## 3. 需求详述

### 3.1 新增角色：测试执行者

#### 3.1.1 定位

| 属性 | 值 |
|------|-----|
| 角色名 | 测试执行者 |
| 角色类型 | standard（执行型，非 producer） |
| 确认方式 | auto（自动确认，无需人工） |
| DAG 位置 | 校验者(confirmed) → **测试执行者** → 部署文档管理者 |
| 循环回退 | 测试失败 → 回退至代码生成者（max_executions: 3） |

#### 3.1.2 职责

测试执行者在沙箱中实际运行代码，验证编译通过和核心功能可用：

| 测试维度 | 执行内容 | 验证标准 |
|---------|---------|---------|
| 依赖安装 | `npm install`（前后端各一次） | exit code 0 |
| 数据库迁移 | `npx prisma migrate dev --name init` | 生成 migration 文件 + dev.db |
| TypeScript 编译 | `npx tsc --noEmit`（前后端各一次） | 零编译错误 |
| 前端构建 | `npm run build` | dist/ 目录生成 |
| 后端启动 + Health | 启动服务 + `curl /api/health` | 返回 `{"status":"ok"}` |
| 核心 API 冒烟 | 创建/查询/更新/删除各一个 | HTTP 200 + 数据正确 |
| 鉴权验证 | 不带 token 请求应返回 401 | HTTP 401 |

#### 3.1.3 产出物

| 产出物 | 路径 | 类型 |
|--------|------|------|
| 测试报告 | `outputs/测试报告.json` | process |

#### 3.1.4 产出物格式

```json
{
  "result": {
    "verdict": "confirmed | BLOCKING",
    "summary": "7 项测试全部通过 / 2 项失败：auth 中间件 path strip / .env 未加载"
  },
  "details": [
    {
      "test": "后端 npm install",
      "status": "pass",
      "detail": "95 packages installed"
    },
    {
      "test": "后端 tsc --noEmit",
      "status": "fail",
      "detail": "src/middleware/auth.ts:13 — 类型错误",
      "fix_hint": "将 req.path === '/api/health' 改为 req.path === '/health'"
    }
  ]
}
```

#### 3.1.5 verdict 语义

| verdict | 含义 | 路由 |
|---------|------|------|
| confirmed | 全部测试通过 | → 部署文档管理者 |
| BLOCKING | 有失败项 | → 代码生成者（回退修复，max_executions: 3） |

#### 3.1.6 输入文件

| 输入 | 来源 |
|------|------|
| 前端代码 | `outputs/src/frontend/` |
| 后端代码 | `outputs/src/backend/` |
| Prisma Schema | `outputs/prisma/schema.prisma` |
| docker-compose.yml | `outputs/docker-compose.yml` |
| 校验报告 | `process/outputs/校验报告.json` |

#### 3.1.7 技术约束

- 测试执行者需要 **Bash 工具**（执行 npm / npx / curl 命令）
- 测试在**本地沙箱**中运行，不需要远程部署环境
- 每次执行使用**临时目录**或**清理已有 node_modules**，确保干净状态
- 启动测试服务后**必须关闭**（kill process），不残留进程
- 环境变量通过命令行注入（`AUTH_TOKEN=test-token DATABASE_URL=file:./test.db PORT=3999`）

### 3.2 新增角色：部署执行者

#### 3.2.1 定位

| 属性 | 值 |
|------|-----|
| 角色名 | 部署执行者 |
| 角色类型 | standard |
| 确认方式 | manual（需用户确认部署目标） |
| DAG 位置 | 综合裁决者(confirmed) → **部署执行者** → 终态 |
| 循环回退 | 无（终态角色，失败则 BLOCKING 由用户介入） |

#### 3.2.2 职责

| 部署步骤 | 执行内容 | 验证标准 |
|---------|---------|---------|
| 构建镜像 | `docker build` | 镜像构建成功 |
| 启动服务 | `docker compose up -d` | 容器健康 |
| 健康检查 | `curl /api/health`（等待 30s 超时） | 返回 ok |
| 端到端验证 | 创建目标 → 分解任务 → 查询统计 | 核心流程走通 |

#### 3.2.2 部署目标配置

部署执行者需要用户提供部署目标信息（通过 awaiting_confirmation 收集）：

```json
{
  "deploy_target": "local | docker | remote",
  "port": 3456,
  "auth_token": "用户提供的 token",
  "docker_compose_path": "outputs/docker-compose.yml"
}
```

#### 3.2.3 产出物

| 产出物 | 路径 | 类型 |
|--------|------|------|
| 部署报告 | `outputs/部署报告.json` | deliverable |

#### 3.2.4 verdict 语义

| verdict | 含义 | 路由 |
|---------|------|------|
| confirmed | 部署成功 + 端到端验证通过 | → 终态（completed） |
| deploy_failed | 部署失败 | BLOCKING（用户介入） |

### 3.3 交互式需求澄清（引擎级增强，非新角色）

#### 3.3.1 当前限制

引擎的执行模型是"角色执行中不与用户交互"。需求接收者只能单次转换用户输入，不能在执行过程中追问。

#### 3.3.2 目标能力

允许角色在执行过程中通过特定机制向用户提问：

```
需求接收者执行中 → 发现歧义 → 产出"待澄清问题" → 引擎暂停 → 用户回答 → 需求接收者继续
```

#### 3.3.3 实现方向（待定，仅列出选项）

| 方案 | 描述 | 改动范围 |
|------|------|---------|
| A. 嵌套 awaiting_confirmation | 角色产出"待澄清问题"作为临时产出物，引擎注入 confirmation | 中（引擎层面） |
| B. 角色拆分 | 将需求接收者拆分为"需求澄清者"+"需求文档撰写者" | 小（仅 app.yaml） |
| C. 暂不实现 | 维持现有 awaiting_confirmation 机制（用户通过/不通过） | 无 |

**建议：方案 C（暂不实现）**。当前 awaiting_confirmation 机制已能覆盖大部分场景（用户看到需求后说"不通过"，附反馈意见，需求接收者重新执行时读取反馈修正）。交互式澄清是锦上添花，优先级低于测试执行者和部署执行者。

## 4. DAG 变更

### 4.1 新增步骤

```
校验者(confirmed) → 测试执行者     [新增]
测试执行者(confirmed) → 部署文档管理者
测试执行者(BLOCKING) → 代码生成者   [max_executions: 3]

综合裁决者(confirmed) → 部署执行者  [新增]
部署执行者(confirmed) → 终态
```

### 4.2 修改后的完整 DAG

```
R1 需求接收者（manual）
  → R1-validate（auto）
  → R2 需求审计者（auto）
    challenged → R1（max_executions: 3）
  → R4 需求文档管理者（auto）
  → R3 架构师（auto）
  → R2' 架构审计者（auto）
    challenged → R3（max_executions: 3）
  → R5 技术文档管理者（auto）
  → R7 代码生成者（producer, auto）
  → R7-validate（auto）
  → R8 校验者（auto）
    BLOCKING → R7（max_executions: 3）
  → R10 测试执行者（auto）              [新增]
    BLOCKING → R7（max_executions: 3）
  → R6 部署文档管理者（auto）
  → R9 综合裁决者（manual）
  → R11 部署执行者（manual）            [新增]
  → 终态
```

## 5. 优先级

| 需求 | 优先级 | 理由 |
|------|--------|------|
| 测试执行者 | P0 | 立即可落地（沙箱执行），价值最高（捕获运行时 bug） |
| 部署执行者 | P1 | 需要明确部署目标（local/docker/remote），中等复杂度 |
| 交互式需求澄清 | P2 | 现有机制可兜底，非紧急 |

## 6. 验收标准

### 6.1 测试执行者

| 编号 | 标准 | 验证方式 |
|------|------|---------|
| TEST-1 | 能执行 npm install 并报告成功/失败 | 检查测试报告中 npm install 状态 |
| TEST-2 | 能执行 tsc --noEmit 并报告编译错误 | 检查测试报告中编译状态 |
| TEST-3 | 能启动后端服务并 curl health | 检查测试报告中 health 状态 |
| TEST-4 | 能执行核心 API 冒烟测试 | 检查测试报告中 API 测试结果 |
| TEST-5 | 测试失败时 verdict=BLOCKING，回退至代码生成者 | 检查 DAG 路由 |
| TEST-6 | 测试通过时 verdict=confirmed，流转至部署文档管理者 | 检查 DAG 路由 |

### 6.2 部署执行者

| 编号 | 标准 | 验证方式 |
|------|------|---------|
| DEPLOY-1 | 能执行 docker compose up 并报告成功/失败 | 检查部署报告 |
| DEPLOY-2 | 部署后能 curl health 验证服务可用 | 检查部署报告 |
| DEPLOY-3 | 部署成功后 terminal_state=completed | 检查 STATE.json |
