# 文档验证标准手册

## 目的

本文档是一致性验证者和回归测试者共用的验证标准手册。定义"文档与引擎一致"和"变更未破坏其他部分"的客观判定标准，解决核心问题：**怎么判断改对了、没改坏**。同时定义 max 耗尽兜底行为——当回退迭代达到上限后的工作流终止/放行规则。

## 适用角色

- **一致性验证者**——使用本文档的"一致性验证检查项"校验修改后的文档是否与引擎实际行为一致
- **回归测试者**——使用本文档的"回归测试检查项"校验本次变更是否破坏了文档其他部分

## 核心内容

### 1. 一致性验证检查项（一致性验证者用）

一致性验证者的校验焦点：**修改的部分是否正确**——修改后的文档描述是否与引擎实际行为一致。

| 检查编号 | 检查项 | 检查方法 | 通过条件 | 失败严重级别 |
|----------|--------|----------|----------|-------------|
| CV-01 | 描述-行为一致性 | 对比变更摘要中每处修改后的描述与引擎实际行为 | 描述与引擎行为完全吻合 | critical |
| CV-02 | 过时描述残留 | 检查修改后的章节是否仍残留引擎已变更前的旧描述 | 无过时描述残留 | critical |
| CV-03 | 新增特性覆盖 | 检查变更涉及的新特性是否已在文档中完整描述 | 新特性描述完整，无遗漏 | major |
| CV-04 | 误导性表述检查 | 检查修改后的描述是否存在模糊、歧义或误导性表述 | 描述准确无歧义 | major |
| CV-05 | 章节结构完整性 | 检查 SDK_SPEC.md 的 9 章结构是否被破坏 | 9 章结构完整，编号未变 | critical |

**一致性验证操作流程**：
```
1. 读取变更摘要（获取本次修改了哪些章节、修改前后的内容）
2. 对每处修改，逐条对照 CV-01~CV-05 检查
3. 记录每条检查的 pass/fail 和严重级别
4. 如有 fail，记录缺陷详情（期望描述 vs 实际描述）
5. 汇总裁决：
   - 无 critical/major → validated
   - 有 critical/major → defects_detected
```

### 2. 回归测试检查项（回归测试者用）

回归测试者的校验焦点：**修改是否破坏了其他部分**——本次变更是否波及了未修改章节、引用是否仍准确、格式是否仍规范。

| 检查编号 | 检查项 | 检查方法 | 通过条件 | 失败严重级别 |
|----------|--------|----------|----------|-------------|
| RT-01 | 未修改章节正确性 | 检查与修改章节有依赖关系的未修改章节是否因变更而变得不准确 | 未修改章节描述仍正确 | critical |
| RT-02 | 交叉引用准确性 | 检查文档内交叉引用（含文档同步者同步的引用）是否仍指向正确内容 | 所有交叉引用准确 | major |
| RT-03 | Markdown 语法完整性 | 检查修改涉及的文档区域是否有断裂的表格、列表、代码块 | Markdown 语法完整 | major |
| RT-04 | 章节编号连续性 | 检查修改后章节编号是否连续无跳号 | 编号连续 | major |
| RT-05 | 表格结构完整性 | 检查修改涉及的表格是否有缺行缺列 | 表格完整 | minor |

**回归测试操作流程**：
```
1. 读取文档更新报告（获取同步后的文档状态和修改清单）
2. 识别与修改章节有依赖关系的未修改章节
3. 对每个相关章节，逐条对照 RT-01~RT-05 检查
4. 记录每条检查的 pass/fail 和严重级别
5. 如有 fail，记录失败详情（哪处修改导致了此处的回归问题）
6. 汇总裁决：
   - 无 critical/major → passed
   - 有 critical/major → failed
```

### 3. 严重级别定义

| 严重级别 | 定义 | 处理方式 | 触发回退 |
|----------|------|----------|----------|
| **critical** | 文档描述与引擎行为矛盾、章节结构被破坏、未修改章节因变更而错误 | 必须回退至代码实施者 | ✅ 是 |
| **major** | 遗漏重要特性、误导性表述、交叉引用断裂、格式严重不规范 | 必须回退至代码实施者 | ✅ 是 |
| **minor** | 措辞不精确、格式小问题（如表格缺行但不影响理解） | 记录但不回退 | ❌ 否 |

### 4. max 耗尽兜底行为说明（重要）

两条回退迭代环均设 `max_executions=2`。当回退次数达到上限后，工作流不再继续回退，而是通过**兜底边**将流程导向预设的终止/放行目标。验证角色在此场景下需通过报告中的 `max_exhausted` 字段如实标注。

#### 一致性验证者 max 耗尽兜底

| 项目 | 说明 |
|------|------|
| 回退环 | 一致性验证者 → 代码实施者（when defects_detected, max_executions=2） |
| 兜底边 | 一致性验证者 → 完成（when defects_detected）——max 耗尽后触发 |
| 触发时机 | 第 3 次 defects_detected（前 2 次已用完回退配额） |
| 行为 | 工作流**终止**（导向完成节点），不传播有缺陷的文档到依赖方 |
| 报告要求 | 验证报告中设置 `max_exhausted: true`，summary 中明确标注"一致性验证最终未通过，已达到迭代上限" |
| 设计意图 | 消除无限循环风险；确保有缺陷的文档不被传播；如实报告让用户手动处理 |

#### 回归测试者 max 耗尽兜底

| 项目 | 说明 |
|------|------|
| 回退环 | 回归测试者 → 代码实施者（when failed, max_executions=2） |
| 兜底边 | 回归测试者 → 依赖同步者（when failed）——max 耗尽后触发 |
| 触发时机 | 第 3 次 failed（前 2 次已用完回退配额） |
| 行为 | 工作流**强制放行**到依赖同步者（不死锁），依赖传播仍执行 |
| 报告要求 | 回归测试报告中设置 `max_exhausted: true`，summary 中明确标注"回归测试最终未通过，已达到迭代上限" |
| 设计意图 | 消除死锁风险；即使回归测试失败，依赖传播仍有价值（让依赖方拿到最新文档）；如实报告让依赖同步者和用户知晓文档可能存在回归问题 |

#### 两个兜底边的设计差异

| 维度 | 一致性验证者兜底 | 回归测试者兜底 |
|------|-----------------|---------------|
| 目标节点 | 完成（终止流程） | 依赖同步者（继续传播） |
| 是否传播文档 | ❌ 不传播（有缺陷的文档不应传播） | ✅ 强制传播（即使有问题也比死锁好） |
| 设计理由 | 一致性验证是质量底线——未通过则不可传播 | 回归测试是增量检查——失败不阻断已有的正确变更传播 |

### 5. 验证报告标准格式

**一致性验证者报告格式**：
```
{
  "result": {
    "verdict": "<validated|defects_detected>",
    "summary": "<一句话概述>",
    "checks": [
      {
        "check_item": "CV-01 描述-行为一致性",
        "result": "pass",
        "detail": "<检查详情>",
        "severity": "n/a"
      },
      {
        "check_item": "CV-02 过时描述残留",
        "result": "fail",
        "detail": "<检查详情>",
        "severity": "critical"
      }
    ],
    "defects": [
      {
        "id": "DEFECT-001",
        "file": "SDK_SPEC.md",
        "section": "§5 执行语义",
        "severity": "critical",
        "description": "修改后仍残留旧版 max_executions 描述",
        "expected": "应描述全局计数器机制",
        "actual": "仍为单边计数描述"
      }
    ],
    "minor_notes": [],
    "max_exhausted": false
  }
}
```

**回归测试者报告格式**：
```
{
  "result": {
    "verdict": "<passed|failed>",
    "summary": "<一句话概述>",
    "checks": [
      {
        "check_item": "RT-01 未修改章节正确性",
        "result": "pass",
        "detail": "<检查详情>",
        "severity": "n/a"
      }
    ],
    "failures": [
      {
        "id": "FAIL-001",
        "file": "SDK_SPEC.md",
        "section": "§6 STATE.json",
        "severity": "critical",
        "description": "§5 新增了全局计数器，但 §6 的 STATE.json 格式表未更新",
        "related_modification": "变更摘要条目 1",
        "fix_suggestion": "在 §6 STATE.json 格式表中新增 global_execution_count 字段"
      }
    ],
    "minor_notes": [],
    "max_exhausted": false
  }
}
```

### 6. 一致性验证 vs 回归测试的职责边界

| 维度 | 一致性验证者 | 回归测试者 |
|------|-------------|-----------|
| 校验焦点 | 修改的部分是否正确 | 修改是否破坏了其他部分 |
| 检查对象 | 变更摘要中修改的章节 | 未修改但可能受影响的章节 |
| 检查项 | CV-01~CV-05 | RT-01~RT-05 |
| verdict | validated / defects_detected | passed / failed |
| 回退边 | → 代码实施者（max_executions=2） | → 代码实施者（max_executions=2） |
| max 耗尽兜底 | → 完成（终止流程，不传播） | → 依赖同步者（强制放行，继续传播） |

**关键区别**：
- 一致性验证者回答："修改后的描述对不对？"（与引擎行为一致吗）
- 回归测试者回答："修改有没有改坏别的？"（其他部分还正常吗）

两者不可互相替代——修改本身正确不代表没有破坏其他部分，没有破坏其他部分也不代表修改本身正确。

## 示例

### 正例：合格的文档修改通过双重验证

**场景**：代码实施者在 SDK_SPEC.md §5 新增了全局执行计数器描述。

**一致性验证者校验**：
- CV-01：描述与引擎实际的全局计数器行为一致 → pass
- CV-02：无旧版单边计数描述残留 → pass
- CV-03：全局计数器的新特性已完整描述 → pass
- CV-04：描述准确无歧义 → pass
- CV-05：9 章结构未变 → pass
- **裁决：validated**

**回归测试者校验**：
- RT-01：§6 STATE.json 已被文档同步者更新（含计数器字段）→ pass
- RT-02：交叉引用准确 → pass
- RT-03~05：格式规范 → pass
- **裁决：passed**

### 反例：修改本身正确但破坏了其他部分

**场景**：代码实施者在 SDK_SPEC.md §5 新增了全局执行计数器描述，但忘记标记 §6 STATE.json 需要同步。

**一致性验证者校验**：
- CV-01~05 全部 pass（修改本身正确）
- **裁决：validated**（修改的部分没问题）

**回归测试者校验**：
- RT-01：§6 STATE.json 未更新，仍无计数器字段 → **fail (critical)**
- 失败原因：§5 新增了计数器机制，但 §6 的状态文件格式表未新增对应字段
- **裁决：failed**（修改破坏了其他部分的一致性）

此例说明：一致性验证通过不代表回归测试也会通过。两个关卡缺一不可。

### max 耗尽兜底示例：一致性验证始终未通过

**场景**：代码实施者多次修改后，一致性验证者始终检出 critical 缺陷，已达到 max_executions=2 上限。

**一致性验证者第 3 次校验**：
- CV-01：仍有描述与引擎行为矛盾 → **fail (critical)**
- **裁决：defects_detected**
- **max_exhausted: true**（标注"一致性验证最终未通过，已达到迭代上限"）
- **工作流行为**：兜底边触发，导向完成节点终止流程。有缺陷的文档不传播到依赖方。

## 检查清单

### 一致性验证者自检
- [ ] 是否逐条检查了 CV-01~CV-05
- [ ] 每条检查是否记录了 pass/fail 和严重级别
- [ ] fail 项是否给出了期望 vs 实际对比
- [ ] 裁决是否与严重级别一致
- [ ] 若为 max 耗尽兜底场景，是否通过 max_exhausted 字段如实标注

### 回归测试者自检
- [ ] 是否逐条检查了 RT-01~RT-05
- [ ] 是否重点检查了与修改章节有依赖关系的未修改章节
- [ ] fail 项是否关联了导致问题的具体修改
- [ ] 裁决是否与严重级别一致
- [ ] 是否避免了与一致性验证者重复检查（你聚焦回归影响）
- [ ] 若为 max 耗尽兜底场景，是否通过 max_exhausted 字段如实标注
