# 依赖传播操作指南

## 目的

本文档是依赖同步者执行文档传播的操作指南。定义如何扫描依赖方 app、如何同步文档、如何处理同步失败，解决核心问题：**怎么把变更传播到所有依赖方**。

## 适用角色

- **依赖同步者**（唯一适用）——作为执行依赖传播的操作指南

## 核心内容

### 1. 依赖方扫描方法

**扫描目标**：找出所有通过 knowledge 段引用了 SDK 权威文档的 app。

**核心匹配规则——按文件名而非完整路径匹配**：

各 app 在 app.yaml 中引用 SDK 文档时，可能使用不同的路径前缀（如 `engine/sdk/SDK_SPEC.md` 或 `knowledge/SDK_SPEC.md`）。因此扫描时**按文件名匹配**，而非精确路径。

SDK 权威文档及其历史文件名的映射：

| 权威文档 | 可匹配的文件名 |
|----------|--------------|
| engine/sdk/SDK_SPEC.md | `SDK_SPEC.md` |
| engine/sdk/编排范式.md | `编排范式.md`、`app.yaml编排范式.md`、`01-app-yaml编排范式.md` |

**扫描步骤**：

```
步骤 1：定位所有 app
  → 遍历 apps/ 目录下的所有子目录
  → 每个子目录代表一个 app

步骤 2：读取每个 app 的 app.yaml
  → 定位 app.yaml 文件
  → 读取 knowledge 段

步骤 3：按文件名匹配 SDK 文档引用
  → 从 knowledge 段的路径中提取文件名（basename）
  → 与上方映射表中的文件名比对
  → 匹配成功则视为依赖方
  → 记录引用关系（app 名 → 匹配到的权威文档列表）

步骤 4：建立依赖方清单
  → 汇总所有引用了 SDK 文档的 app
  → 清单格式：{ app_name, referenced_docs, knowledge_dir }
  → knowledge_dir 为该 app 存放 knowledge 文件的目录（通常为 apps/<app_name>/knowledge/）
```

**扫描检查项**：
- 是否遍历了 apps/ 下的所有子目录
- 是否对 knowledge 段的路径做了 basename 提取后再匹配
- 是否覆盖了所有历史文件名变体（如 app.yaml编排范式.md）
- 是否遗漏了间接引用（如某 app 通过变量/别名引用）

### 2. 同步操作流程

```
步骤 1：读取最新权威文档
  → 读取 engine/sdk/SDK_SPEC.md 的最新内容
  → 读取 engine/sdk/编排范式.md 的最新内容
  → 确认本次变更涉及的文档（从变更摘要获取）

步骤 2：逐个依赖方同步
  → 对依赖方清单中的每个 app：
    a. 定位该 app 的 knowledge 目录（通常为 apps/<app_name>/knowledge/）
    b. 确定目标路径——在 knowledge 目录中查找与权威文档匹配的文件名
       （如 SDK_SPEC.md 或 app.yaml编排范式.md，按扫描阶段匹配到的文件名）
    c. 将最新权威文档内容写入目标路径（覆盖旧内容）
    d. 记录同步结果

步骤 3：验证同步结果
  → 对每个已同步的文件，确认内容与源文档一致
  → 如内容不一致，标记为 failed 并记录原因

注意：目标文件名可能与权威源不同（如 app.yaml编排范式.md 对应 编排范式.md），
      但写入的内容始终是权威源的最新内容。
```

**同步操作记录格式**：
```
{
  "app_name": "<依赖方 app 名>",
  "synced_files": [
    {
      "file": "SDK_SPEC.md",
      "source": "engine/sdk/SDK_SPEC.md",
      "target": "<app>/knowledge/SDK_SPEC.md",
      "status": "success|failed",
      "failure_reason": "<仅 failed 时填写>"
    }
  ]
}
```

### 3. 同步失败处理

**核心原则**：同步失败不中断整体流程——记录失败原因，继续同步其他依赖方，最终如实报告。

**常见失败场景与处理**：

| 失败场景 | 处理方式 | 记录内容 |
|----------|----------|----------|
| knowledge 目录不存在 | 记录失败，跳过该 app | failure_reason: "knowledge 目录不存在" |
| 目标文件不存在（首次同步） | 创建文件并写入内容 | status: success（创建也算成功） |
| 文件被锁定/无写入权限 | 记录失败，跳过该文件 | failure_reason: "文件锁定/权限不足" |
| 源文档读取失败 | 记录失败，跳过该文件 | failure_reason: "源文档读取失败" |

**失败处理流程**：
```
1. 捕获失败异常
2. 记录失败原因到 sync_results
3. 不抛出异常，不中断流程
4. 继续处理下一个依赖方/文件
5. 所有同步完成后，汇总裁决
```

### 4. 依赖同步报告标准格式

```
{
  "result": {
    "verdict": "<confirmed|partial>",
    "summary": "<一句话概述同步结果>",
    "synced_documents": ["SDK_SPEC.md", "编排范式.md"],
    "dependency_scan": {
      "total_apps_scanned": "<扫描的 app 总数>",
      "dependent_apps_found": "<引用了 SDK 文档的 app 数>",
      "method": "遍历 apps/ 目录，匹配 knowledge 段引用"
    },
    "sync_results": [
      {
        "app_name": "<app 名>",
        "synced_files": [
          {
            "file": "SDK_SPEC.md",
            "target_path": "<path>",
            "status": "success",
            "failure_reason": null
          }
        ]
      }
    ],
    "summary_stats": {
      "total_syncs": "<同步操作总数>",
      "successful": "<成功数>",
      "failed": "<失败数>"
    }
  }
}
```

### 5. 裁决规则

| 条件 | verdict | 含义 |
|------|---------|------|
| 全部依赖方同步成功 | **confirmed** | 所有依赖方已获得最新文档 |
| 部分依赖方同步失败 | **partial** | 部分依赖方未同步成功，需手动处理 |

**两种裁决均到达终态**。区别在于：
- confirmed：一切正常，用户无需额外操作
- partial：有部分同步失败，用户需根据报告中的失败项手动处理

**重要**：partial 不是"失败"裁决，而是"透明性保障"——如实报告失败项，让用户知晓哪些 app 需要手动处理。

### 6. 边界声明

依赖同步者的操作边界：

| 操作 | 是否允许 | 说明 |
|------|----------|------|
| 将权威文档内容复制到依赖方 knowledge 目录 | ✅ 允许 | 核心职责 |
| 创建 knowledge 目录（如不存在） | ✅ 允许 | 辅助同步 |
| 修改依赖方 app 的 app.yaml | ❌ 禁止 | 不修改依赖方配置 |
| 修改依赖方 app 的角色代码（skill.md / schema.json） | ❌ 禁止 | 不修改依赖方角色 |
| 修改依赖方 app 的其他 knowledge 文档 | ❌ 禁止 | 只同步 SDK 权威文档 |
| 删除依赖方 app 的任何文件 | ❌ 禁止 | 不删除依赖方文件 |

## 示例

### 示例：SDK_SPEC.md 变更后的依赖传播

**场景**：SDK_SPEC.md §5 新增了全局执行计数器描述，已通过全部验证。

**扫描结果（按文件名匹配）**：
```
apps/ 目录下共 6 个 app：
- app-builder：knowledge 段含 SDK_SPEC.md 和 编排范式.md（路径 knowledge/） → 依赖方
- sdk-maintainer：knowledge 段含 SDK_SPEC.md 和 编排范式.md（路径 engine/sdk/） → 依赖方（自身）
- app-architect-v2：knowledge 段含 SDK_SPEC.md（路径 knowledge/） → 依赖方
- prose-loop：knowledge 段含 SDK_SPEC.md（路径 knowledge/） → 依赖方
- engine-doc-distiller：knowledge 段含 SDK_SPEC.md 和 app.yaml编排范式.md（路径 knowledge/） → 依赖方
- lxp-eng-planning：knowledge 段含 编排范式.md（路径 knowledge/） → 依赖方
```

**同步操作**：
```
1. app-builder：
   - SDK_SPEC.md → apps/app-builder/knowledge/SDK_SPEC.md → success
   - 编排范式.md → apps/app-builder/knowledge/编排范式.md → success

2. sdk-maintainer：
   - SDK_SPEC.md → engine/sdk/SDK_SPEC.md（权威源已在自身位置，确认一致）
   - 编排范式.md → engine/sdk/编排范式.md

3. app-architect-v2：
   - SDK_SPEC.md → apps/app-architect-v2/knowledge/SDK_SPEC.md → success

4. prose-loop：
   - SDK_SPEC.md → apps/prose-loop/knowledge/SDK_SPEC.md → success

5. engine-doc-distiller：
   - SDK_SPEC.md → apps/engine-doc-distiller/knowledge/SDK_SPEC.md → success
   - 编排范式.md → apps/engine-doc-distiller/knowledge/app.yaml编排范式.md → success（文件名不同，内容为编排范式.md）

6. lxp-eng-planning：
   - 编排范式.md → apps/lxp-eng-planning/knowledge/编排范式.md → success
```

**裁决**：全部成功 → **confirmed**

**报告摘要**：本次变更同步 SDK_SPEC.md 至 2 个依赖方 app，全部成功。

### 示例：部分同步失败

**场景**：同上，但 app-builder 的 knowledge 目录被锁定。

**同步结果**：
```
1. app-builder：
   - SDK_SPEC.md → failed（文件锁定/权限不足）
   
2. sdk-maintainer：
   - SDK_SPEC.md → success
```

**裁决**：部分失败 → **partial**

**报告**：1 个依赖方同步失败（app-builder，原因：文件锁定），1 个成功。用户需手动同步 app-builder。

## 检查清单

依赖同步者完成同步后自检：
- [ ] 是否遍历了所有 app 的 knowledge 段，未遗漏任何依赖方
- [ ] 每个依赖方 app 的同步状态是否逐一记录
- [ ] 同步失败项是否记录了具体失败原因
- [ ] 裁决是否符合规则（全成功→confirmed，有失败→partial）
- [ ] 是否未静默忽略任何同步失败
- [ ] 是否未修改依赖方 app 的 app.yaml 或角色代码
- [ ] summary_stats 的 successful + failed 是否等于 total_syncs
- [ ] 依赖同步报告格式是否完整
