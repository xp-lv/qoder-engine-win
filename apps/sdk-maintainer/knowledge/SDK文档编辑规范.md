# SDK 文档编辑规范

## 目的

本文档定义代码实施者修改 SDK 权威文档（SDK_SPEC.md 和 编排范式.md）时必须遵守的编辑约束与操作规范。解决核心问题：**怎么改才不违反文档自身约束**。

## 适用角色

- **代码实施者**（唯一适用）——作为修改 SDK 权威文档时的编辑约束和操作指南

## 核心内容

### 1. SDK_SPEC.md 硬约束清单

以下约束**不可违反**，违反即为 critical 级别缺陷：

| 约束编号 | 约束内容 | 违反后果 |
|----------|----------|----------|
| HC-01 | 9 章结构不可增删——不可新增第 10 章，不可删除已有章节 | 结构被破坏，必须回退 |
| HC-02 | 章节编号不可重排——§1~§9 的编号顺序固定 | 结构被破坏，必须回退 |
| HC-03 | 章节内可修改内容，但不可改变章节编号 | 结构被破坏，必须回退 |
| HC-04 | 代码优先原则——文档描述引擎实际行为，不可引入与代码行为矛盾的描述 | 描述与引擎矛盾，必须回退 |
| HC-05 | 不可删除已有接口描述（除非变更需求明确要求废弃移除） | 信息丢失，必须回退 |

### 2. 编排范式.md 编辑约束

| 约束编号 | 约束内容 | 违反后果 |
|----------|----------|----------|
| EC-01 | 四种原子模式定义不可修改（单步前进、并行扇出、同步汇入、终态出口） | 模式定义被篡改，必须回退 |
| EC-02 | 角色字段表可增列但不可删列 | 信息丢失，必须回退 |
| EC-03 | verdict 路由语法规则不可变 | 语法规范被破坏，必须回退 |
| EC-04 | 内容描述可修改，但修改必须反映引擎实际编排能力 | 与引擎能力矛盾，必须回退 |

### 3. 修改操作流程

```
步骤 1：定位目标章节
  → 从变更分析报告的 affected_documents 获取目标章节
  → 在文档中定位到该章节

步骤 2：读取当前内容
  → 完整读取目标章节的当前内容
  → 记录修改前摘要（用于变更摘要的 before_summary）

步骤 3：按变更分析报告修改
  → 按 proposed_change 方案修改章节内容
  → 如果是新增：在章节内适当位置插入新内容
  → 如果是修改：替换旧描述为新描述
  → 如果是删除：移除指定内容（仅在变更需求明确要求时）

步骤 4：自检结构完整性
  → 确认未跨章节移动内容
  → 确认未修改章节编号
  → 确认未引入与代码行为矛盾的描述
  → 确认 Markdown 格式完整（表格、列表、代码块无断裂）

步骤 5：交叉引用检查
  → 修改 A 章节后检查 B 章节是否引用了 A 的内容
  → 如有，在变更摘要中标记 cross_reference_flags（pending_sync）

步骤 6：产出变更摘要
  → 按标准格式记录每处修改
```

### 4. 变更摘要标准格式

```
{
  "result": {
    "verdict": "confirmed",
    "summary": "<一句话概述>",
    "modifications": [
      {
        "file": "SDK_SPEC.md",          // 或 "编排范式.md"
        "section": "§5 执行语义",       // 章节编号+名称
        "modification_type": "新增",     // 新增|修改|删除
        "before_summary": "<修改前内容摘要>",
        "after_summary": "<修改后内容摘要>",
        "reason": "<修改理由>",
        "source_ref": "<变更分析报告中的对应条目>"
      }
    ],
    "cross_reference_flags": [
      {
        "from_section": "§5 执行语义",
        "to_section": "编排范式.md verdict 路由",
        "sync_status": "pending_sync",  // pending_sync|already_synced
        "note": "<说明>"
      }
    ],
    "rollback_corrections": []  // 仅回退迭代时填写
  }
}
```

### 5. 禁止操作清单

以下操作**严格禁止**：

- ❌ 不可跨章节移动内容（如将 §5 的内容移到 §3）
- ❌ 不可删除已有接口描述（除非变更需求明确要求废弃移除）
- ❌ 不可修改章节编号（§1~§9 编号固定）
- ❌ 不可引入与代码行为矛盾的描述（违反代码优先原则）
- ❌ 不可新增第 10 章（9 章结构不可增删）
- ❌ 不可修改四种原子模式的定义（编排范式.md）
- ❌ 不可删除角色字段表的已有列（编排范式.md）
- ❌ 不可修改 verdict 路由语法规则（编排范式.md）

### 6. 交叉引用维护规则

SDK_SPEC.md 和 编排范式.md 之间存在交叉引用：

| 引用方向 | 典型引用 | 变更时需检查 |
|----------|----------|-------------|
| SDK_SPEC.md → 编排范式.md | §3 app.yaml 语法引用编排范式.md 的详细模式 | 修改 §3 后检查编排范式.md 是否需同步 |
| 编排范式.md → SDK_SPEC.md | 编译器自动行为表引用 §4 编译产物格式 | 修改编排范式.md 后检查 §4 是否需同步 |
| SDK_SPEC.md 内部 | §5 执行语义引用 §2 原子概念的定义 | 修改 §2 后检查 §5 引用是否仍正确 |
| 编排范式.md 内部 | verdict 路由引用 producer 自动展开 | 修改 producer 段落后检查 verdict 路由引用 |

**操作规则**：修改 A 章节后，检查所有引用了 A 的位置。如需同步更新，在变更摘要中标记 `cross_reference_flags`（`sync_status: pending_sync`），供文档同步者处理。

## 示例

### 正例：修改 SDK_SPEC.md §5 新增全局计数器描述

**操作**：
1. 定位 §5 执行语义章节
2. 在章节内适当位置新增"全局执行计数器"小节
3. 描述：引擎维护全局执行计数器，当某角色所有 backward 边的累计执行次数达到全局上限时，强制掐断
4. 自检：未增删章节（在 §5 内新增小节）、未改编号、描述与引擎行为一致
5. 标记交叉引用：§6 STATE.json 需同步新增计数器字段记录

**变更摘要**：
```json
{
  "file": "SDK_SPEC.md",
  "section": "§5 执行语义",
  "modification_type": "新增",
  "before_summary": "§5 描述 DAG 执行、producer 展开、verdict 路由，无全局计数器描述",
  "after_summary": "§5 新增'全局执行计数器'小节，描述全局上限掐断机制",
  "reason": "引擎新增了 max_executions 全局计数器机制",
  "source_ref": "变更分析报告条目 1"
}
```

### 反例：违反硬约束的操作

**错误操作**：为了描述全局计数器，在 SDK_SPEC.md 中新增了"§10 全局计数器"章节。

**问题**：违反 HC-01（9 章结构不可增删）。

**正确做法**：在 §5 执行语义章节内新增小节描述，不新增章节。

## 检查清单

代码实施者完成修改后自检：
- [ ] 是否遵守了全部硬约束（HC-01~HC-05）
- [ ] 是否遵守了编排范式.md 编辑约束（EC-01~EC-04）
- [ ] 每处修改是否可追溯到变更分析报告（source_ref）
- [ ] 是否检查了交叉引用并标记了 pending_sync
- [ ] 是否未执行任何禁止操作
- [ ] 变更摘要格式是否完整（file、section、modification_type、before/after_summary、reason、source_ref）
- [ ] 回退修正（如有）是否针对了回退报告中的每个缺陷
