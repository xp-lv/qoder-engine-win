# 变更影响分析方法论

## 目的

本文档指导变更分析师从 SDK 变更需求推导到结构化影响分析报告的逐步方法论。解决核心问题：**收到一段变更描述后，怎么判断哪些章节需要改、改什么、有什么连带影响**。

## 适用角色

- **变更分析师**（唯一适用）——作为分析 SDK 变更影响范围的操作方法论

## 核心内容

### 1. 变更分类框架

收到变更需求后，首先将其归类。不同类型的变更，分析侧重点不同：

| 变更类型 | 定义 | 分析侧重 | 典型场景 |
|----------|------|----------|----------|
| **新增特性** | 引擎新增了某机制/某接口/某行为 | 需在文档中增加描述，重点找"新内容应插入哪个章节" | 引擎新增了 backward 边自动生成机制 |
| **语义变更** | 已有机制的行为/接口签名发生变化 | 需更新已有描述，重点找"旧描述在哪里，改成什么" | step.py 的 --submit 参数语义从"提交并推进"变为"提交并等待确认" |
| **接口变更** | step.py / schema.json / app.yaml 的接口定义发生改变 | 需更新接口描述，重点检查接口签名表和示例 | schema.json 新增了 _required_files 字段 |
| **废弃移除** | 引擎移除了某特性 | 需删除文档中对应描述，重点检查"删除后是否有悬空引用" | 废弃了 confirm: manual 的人工确认机制 |

### 2. 影响范围识别方法

**步骤 1：关键词提取**

从变更需求描述中提取技术关键词：
- 机制名（如"backward 边""producer 展开""knowledge 注入"）
- 接口名（如"step.py --submit""schema.json""app.yaml"）
- 概念名（如"verdict""max_executions""Gate 校验"）

**步骤 2：关键词→SDK_SPEC.md 章节映射**

SDK_SPEC.md 的 9 章结构（固定不可变）：

| 章节 | 内容 | 关键词触发 |
|------|------|------------|
| §1 系统概述 | 系统是什么、做什么 | 整体架构变更 |
| §2 原子概念 | 角色、物料、verdict 等基础定义 | 新概念/概念重定义 |
| §3 app.yaml 语法 | 角色定义、edges、knowledge | app.yaml 字段变更 |
| §4 编译产物格式 | registry.json、schema.json | 编译器行为/产物格式变更 |
| §5 执行语义 | DAG 执行、producer 展开、verdict 路由 | 执行流程变更 |
| §6 STATE.json | 状态文件格式 | 状态记录机制变更 |
| §7 step.py 接口 | 命令行接口 | step.py 接口变更 |
| §8 schema.json | 输出约束格式 | schema.json 结构变更 |
| §9 Gate 校验 | 校验规则 | Gate 规则变更 |

**步骤 3：关键词→编排范式.md 段落映射**

编排范式.md 的主要段落：
- 文件结构（app.yaml 整体结构）
- 四种原子模式（单步前进、并行扇出、同步汇入、终态出口）
- 角色字段（type、confirm、inputs、outputs）
- 物料 type 分类
- verdict 路由
- producer 自动展开
- knowledge 注入
- 编译器自动行为
- 文档驱动分层模式

**步骤 4：汇总受影响章节清单**

将映射结果汇总为结构化清单，每项标注：
- 文档名（SDK_SPEC.md / 编排范式.md）
- 章节/段落
- 修改类型（新增/修改/删除）
- 当前内容摘要
- 拟修改方案

### 3. 连带影响分析

**核心原则**：一处变更可能导致其他章节的描述自相矛盾。

**连带影响检查矩阵**：

| 变更发生处 | 需检查的连带章节 | 连带风险 |
|------------|-----------------|----------|
| §2 原子概念（定义变更） | §5 执行语义、编排范式.md 四种原子模式 | 概念重定义后，引用该概念的章节描述是否仍正确 |
| §3 app.yaml 语法（字段变更） | §4 编译产物格式、编排范式.md 角色字段表 | 新增/修改字段后，编译器行为表和角色字段表是否需同步 |
| §5 执行语义（执行流程变更） | §7 step.py 接口、§6 STATE.json | 执行流程变更后，接口调用和状态记录是否受影响 |
| §7 step.py 接口（接口变更） | §5 执行语义、编排范式.md 编译器自动行为 | 接口签名变更后，执行语义描述和编译器行为表是否需更新 |
| §8 schema.json（结构变更） | §4 编译产物格式、§9 Gate 校验 | schema.json 结构变更后，编译产物格式和 Gate 校验规则是否需更新 |

**连带影响分析步骤**：
1. 对每处直接受影响章节，检查上方矩阵的连带章节
2. 读取连带章节当前内容，判断是否因本次变更而变得不准确
3. 如有连带影响，加入受影响章节清单并标注连带关系

### 4. 修改方案评估

对每处受影响章节制定修改方案时，评估：

| 评估维度 | 判据 | 风险等级 |
|----------|------|----------|
| 结构约束 | 是否破坏 SDK_SPEC.md 的 9 章结构（不增删章节、不重排编号） | 破坏→高风险（禁止） |
| 代码优先原则 | 修改后的描述是否与引擎实际行为一致（非臆想） | 矛盾→高风险（禁止） |
| 连带影响范围 | 修改是否导致大量连带章节需要同步 | 连带>3处→中风险 |
| 核心章节 | 是否涉及 §5 执行语义等核心章节 | 涉及核心→中风险 |

### 5. 变更分析报告标准结构

```
{
  "result": {
    "verdict": "confirmed",
    "summary": "<一句话概述>",
    "change_type": "<新增特性|语义变更|接口变更|废弃移除>",
    "change_description": "<引擎变更的详细描述>",
    "affected_documents": [
      {
        "document": "SDK_SPEC.md",
        "affected_sections": [
          {
            "section": "§X 章节名",
            "modification_type": "<新增|修改|删除>",
            "current_summary": "<当前内容摘要>",
            "proposed_change": "<拟修改为>",
            "reason": "<为什么需要改>",
            "cascade_from": "<如有连带影响，标注来源章节>"
          }
        ]
      }
    ],
    "cross_references": [
      {
        "source": "<SDK_SPEC.md §X>",
        "target": "<编排范式.md 某段落>",
        "sync_needed": true/false,
        "note": "<说明>"
      }
    ],
    "risk_assessment": {
      "structure_constraint_impact": "<是否破坏 9 章结构约束>",
      "risk_level": "<低|中|高>",
      "mitigation": "<风险缓解措施>"
    }
  }
}
```

## 示例

### 示例：引擎新增了 max_executions 全局计数器

**变更需求**："引擎新增了 max_executions 全局计数器机制——之前 max_executions 只在单条边上生效，现在引擎增加了全局计数器，当某个角色所有 backward 边的累计执行次数达到全局上限时，强制掐断。"

**分析过程**：

1. **变更分类**：新增特性（引擎新增了全局计数器机制）

2. **关键词提取**：max_executions、全局计数器、backward 边、累计执行次数

3. **影响范围映射**：
   - SDK_SPEC.md：
     - §5 执行语义：需新增"全局计数器"机制的描述
     - §4 编译产物格式：registry.json 中需记录全局计数器配置
   - 编排范式.md：
     - verdict 路由段落：max_executions 的描述需补充全局计数器语义
     - 编译器自动行为表：fail 边生成的 max_executions 行为需补充

4. **连带影响分析**：
   - §5 执行语义变更 → §6 STATE.json：状态文件是否需记录全局计数器当前值？（是，需记录）
   - 编排范式.md verdict 路由变更 → 编译器自动行为表：需同步更新

5. **修改方案**：
   - SDK_SPEC.md §5：新增"全局执行计数器"小节，描述全局上限语义
   - SDK_SPEC.md §6：STATE.json 格式表新增 global_execution_count 字段
   - 编排范式.md verdict 路由：max_executions 描述补充"单边上限 + 全局上限双层控制"
   - SDK_SPEC.md §4：registry.json 格式补充 global_max_executions 配置项

6. **风险评估**：
   - 结构约束：不破坏 9 章结构（在现有章节内新增小节）→ 低风险
   - 代码优先：描述引擎已有行为 → 符合
   - 连带影响：涉及 4 处修改 → 中风险

## 检查清单

变更分析师完成分析后自检：
- [ ] 变更分类是否准确（四类之一）
- [ ] 影响章节清单是否覆盖了 SDK_SPEC.md 和 编排范式.md 两侧
- [ ] 是否对每处直接受影响章节执行了连带影响矩阵检查
- [ ] 连带影响是否已标注 cascade_from
- [ ] 交叉引用是否已识别（SDK_SPEC.md ↔ 编排范式.md）
- [ ] 修改方案是否遵循 9 章结构约束
- [ ] 修改方案是否遵循代码优先原则
- [ ] 风险评估是否标注了结构约束影响和连带影响范围
