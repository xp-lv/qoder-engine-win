# 策略制定指南

> **文档类型**: 方法论文档  
> **适用角色**: 策略建议者  
> **目的**: 为策略建议者提供从验证通过的商业机会到可执行策略的转化方法，含副业启动路线图方法论和引擎能力展示获客策略，确保每个存活机会有具体、分阶段、与用户背景 + 副业约束匹配的行动路线

---

## 一、策略制定定位

### 1.1 策略建议者的核心任务

```
通过验证的机会清单 + 对抗报告 + 探索任务书
                    │
                    ▼
            策略建议（04-策略建议.md）
                    │
        ┌───────────┼───────────┐───────────┐
        ▼           ▼           ▼           ▼
    近期行动     中期路径     资源需求    副业启动路线图
   (1-3个月)   (3-12个月)   (四维度评估)  (引擎能力变现)
```

### 1.2 策略现实性原则

策略必须与探索任务书中的投入意愿和**副业约束**一致：
- 时间有限（如学生身份）→ 兼职可启动的策略
- 资金有限 → 低成本验证优先
- 能力有限 → 能力补齐作为行动项之一
- **副业约束适配**：所有策略必须满足**兼职可执行、低启动成本、快速验证**原则
  · 兼职可执行：每周 ≤ 用户副业约束中的时间预算
  · 低启动成本：启动资金 ≤ 用户副业约束中的资金上限
  · 快速验证：在用户期望变现周期内能看到初步成果

## 二、近期行动项设计方法（1-3 个月）

### 2.1 三类行动项

| 类型 | 目的 | 示例 |
|------|------|------|
| **验证类** | 验证核心假设是否成立 | 用户访谈、POC、MVP测试 |
| **探索类** | 深入了解市场和竞争 | 竞品深度调研、社群渗透 |
| **能力建设类** | 补齐执行所需的短板 | 学习产品方法论、建立行业人脉 |

### 2.2 行动项设计原则

每个行动项必须包含：
- **目标**：这个行动要验证/解决什么问题
- **步骤**：具体怎么执行（≥2 步）
- **预期产出**：完成后得到什么（可验证的结果）
- **时间节点**：预计在 X 月内完成

### 2.3 行动项设计示例

> **机会**：AI 辅助学术文献管理工具  
> **行动项**：目标用户深度访谈  
> - **目标**：验证学术研究者对 AI 辅助文献管理的真实需求和付费意愿  
> - **步骤**：① 招募 10 位 AI 领域研究者（通过学术社群/导师推荐）；② 设计半结构化访谈大纲（含痛点挖掘、现有方案评价、付费意愿测试）；③ 进行 30-45 分钟一对一访谈；④ 整理需求频次和付费意愿分布  
> - **预期产出**：10 份访谈记录 + 1 份需求验证报告（含 TOP5 痛点和付费意愿统计）  
> - **时间节点**：第 1 个月完成

### 2.4 行动项优先级排序

```
优先级 = 影响范围 × 紧迫程度 ÷ 执行成本
```

高优先级行动项特征：
- 直接验证核心假设（影响范围大）
- 不做就会走弯路（紧迫程度高）
- 成本低、速度快（执行成本低）

## 三、中期路径规划方法（3-12 个月）

### 3.1 里程碑设计

| 里程碑 | 时间 | 标志 | 退出条件 |
|--------|------|------|----------|
| **M1: 需求验证完成** | 3 月末 | 完成 ≥10 次用户访谈，需求假设验证 | 如 <30% 用户认可痛点 → 放弃 |
| **M2: MVP 上线** | 6 月末 | 核心功能可用，有 ≥50 种子用户 | 如无法获取种子用户 → 转型或放弃 |
| **M3: 初步商业化** | 9 月末 | 有 ≥10 付费用户或明确收入路径 | 如 6 个月无付费转化 → 重新评估 |
| **M4: 模式验证** | 12 月末 | 可复制的获客模式 + 稳定收入 | 如无法找到可复制获客 → 转型 |

### 3.2 资源投入节奏

| 策略模式 | 适用条件 | 投入节奏 |
|----------|----------|----------|
| **渐进式** | 资源有限、不确定性高 | 低成本验证 → 验证通过后逐步加码 |
| **集中突破** | 有明确窗口期、资源充足 | 前期集中投入抢占先机 |

> **对学术背景用户的建议**：优先选择渐进式，在科研之余小步验证，验证通过后再决定是否全力投入。

### 3.3 阶段退出条件

每个机会必须设定明确的退出信号：

| 退出信号类型 | 示例 | 触发动作 |
|-------------|------|----------|
| **需求证伪** | <30% 目标用户认可核心痛点 | 放弃或大幅调整方向 |
| **竞争碾压** | 大厂推出免费替代方案且功能全面 | 转型细分场景或放弃 |
| **资源断裂** | 无法获取关键技术/人才/数据 | 寻求合作或降低目标 |
| **时间窗口关闭** | 市场格局已定，后发者无空间 | 及时止损 |

## 四、资源需求评估框架

### 4.1 四维度评估

| 维度 | 评估内容 | 评估方法 | 学术背景用户特殊考量 |
|------|----------|----------|---------------------|
| **人力** | 需要的角色与技能 | 列出 MVP 所需的核心角色 | 用户可兼任技术角色，但通常需要产品/运营合伙人 |
| **资金** | 启动与运营成本 | 分项估算（开发/服务器/营销/法务） | 学生身份可利用校园资源和学生创业扶持 |
| **技术** | 技术栈与基础设施 | 列出核心技术依赖（框架/云服务/数据源） | 学术背景通常具备技术能力，但需评估工程化差距 |
| **时间** | 关键路径时间线 | 估算各阶段最短/预期/最长时间 | 需考虑科研任务对时间的挤压 |

### 4.2 资源需求评估示例

```
机会：AI辅助学术文献管理工具

人力需求：
  - 技术开发：用户可兼任（AI/后端），需要前端协作（兼职/外包）
  - 产品设计：需要合伙人或自学（初期可简化）
  - 运营推广：初期自运营，后期需要专职

资金需求（首6个月）：
  - 服务器/云服务：$500-1000/月
  - 外包前端开发：$5,000-10,000（一次性）
  - 用户获取：$1,000-2,000（社群运营+少量广告）
  - 法务/合规：$1,000-2,000（开源许可/隐私政策）
  - 总计：$15,000-30,000（约 10-20 万人民币）

技术需求：
  - LLM API 调用（OpenAI/开源模型部署）
  - 向量数据库（文献语义检索）
  - Web 前端框架
  - 学术数据库接入（arXiv/Semantic Scholar API）

时间需求：
  - MVP 开发：2-3 个月（兼职节奏）
  - 种子用户获取：1-2 个月
  - 反馈迭代：持续
  - 关键路径总时长：4-6 个月达到 M2
```

## 五、学术背景转化策略

### 5.1 学术-商业桥梁构建

| 学术资源 | 商业化路径 | 转化方法 |
|----------|-----------|----------|
| **研究成果/论文** | 技术壁垒起点 | 将核心算法产品化，申请专利保护 |
| **导师网络** | 行业人脉/背书 | 通过导师引荐潜在用户/合作伙伴/投资人 |
| **实验室资源** | 初始数据/算力 | 利用实验室数据构建初始模型（注意知识产权） |
| **学术声誉** | 品牌信任 | 通过学术影响力建立专业品牌 |
| **研究方法论** | 产品方法论 | 将实验设计能力转化为 A/B 测试/用户研究能力 |

### 5.2 能力补齐计划

| 商业能力短板 | 补齐方法 | 时间投入 |
|-------------|----------|----------|
| 产品设计 | 阅读《精益创业》《用户体验要素》；分析 ≥10 个竞品的产品设计 | 2-4 周 |
| 市场营销 | 学习增长黑客基础；参与创业社群；研究学术工具的获客案例 | 持续 |
| 商业模式 | 学习 SaaS 定价模型；研究学术工具的商业模式（订阅/ freemium等） | 1-2 周 |
| 融资知识 | 了解天使/种子轮流程；准备 elevator pitch | 按需 |
| 法务基础 | 了解公司注册/股权分配/开源许可/数据合规基础 | 1-2 周 |

### 5.3 学术思维 → 商业思维的转变清单

| 学术思维 | 商业思维 | 转变要点 |
|----------|----------|----------|
| 追求完美再发布 | 快速迭代，先发再优化 | MVP 思维：70分先上线 |
| 理论优先 | 用户优先 | 用户说的比理论推导重要 |
| 深度优先 | 速度优先 | 先验证方向再深入 |
| 独立完成 | 团队协作 | 学会委托和信任 |
| 长期投入 | 阶段验证 | 设定退出条件，及时止损 |

## 六、副业启动路线图方法论

### 6.1 冷启动策略设计

针对引擎能力变现机会，首批客户获取路径：

| 获客渠道 | 操作方式 | 适用场景 | 预计获客周期 |
|----------|----------|----------|-------------|
| **技术社群** | 在 GitHub/V2EX/掘金等技术社区发布编排引擎案例/开源项目 | AI 流程外包、工具开发 | 1-2 周 |
| **学术圈** | 通过导师/同学/学术会议推荐有 AI 化需求的企业 | AI 化咨询、定制方案 | 2-4 周 |
| **行业论坛** | 在垂直行业论坛发布 AI 自动化案例/解决方案 | 多角色协作定制 | 2-4 周 |
| **人脉推荐** | 通过朋友圈/LinkedIn/前同事主动触达潜在客户 | 所有服务类方向 | 1-3 周 |
| **内容营销** | 在公众号/知乎/B站发布 AI 编排技术文章吸引客户 | 培训/课程/SaaS | 1-3 个月 |

**冷启动策略选择原则**：
- 优先选择已有信任基础的渠道（学术圈 > 人脉推荐 > 技术社群）
- 首批客户重点在于积累案例和口碑，不追求利润最大化
- 前 3 个客户可考虑低价/免费换取标杆案例

### 6.2 定价模型选择

| 定价模型 | 适用场景 | 优势 | 劣势 |
|----------|----------|------|------|
| **项目制** | 初期定制项目 | 现金流快、门槛低 | 不可规模化、天花板低 |
| **订阅制** | 产品化后的 SaaS/工具 | 可预期收入、可规模化 | 需要先完成产品化 |
| **按效果付费** | 风险共担场景 | 降低客户决策门槛 | 收入不确定 |
| **混合制** | 项目制 + 订阅制过渡 | 兼顾短期现金流和长期价值 | 定价复杂度高 |

**定价演进路径**：项目制（1-3 月）→ 半标准化方案（3-6 月）→ 订阅制 SaaS（6-12 月）

### 6.3 从项目到产品的演进路径

```
阶段一：接定制项目（1-3 个月）
  · 目标：积累经验 + 案例 + 现金流
  · 关键动作：服务 2-3 个客户，理解共性需求
  · 产出：标杆案例 × 2-3

阶段二：提炼共性需求（3-6 个月）
  · 目标：开发半标准化方案/模板
  · 关键动作：将定制经验抽象为可复制的方案
  · 产出：标准化服务流程 + 报价体系

阶段三：产品化（6-12 个月）
  · 目标：开发可复制的 SaaS/工具
  · 关键动作：将方案产品化，建立自助服务体系
  · 产出：SaaS 产品 / 工具 + 订阅收入
```

### 6.4 兼职时间管理方法

| 方法 | 说明 |
|------|------|
| **时间块分配** | 将每周固定时间块用于副业（如周末 4 小时 + 工作日晚上 2 小时 × 3） |
| **自动化优先** | 利用编排引擎能力自动化重复性工作，最大化单位时间产出 |
| **里程碑驱动** | 设定明确的短期里程碑，避免无限拖延 |
| **主业优先** | 确保副业不影响科研主业进度，设置"主业红线" |
| **外包/协作** | 将低价值环节外包或与合伙人协作 |

## 七、引擎能力展示获客策略

### 7.1 如何将 app 产出报告作为获客素材

| 获客素材 | 使用方式 | 展示价值 |
|----------|----------|----------|
| **综合报告** | 向潜在客户展示完整的商业分析报告 | 展示多角色协作的全链路分析能力 |
| **编排引擎 demo** | 向客户演示编排引擎的实际运行过程 | 展示技术实力和自动化能力 |
| **标杆案例** | 将前 1-2 个成功项目包装为案例 | 用真实结果证明交付能力 |

### 7.2 如何用编排能力演示打动客户

**客户演示话术框架**：
1. **痛点引入**："您目前哪些业务流程还在手动操作？"
2. **能力展示**："让我演示一下我们的 AI 编排引擎如何自动化这个流程——"
3. **效果对比**：展示自动化前后的效率/成本对比
4. **价值量化**："按照您目前的业务量，自动化后预计每月节省 [X] 小时/[Y] 成本"
5. **行动建议**："我们可以先做一个 2 周的概念验证（POC），验证可行性"

### 7.3 标杆案例包装方法

| 包装要素 | 内容 |
|----------|------|
| **客户背景** | 行业/规模/痛点（脱敏处理） |
| **解决方案** | 使用了哪些编排能力、如何实现自动化 |
| **效果数据** | 效率提升 X%、成本降低 Y%、周期缩短 Z 天 |
| **客户证言** | 客户评价/推荐语（征得同意后使用） |
| **适用场景** | 该案例适合哪些类似客户 |

## 八、风险标注处理（risk_flag=true）

### 8.1 保守策略制定原则

当机会有 risk_flag=true 时：
- **降低投入**：不建议全职投入，保持兼职验证节奏
- **延长验证**：比正常多 1-2 个月的验证周期
- **缩小范围**：先聚焦最小可行场景，验证后再扩展

### 8.2 风险缓释措施

针对红队指出的未解决问题，制定具体缓释措施：

| 风险类型 | 缓释措施 | 观察指标 |
|----------|----------|----------|
| 市场规模不确定 | 先验证小规模付费意愿 | 付费转化率 |
| 技术壁垒不足 | 加速构建数据飞轮/网络效应 | 用户留存率/数据量增长 |
| 竞争风险 | 聚焦大厂不愿做的细分场景 | 细分市场占有率 |
| 伪需求风险 | 进行更严格的用户访谈 | NPS/复购率 |
| **副业不可行** | **调整时间投入预估/降低启动门槛/缩短变现周期** | **时间利用率/变现进度** |

### 8.3 观察指标设计

为每个 risk_flag=true 的机会设计"风险消除/加剧"的观察指标：
- **风险消除信号**：什么数据说明红队的担忧被证明不成立
- **风险加剧信号**：什么数据说明红队的担忧正在成为现实
- **决策时点**：什么时候必须做出继续/放弃的决定

## 十、检查清单

- [ ] 每个存活机会是否有 ≥3 条可执行行动项？
- [ ] 行动项是否含目标+步骤+预期产出+时间节点？
- [ ] 是否设计了近期（1-3月）和中期（3-12月）两个时间层？
- [ ] 中期路径是否有里程碑和退出条件？
- [ ] 资源需求是否覆盖四维度（人力/资金/技术/时间）？
- [ ] 是否设计了学术背景转化策略？
- [ ] 引擎能力变现机会是否包含副业启动路线图（冷启动/定价/从项目到产品/时间管理）？
- [ ] 引擎能力变现机会是否包含引擎能力展示获客策略（报告素材/演示话术/标杆案例）？
- [ ] 策略是否满足副业约束适配（兼职可执行/低启动成本/快速验证）？
- [ ] risk_flag=true 的机会是否有保守策略和风险缓释？
- [ ] 策略是否与用户投入意愿和副业约束一致？
- [ ] 行动项优先级排序是否合理？
