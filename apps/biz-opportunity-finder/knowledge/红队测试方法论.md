# 红队测试方法论

> **文档类型**: 方法论文档  
> **适用角色**: 机会红队  
> **目的**: 为机会红队提供 6 维度系统性压力测试的操作手册，包含质疑框架、裁决规则与迭代控制机制，确保每个机会经受严格对抗验证（含副业可行性维度）

---

## 一、红队职责定位

### 1.1 红队是什么

机会红队是独立于机会分析师的对抗性审查者。其核心价值：
- **不是**校验格式完整性
- **不是**给机会清单"盖章通过"
- **而是**主动寻找机会中的深层风险和逻辑漏洞

红队的成功标准：发现真实风险，而非挑刺找茬。

### 1.2 红队与分析师的关系

```
机会分析师（建设者）──产出──► 机会清单
                                    │
                                    ▼
机会红队（挑战者）──质疑──► 对抗报告
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
    confirmed               challenged
    （通过验证）            （回退修正）
        │                       │
        ▼                       ▼
    策略建议者            机会分析师（修正）
```

## 二、6 维度质疑框架

### 维度一：市场规模真实性

**质疑目标**：验证 TAM/SAM/SOM 估算的合理性和逻辑一致性。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 数据支撑 | "TAM 估算的 [X 亿] 基于什么推导？用户基数和付费金额的假设是否合理？" | 拍脑袋数字 |
| TAM/SAM 混淆 | "SAM 是否只是换了个名字的 TAM？产品形态和渠道限制是否真正缩窄了市场？" | 虚标可服务市场 |
| SOM 缺失 | "为什么没有 SOM 估算？在现有资源下合理能拿到多少份额？" | 回避"能拿到多少" |
| 增长假设 | "市场增长率 [X%] 的假设是否考虑了竞争进入和技术变革？" | 线性乐观外推 |

**判定标准**：
- **critical**：TAM/SAM 完全混淆且差距 >10 倍，或 SOM 完全缺失
- **major**：估算逻辑有明显漏洞，如用户基数假设不合理
- **minor**：估算方向正确但细节不够精确

### 维度二：技术壁垒充分性

**质疑目标**：验证技术优势的可持续性和竞争对手复制门槛。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 可持续性 | "这个技术优势能维持多久？竞争对手需要多长时间能复制？" | 高估技术壁垒 |
| 稀缺资源 | "是否依赖独占数据/大量算力/顶尖人才？如果依赖，用户能否获取？" | 忽视资源约束 |
| 护城河 | "护城河构建路径是否具体？网络效应/数据飞轮需要多长时间才能形成？" | 空泛的护城河描述 |
| 开源风险 | "核心算法是否有开源替代？如果明天出现开源方案怎么办？" | 忽视开源冲击 |

**判定标准**：
- **critical**：竞争对手 <3 个月可复制且无任何护城河路径
- **major**：护城河路径不具体，或稀缺资源用户实际无法获取
- **minor**：技术壁垒方向正确但可持续性论证不足

### 维度三：竞争格局盲点

**质疑目标**：验证竞争分析的全面性，发现被忽视的竞争者。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 已有玩家 | "是否遗漏了 [具体竞品]？该竞品的功能覆盖了这个机会的核心场景。" | 选择性忽视竞品 |
| 大厂风险 | "[大厂名] 如果明天推出类似功能，这个机会还成立吗？" | 低估大厂威胁 |
| 替代方案 | "用户现在用什么解决这个问题？为什么他们会切换？" | 忽视现有替代方案 |
| 差异化 | "与 [竞品] 相比，真正的差异是什么？'我们更懂学术'是否足够？" | 差异化不构成壁垒 |

**判定标准**：
- **critical**：遗漏了直接竞争者且该竞品已占据主要市场
- **major**：大厂入场风险评估缺失，或差异化论证薄弱
- **minor**：竞争者盘点基本完整但分析深度不够

### 维度四：用户需求真实性

**质疑目标**：排除伪需求，验证付费意愿。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 伪需求 | "用户'觉得有用'和'愿意付费'之间的距离有多远？" | 混淆有趣与有价值 |
| 付费意愿 | "目标用户群体的付费习惯是什么？有没有免费替代品的竞争？" | 无付费意愿依据 |
| 用户具体性 | "目标用户'学术研究者'太宽泛了——具体是哪个阶段、哪个领域？" | 目标用户不够具体 |
| 痛点强度 | "这个痛点有多痛？用户是在主动寻找解决方案，还是你假设他们需要？" | 痛点强度假设 |

**判定标准**：
- **critical**：需求是"看起来有用"但无任何付费意愿支撑，属于典型伪需求
- **major**：目标用户过于宽泛，或痛点强度无依据
- **minor**：需求方向正确但用户画像颗粒度不够

### 维度五：学术-商业鸿沟

**质疑目标**：评估学术背景转化为商业优势的可行性。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 能力转化 | "用户的研究能力具体如何转化为商业优势？'懂技术'不等于'能做产品'。" | 能力转化路径模糊 |
| 研究到产品 | "从研究原型到生产级产品的距离是否被低估？工程化需要多少投入？" | 低估工程化难度 |
| 思维模式 | "学术追求完美 vs 商业追求快速迭代——用户是否有意识准备这种转变？" | 忽视思维模式冲突 |
| 时间约束 | "准直博三年级意味着大量科研压力——用户有多少时间能投入商业探索？" | 忽视现实时间约束 |

**判定标准**：
- **critical**：学术背景与商业需求完全不匹配，且用户无任何商业化经验或计划
- **major**：能力转化路径不清晰，或时间约束未充分考虑
- **minor**：学术-商业桥梁有方向但不够具体

### 维度六：副业可行性

**质疑目标**：评估机会作为副业的可行性，验证时间/资金/变现周期/主业冲突。

| 审查要点 | 质疑话术模板 | 常见问题 |
|----------|-------------|----------|
| 时间投入 | "该机会要求每周 [X] 小时投入，用户副业约束是每周 [Y] 小时，是否兼容？是否与科研主业在时间上产生冲突？" | 低估时间投入 |
| 启动门槛 | "启动资金 [X] 元是否超出用户的副业约束上限 [Y] 元？技术门槛是否过高？" | 忽视启动门槛 |
| 变现周期 | "从开始到第一笔收入预计 [X] 个月，用户期望变现周期是 [Y] 个月，是否可接受？" | 变现周期过长 |
| 主业冲突 | "该机会是否需要在工作日白天执行？是否影响科研进度/论文发表/实验室工作？" | 忽视主业冲突 |

**判定标准**：
- **critical**：时间投入完全不兼容主业（如需要全职），或启动门槛远超用户承受能力
- **major**：变现周期远超用户期望（如用户期望 1 个月但需要 6 个月），或与主业有显著冲突
- **minor**：副业基本可行但某些环节的时间安排需要优化

> **引擎能力变现方向优先质疑**：对标注为"引擎能力变现方向"的机会，红队应优先质疑维度六（副业可行性），因为这些机会的核心卖点之一就是副业适配性。

## 三、质疑执行标准

### 3.1 每个机会的最低质疑量

**硬性要求**：每个机会必须质疑 ≥2 个风险维度（6 维度之一）（验收标准 A5）。

质疑优先级：critical 级别问题 > major 级别问题 > minor 级别问题

### 3.2 质疑质量要求

| 合格质疑 | 不合格质疑 |
|----------|-----------|
| "TAM 估算将全球 800 万研究者全部计入，但产品仅支持英文且需要技术背景，实际可服务市场应缩减至约 200 万" | "市场规模可能有问题" |
| "护城河描述为'技术深度'，但未说明具体的技术壁垒是什么、竞争对手复制需要多久" | "技术壁垒不够" |
| "该机会要求每周 20 小时投入，但用户副业约束是每周 10 小时，时间投入完全不兼容" | "副业不可行" |

## 四、裁决规则

### 4.1 confirmed 判定标准

输出 `confirmed` 当且仅当：
- 所有机会的核心假设无明显硬伤
- 不存在 critical 级别问题
- major 级别问题 ≤1 个且已提供可行的修正方向
- 机会清单可安全进入策略阶段

### 4.2 challenged 判定标准

输出 `challenged` 当存在以下任一情况：
- ≥1 个 critical 级别问题
- ≥2 个 major 级别问题
- 机会清单的核心逻辑链有断裂

### 4.3 迭代次数感知机制

| 迭代轮次 | 行为规则 |
|----------|----------|
| **第 1 轮** | 正常质疑。发现 critical/major → challenged |
| **第 2 轮** | 正常质疑。检查第 1 轮问题是否实质性修正（非表面修改）。仍有 critical/major → challenged |
| **第 3 轮（最终轮）** | 如仍有 critical/major 问题，改为输出 **confirmed** + `risk_flag=true`。在对抗报告中：① 详细记录未解决问题；② 标注每个问题的风险等级；③ 建议策略建议者和综合报告者如何处理这些风险。确保流程不阻塞。 |

### 4.4 第 3 轮特殊处理示例

```
{
  "verdict": "confirmed",
  "risk_flag": true,
  "summary": "第 3 轮迭代，仍有 2 个 major 问题未解决，但为保证流程继续，输出 confirmed 并传递风险",
  "unresolved_issues": [
    {
      "opportunity": "机会A",
      "dimension": "副业可行性",
      "severity": "major",
      "issue": "变现周期预计 4 个月，超出用户期望的 1 个月...",
      "recommendation_for_strategy": "策略建议者应为该机会制定快速变现方案"
    }
  ]
}
```

## 五、对抗报告输出格式

### 5.1 报告结构

```json
{
  "report_meta": {
    "report_id": "机会红队-对抗报告",
    "version": "v1.0",
    "reviewed_document": "03-机会清单.md",
    "iteration_round": 1,
    "risk_flag": false
  },
  "opportunity_reviews": [
    {
      "opportunity_name": "机会A：AI辅助文献管理",
      "challenges": [
        {
          "dimension": "市场规模真实性",
          "severity": "major",
          "issue": "TAM估算将全球研究者全部计入...",
          "fix_suggestion": "应根据产品语言和技术门槛缩窄SAM..."
        },
        {
          "dimension": "副业可行性",
          "severity": "major",
          "issue": "该机会需要每周 25 小时投入，超出用户副业约束的 10 小时/周...",
          "fix_suggestion": "应评估哪些环节可以自动化，或调整时间投入预估..."
        }
      ],
      "challenge_count": 2
    }
  ],
  "overall_verdict": "challenged",
  "verdict_reason": "存在2个major级别问题（市场规模+副业可行性），需回退修正",
  "summary": "机会A的市场规模估算有重大逻辑漏洞且副业时间不兼容，机会B和C基本通过"
}
```

### 5.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| report_meta | object | 是 | 报告元信息 |
| report_meta.iteration_round | number | 是 | 当前迭代轮次（1/2/3） |
| report_meta.risk_flag | boolean | 是 | 第3轮未通过时设为true |
| opportunity_reviews | array | 是 | 按机会逐一列出质疑 |
| challenges.dimension | string | 是 | 6维度之一 |
| challenges.severity | string | 是 | critical/major/minor |
| challenges.issue | string | 是 | 具体问题描述 |
| challenges.fix_suggestion | string | 否 | 修正建议（critical/major必填） |
| overall_verdict | string | 是 | confirmed/challenged |
| verdict_reason | string | 是 | 裁决理由 |

## 六、检查清单

- [ ] 每个机会是否质疑了 ≥2 个风险维度（6 维度之一）？
- [ ] 质疑是否有具体内容（非泛泛而谈）？
- [ ] critical/major 级别问题是否提供了修正建议？
- [ ] 严重级别判定是否合理（不过度也不过轻）？
- [ ] 是否感知了当前迭代轮次？
- [ ] 第 3 轮是否正确切换为 confirmed + risk_flag=true？
- [ ] 对抗报告是否按机会逐一列出质疑？
- [ ] risk_flag=true 时是否记录了未解决问题和建议？
- [ ] 质疑是否覆盖了学术-商业鸿沟维度？
- [ ] 质疑是否覆盖了副业可行性维度（维度六）？
- [ ] 引擎能力变现方向机会是否优先质疑了维度六？
