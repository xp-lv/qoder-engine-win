# 商业机会探索器 — 需求描述文档

> **文档编号**: 00-需求描述  
> **版本**: v1.0  
> **状态**: 已确认  
> **产出者**: 需求接收者（producer 入口角色）

---

## 一、目标

构建一个 **商业机会探索器** APP，帮助用户（特别是学术背景人士，如准直博学生）系统性地发现、分析和评估 AI 时代的商业机会。APP 通过多角色协作完成从 **市场信息搜集 → 机会识别 → 对抗性质疑 → 策略建议 → 综合报告** 的全链路分析，最终产出一份结构化、可执行的商业机会探索报告。

### 1.1 目标用户画像

| 维度 | 描述 |
|------|------|
| **身份** | 准直博三年级学生，有学术研究背景 |
| **技术栈** | 熟悉 AI/ML 领域的前沿技术动态 |
| **核心诉求** | 探索 AI 时代是否存在值得投入的商业机会 |
| **知识盲区** | 缺乏市场分析、商业可行性评估、竞品调研经验 |
| **期望产出** | 可落地执行的商业机会清单 + 行动路线图 |

---

## 二、核心能力

| 编号 | 能力域 | 描述 |
|------|--------|------|
| C1 | **用户画像与意图理解** | 基于用户背景（学术经历、技术栈、兴趣领域）生成个性化的商业探索方向，明确探索边界（关注行业、技术领域、投入意愿） |
| C2 | **市场情报搜集** | 系统搜集 AI 领域的多维度市场信息：技术趋势、融资数据、竞品格局、用户需求信号、政策导向 |
| C3 | **机会识别与评估** | 从市场情报中识别潜在商业机会，每个机会包含：机会描述、目标用户、市场规模估算、技术壁垒、竞争分析、可行性评分 |
| C4 | **对抗性质疑** | 对识别出的机会进行红队压力测试，从市场规模真实性、技术壁垒充分性、竞争格局盲点、伪需求风险等维度逐一挑战 |
| C5 | **策略建议** | 基于通过对抗验证的机会，制定可执行的行动策略：近期行动项（1-3 个月）、中期路径（3-12 个月）、资源需求 |
| C6 | **综合报告** | 整合全链路分析结果，产出结构化的机会探索报告，包含执行摘要、详细分析、行动路线图、风险提示 |

## 三、角色清单

| 序号 | 角色名 | 类型 | confirm | 职责 |
|------|--------|------|---------|------|
| 1 | **需求接收者** | producer | manual | 入口角色，理解用户背景与探索意图，生成结构化的探索任务书（含用户画像、关注领域、探索边界） |
| 2 | **市场情报搜集者** | standard | auto | 搜集 AI 领域多维度市场信息，产出结构化市场情报报告 |
| 3 | **机会分析师** | standard | auto | 分析市场情报，识别潜在商业机会并逐一评估，产出机会清单 |
| 4 | **机会红队** | standard | auto | 对抗角色，对机会清单进行压力测试，出具 confirmed/challenged 裁决 |
| 5 | **策略建议者** | standard | auto | 基于通过验证的机会，制定可执行的行动策略 |
| 6 | **综合报告者** | standard | auto | 整合全链路产出，撰写最终机会探索报告 |

> **角色约束**: 共 6 个角色，含 1 个 producer 入口（需求接收者），满足 ≥2 角色 + ≥1 producer 要求。

## 四、协作流程

### 4.1 流程拓扑

```
需求接收者 (producer)
    │
    ▼ [自动校验]
市场情报搜集者
    │
    ▼
机会分析师 ◄──────────────┐
    │                      │
    ▼                      │
机会红队 (对抗)            │
    │                      │
    ├── confirmed ────────►│ (通过，进入下游)
    │                      │
    └── challenged ────────┘ (回退，迭代修正，max 3 次)
    │
    ▼ [confirmed]
策略建议者
    │
    ▼
综合报告者
    │
    ▼
完成
```

### 4.2 流程说明

| 阶段 | 模式 | 说明 |
|------|------|------|
| 需求接收者 → 校验 | producer 自动校验 | 确认探索任务书七大要素齐全 |
| 校验 → 市场情报搜集者 | 线性前进 | 校验通过后启动情报搜集 |
| 市场情报搜集者 → 机会分析师 | 线性前进 | 情报搜集完成后启动机会分析 |
| 机会分析师 → 机会红队 | 线性前进 | 机会清单提交对抗审阅 |
| 机会红队 → 机会分析师 | 迭代回退（max 3 次） | challenged 时回退修正 |
| 机会红队 → 策略建议者 | 条件路由 | confirmed 时进入策略阶段 |
| 策略建议者 → 综合报告者 | 线性前进 | 策略制定完成后整合报告 |
| 综合报告者 → 完成 | 终态出口 | 报告产出后工作流结束 |

### 4.3 迭代控制

- **对抗迭代回路**（机会红队 → 机会分析师）: 最多 3 轮
  - 每轮：机会红队出具 challenged + 质疑报告 → 机会分析师根据质疑修正机会清单 → 重新提交
  - 超过 3 轮仍未通过：以最后一次机会清单进入下游（带 risk flag 标记）

## 五、对抗性要求

### 5.1 机会红队职责

机会红队是本 APP 的核心对抗角色，负责对机会分析师产出的机会清单进行系统性压力测试：

| 质疑维度 | 审查要点 |
|----------|----------|
| **市场规模真实性** | 市场估算是否有数据支撑？TAM/SAM/SOM 是否合理？是否混淆潜在市场与可服务市场？ |
| **技术壁垒充分性** | 技术优势是否可持续？竞争对手复制门槛有多高？是否依赖稀缺资源（数据/算力/人才）？ |
| **竞争格局盲点** | 是否忽视了已有玩家？大厂入场风险如何？是否存在替代方案？ |
| **用户需求真实性** | 是否为伪需求？用户付费意愿如何？是否混淆"有趣"与"有价值"？ |
| **学术-商业鸿沟** | 用户学术背景能否有效转化为商业优势？从研究到产品的距离是否被低估？ |

### 5.2 裁决规则

- **confirmed**: 机会清单经受住质疑，各机会的核心假设无明显硬伤 → 进入策略建议阶段
- **challenged**: 存在可修复的重大缺陷 → 回退机会分析师修正

## 六、验收标准

以下标准全部可客观判定通过/不通过：

| 编号 | 验收项 | 判定方法 |
|------|--------|----------|
| A1 | 探索任务书包含用户画像 ≥ 3 个维度（背景、技能、兴趣） | 检查产出文件中用户画像段是否含 ≥3 维度字段 |
| A2 | 市场情报报告覆盖 ≥ 3 个信息维度（技术趋势、融资数据、竞品分析） | 检查报告是否含 ≥3 个维度的独立章节 |
| A3 | 机会清单识别 ≥ 3 个潜在机会 | 检查清单条目数 ≥ 3 |
| A4 | 每个机会含 6 项必填要素（描述、目标用户、市场规模、技术壁垒、竞争分析、可行性评分） | 逐条检查每个机会条目字段完整性 |
| A5 | 对抗报告对每个机会质疑 ≥ 2 个风险维度 | 检查报告中对每个机会的质疑条目数 ≥ 2 |
| A6 | 策略建议为每个存活机会提供 ≥ 3 条可执行行动项 | 检查每个机会的行动建议数 ≥ 3 |
| A7 | 综合报告包含：执行摘要、详细分析、行动路线图、风险提示四个章节 | 检查报告目录结构完整性 |
| A8 | DAG 可达性：所有角色可从需求接收者到达，综合报告者可达终态 | 检查 app.yaml 边拓扑无孤岛、有终态出口 |
| A9 | verdict 完备性：机会红队的每条 `when:` 条件路由值在 edges 中均有对应出口 | 检查 confirmed 和 challenged 均有明确路由目标 |

## 七、文档层级

### 7.1 文档分层结构

本 APP 采用 **三层文档体系**（需求文档 → 层分解文档 → 实现规格文档）：

| 层级 | 文档类型 | 产出者 | 输入依据 | 职责 | 确认方式 |
|------|----------|--------|----------|------|----------|
| L0 | **需求文档**（本文档） | 需求接收者 | 用户原始需求 | 定义 APP 目标、核心能力、角色清单、协作流程、对抗要求、验收标准 | manual（用户确认） |
| L1 | **层分解文档**（app.yaml） | 架构师 | 需求文档 | 定义角色拓扑、edges 路由、物料声明、知识注入 | auto |
| L2 | **实现规格文档**（skill.md + schema.json） | 技能填充者 | 层分解文档 | 为每个角色定义执行指令、输出约束、校验规则 | auto |

### 7.2 文档间依赖关系

```
用户原始需求
    │
    ▼
L0 需求文档（唯一权威源：目标、角色、流程、验收标准）
    │
    ▼（上层文档是下层文档的唯一输入依据）
L1 层分解文档 app.yaml（唯一权威源：角色拓扑、edges、物料）
    │
    ▼（上层文档是下层文档的唯一输入依据）
L2 实现规格文档 skill.md + schema.json × N（唯一权威源：每个角色的执行逻辑与输出约束）
```

> **依赖规则**: 上层文档是下层文档的唯一输入依据，禁止下层文档反向修改上层文档定义。如发现矛盾，回退至上层修正。

### 7.3 文档层与执行层的关系

- **执行依据**: 运行时引擎以 L2 实现规格文档（skill.md + schema.json）为每个角色的执行依据
- **路由依据**: 运行时引擎以 L1 层分解文档（app.yaml 编译产物 ROUTER.json + registry.json）为调度依据
- **验收依据**: L0 需求文档中的验收标准用于判定最终产出物是否达标

### 7.4 每层审阅/对抗角色

| 文档层 | 是否需要审阅 | 是否需要对抗 | 说明 |
|--------|-------------|-------------|------|
| L0 需求文档 | ✅ producer 自动校验 | ✅ 需求红队（如引入） | 本 APP 由 producer 自动校验保障 |
| L1 层分解文档 | ✅ 可引入架构审阅 | 可选 | 由架构师产出后可直接进入技能填充 |
| L2 实现规格文档 | ✅ 可引入模拟验证 | 可选 | 由技能填充者产出后可直接运行 |

## 八、约束与边界

### 8.1 输入边界
- 用户背景信息：准直博三年级、AI 领域学术背景、探索商业机会意愿
- 知识依赖：app-builder 的编排范式文档（编排范式.md）

### 8.2 产出边界
- 6 个角色各自产出独立的 deliverable 文件
- 机会红队的对抗报告为 process 类型（反馈物料）
- 综合报告者为最终交付物

### 8.3 不做什么
- 不执行实际的市场数据采集（APP 输出为分析框架和建议，非实时数据源）
- 不提供财务/法律/投资建议（聚焦商业机会识别与策略方向）
- 不替代用户的最终决策（APP 是辅助探索工具，决策权在用户）
