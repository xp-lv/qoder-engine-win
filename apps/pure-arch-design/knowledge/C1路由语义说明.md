# C1 路由语义说明

> 本文档说明 L2 需求规格文档相关的 **SDK 路由机制**——manual confirm 路由语义、REQ-ID 增补路由实现。
>
> **职责边界**：本文档只覆盖 SDK 路由机制（"路由如何实现"）。L2 的业务编写规则见《L2需求规格编写指南》，二者正交、独立演进。
>
> **适用角色**：需求分析师（manual confirm 节点）、需求分析师校验（路由消费方）。

---

## 陷阱 1：L1 不充分时的路由实现

**问题背景**：需求分析师读 L1 后，若 L1 充分性四要素检查不通过，需要产出需求澄清请求 + 占位 L2，并路由回自身等待用户补充 L1。

**路由实现（v8.0）**：

```
需求分析师（L1 不充分）
    ↓ 产出需求澄清请求 + 占位 L2
    ↓ emit confirmed（产出完成）
    ↓ 用户 manual confirm
    ↓ 用户看到需求澄清请求 → reject（fail）
    ↓ SDK Phase 3 覆盖为 fail
    ↓ ROUTER.json fail 边
需求分析师（回退自身，等待用户补充 L1 后重新触发）
```

**关键约束**：
- 需求分析师是 `confirm: manual` 角色，用户确认行为驱动路由
- 用户看到需求澄清请求后，**必然 reject**（因为 L1 不充分需要补充）
- reject 触发 SDK Phase 3 产生 `fail` verdict → fail 边回到自身
- 用户补充 L1 后重新触发需求分析师执行

**与 v7.x 的区别**：v7.x 依赖 producer + manual 的 SDK Phase 3 覆盖语义；v8.0 后所有 manual 角色统一行为（用户 confirmed 保留原始 verdict，用户 fail 覆盖为 fail）。

## 陷阱 2：REQ-ID 增补后的路由实现

**问题背景**：对抗期发现隐含需求时，需求分析师需要执行 L2 解冻-写入-重冻结。更新完成后，架构师需要识别"这是增补后的重跑"以触发 F3-001（从 R1 重跑）。

**路由实现（v8.0）**：

```
终审裁决者 emit reqid_approved
    ↓ ROUTER.json transitions
需求分析师（L2 解冻-写入-重冻结）
    ↓ emit confirmed（L2 更新完成）
    ↓ 用户 manual confirm
    ↓ 用户 confirmed → SDK Phase 3 保留原始 verdict（confirmed）
    ↓ ROUTER.json confirmed 边
需求分析师校验
    ↓ emit confirmed（校验通过）
    ↓ ROUTER.json confirmed 边
架构设计师
    ↓ 通过 carries 中的 REQ-ID增补审批.json 识别"增补后重跑"（F3-001 信号）
    ↓ 从 R1 重跑全部 4 轮
```

**关键约束**：
- 需求分析师更新 L2 后 emit confirmed（不是 `l2_updated` 等自定义 verdict）
- 用户 confirmed 时 SDK Phase 3 **保留原始 verdict**（v7.0.3 修复）
- 架构师通过 **carries 信号**（REQ-ID增补审批.json）识别增补后重跑，不依赖 verdict 标签
- 详细的增补路由链路见《SDK陷阱规避》§陷阱 2

**与 v7.x 的区别**：v7.x 依赖 producer emit confirmed + validate step 转译；v8.0 后校验角色直接 emit confirmed，架构师通过 carries 信号识别。

## 汇总：manual confirm 的 verdict 行为

| 场景 | 角色执行 | 用户行为 | SDK Phase 3 产生 | 路由 |
|------|---------|---------|-----------------|------|
| L1 充分，L2 合规 | emit confirmed | confirmed | confirmed（保留）| → 需求分析师校验 |
| L1 不充分 | emit confirmed（产澄清请求）| reject（fail）| fail（覆盖）| → 回退自身 |
| REQ-ID 增补后 L2 更新 | emit confirmed | confirmed | confirmed（保留）| → 需求分析师校验 → 架构设计师 |
