# 架构蓝图设计原则

> 本文档是「架构设计师」角色产出架构蓝图 / 响应 problem 时的设计指导。校验角色不读本文档，改读《架构蓝图校验清单》。
>
> 每条原则后的"Why"提示这条原则背后的架构思想。完整论证详见《架构蓝图规范》。

---

## 1. 蓝图必须含 7 个强制章节（AC-2）

①系统总体 ②层划分 ③模块清单+RACI ④并行兼容性 ⑤文档层级 ⑥目标架构的失败模式与应对 ⑦3 轮对抗记录。

**Why**：7 个章节对应架构的 4 个核心问题（组件/关系/边界/依据，Ralph Johnson）。省略任一章 = 架构在某个维度缺失。

## 2. 章节⑥「失败模式」必须覆盖三类（AC-7）

单点失效 / 依赖不可用 / 过载，每类含触发条件 + 应对策略；明确为**目标架构**（蓝图描述对象）的失败模式，不是本 app 对抗流程的失败模式，也不是运维手册（F-009 消歧）。

**Why**：这三类覆盖了分布式系统的三大失效维度（时间/空间/容量，Release It! Nygard 2007）。少一类 = 架构在赌博。

## 3. 每个 REQ-ID 必须满足落位最小形态三要素（high-①）

REQ-ID 显式引用（grep 蓝图必须命中）+ 对每条 acceptance_criteria 逐条响应（接受/转化/反证之一，不能跳过）+ 章节末尾落位结论句。

**Why**：三要素是追溯链的三个环节（能找到 → 能对上 → 能收尾）。任一缺失 = 追溯链断裂 = 需求可能被静默丢失。

## 4. 双向可追溯（AC-3）

正向（REQ → 蓝图章节）+ 反向（蓝图章节 → REQ），任一方向断裂判 high-①。

**Why**：正向防"需求丢失"，反向防"架构师虚构需求"。这是 IEEE 830-1998 需求标准的强制要求。

## 5. RACI 责任清晰（high-③）

每个关键产出物必须有 Responsible；同一产出物不可 ≥ 2 个 R；RACI 中角色必须在模块清单中存在。

**Why**：没有 R = 责任真空 = 没人会真正实现它。多 R = 责任不清 = 多人重复实现或都不实现。RACI 是架构责任的显式化。

## 6. 并行兼容性（high-④）

并行角色读写同一资源必须有锁/序约束。

**Why**：架构层不声明锁策略 = 把 CAP 决策下放给实施者 = 分布式锁不一致，更难调试。

## 7. 架构师响应机制（详见《架构蓝图规范》§四，唯一权威源）

对每条 high problem 必须二选一响应——接受（resolved，标注修订位置）或拒绝（accepted_with_reason，必须给机制级理由三类反证之一；不能仅说"不需要"）。

**Why**：三类反证设计目的是击穿架构师的本能防御（搪塞性拒绝）。机制级理由 = 可证伪 = 红队可以验证。这是 Devil's Advocate 的工程化。

## 8. 禁用模糊词（high-⑥）

禁用"合理""适当""较快""足够""良好"等无法量化的词。

**Why**：模糊词 = 不可证伪 = 主观解读 = 对抗退化成争论。Karl Popper："无法被证伪的命题是信仰，不是科学。"

## 9. 禁止私自添加未注册的 REQ-XXX（§9.4 F-001 修复约束）

发现隐含需求必须走 REQ-ID 增补流程（标注 unregistered_requirement + 推演依据 → 终审审批），违规红队可直接判 fail。

**Why**：私自添加 = 绕过需求分析师 = 需求追溯链断裂。所有需求必须经 L2 唯一入口。

## 10. 重跑从空白蓝图重起（§9.2）

终审 consistency_defect 回退时不保留前次 v1。

**Why**：保留 v1 = 架构师被旧设计锡定，重跑只是修修补补。从空白起 = 击穿锡定效应，有机会重新思考。

## 11. REQ-ID 增补后从 R1 重跑（F3-001）

增补后所有已完成轮次自动失效。

**Why**：新增 REQ 可能影响所有维度。部分重跑 = 某些维度未检查新 REQ = 质量门有漏洞。

## 12. 跨层回退归档强制

跨层回退（consistency_defect / reqid_approved 链路）触发时，**覆盖固定路径前必须先归档**（操作步骤详见《归档策略》，唯一权威源）。

**Why**：不归档 = 前次对抗记录丢失 = AC-4 severity 变化链不可追溯 = 后期无法回溯决策依据。

## 13. C1 修复配套-F3-001 信号识别

通过 dispatch 注入的 REQ-ID增补审批.json（carries 信号）识别"增补后重跑"，不依赖 l2_updated verdict 标签。

**Why**：依赖 verdict 标签 = 架构师可以伪造标签跳过识别。依赖 carries 物理文件 = 架构师无法绕过。

## 14. 蓝图头部版本号 + 修订日志

每次修订必须更新版本号并记录修订日志。

**Why**：版本号 + 日志 = 后期维护者可以回溯"为什么这么改"。这是 ADR（Architecture Decision Records, Nygard 2011）的最低实现。
