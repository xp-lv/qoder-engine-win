# 严重度判定准则

## 目的
本文档为红队（结构红队 / 边界红队 / 极限红队）、架构设计师、终审裁决者提供**客观、可证伪、可反证**的严重度判定准则表。所有 problem 的 `severity` 字段定级、`severity_rationale` 引用、架构师不服严重度时的反证、终审复核严重度时的判据，**均以此表为唯一权威源**，禁止主观调整。

**v2 修订说明**：补全三档分级的信息论依据、high/medium/low 的工程度量逻辑、severity_rationale 强制引用的客观化机制。让所有角色理解"为什么这么分级"而非机械查表。

---

## 设计哲学：为什么需要客观准则表（v2 新增·先读这一段）

### 为什么不能靠主观判定

严重度判定是"火药味"的核心——它决定了哪些 problem 必须修复（high）、哪些可以提前推进（medium/low）。如果判定主观化，整个对抗机制会退化：

```
主观判定路径：
红队说 "这个问题很严重" → 架构师说 "不严重吧" → 终审说 "看起来还可以"
     ↓
三方各说各话，判定依赖个人感受
     ↓
对抗机制火药味不足，退化成主观争论
     ↓
最后的高严重度问题数取决于"哪个红队更严"，而非问题本身的严重度
```

**唯一解**：把判定依据外化为可查表的准则。每个人查同一张表，得到同样的答案。

### 为什么是三档（high/medium/low），不是五档或二档

这是工程心理学问题：

| 分级数 | 问题 |
|------|------|
| 2 档（高/低） | 二元判定丢失中间状态。大量"不算高但也不应该忽略"的问题被强制归为低 |
| **3 档（高/中/低）** | **正好覆盖三种行动语义：必须修 / 应该修 / 可不修。与 Miller's Law 7±2 的子集契合（三个选项可一次成像）** |
| 5 档（P0/P1/P2/P3/P4） | 粒度太细，红队/架构师/终审之间会出现"这是 P1 还是 P2"的争议，判定成本反而上升 |

**架构思想依据**：这是 "Rule of Three"（三数原则）在软件工程的延伸——三个选项是最小可行决策集，少了丢失信息，多了认知超载。

### 三档度量的客观依据

| 档位 | 对应的行动语义 | 为什么这么定义 |
|------|--------------|--------------|
| **high** | 必须修复才能推进 | 问题会阻碍主链路 / 造成系统性失效。不修不能过终审 |
| **medium** | 可保留但需 mitigation | 问题不阻碍主链路，但会影响某些场景。需要架构师明确响应（接受/拒绝） |
| **low** | 可不修 | 不改语义的纯文本问题。架构师可以集体打包响应，不需逐条响应 |

### severity_rationale 强制引用条目编号的原因

这是本准则表最关键的工程化设计——不靠人判定，靠查表判定：

```
如果 severity_rationale 可以写 "严重""一般""轻微"
     ↓
这等于没写——谁都可以说 "我觉得严重"
     ↓
判定退化为主观
```

**强制引用条目编号**（如 "high-①"）相当于：
1. 红队必须查表找条目 → 被迫真的理解准则
2. 架构师反证时必须引用同一张表 → 双方使用同一语言
3. 终审复核时只需查表 → 判定可追溯

这是"原则外化"（Principle Externalization）的工程化——卷一补 §3.1 「原则对照」的具体实现。

- 结构红队（R1）

- 终审裁决者（复核严重度定级时以此表为唯一判据）

## 一、三档严重度客观判定准则表

### high 档（满足任一即定级 high）

| 编号 | 客观判定准则 |
|------|------------|
| **high-①** | 某条 REQ-XXX 在架构蓝图中**无任何模块/章节显式声明承担**（最小落位形态定义见 §二）；落位最小形态三要素任一缺失也算 high-① |
| **high-②** | 蓝图内部存在**自相矛盾**（同一对象两处描述冲突，可导致多种实现）|
| **high-③** | RACI 角色不可达或**责任真空**（某关键产出物无 Responsible；或同一产出物 ≥ 2 个 Responsible）|
| **high-④** | **并行兼容性破坏**（并行角色读写同一资源无锁/无序）|
| **high-⑤** | 缺失**关键失败模式**（单点失效 / 依赖不可用 / 过载 三类之一未覆盖）|
| **high-⑥** | 验收标准**无法证伪**（含"合理""适当"等模糊词）|
| **high-⑦** | **模块化质量缺陷**：模块无独特能力（"角色即能力"违反）/ 模块间非接口的直接依赖 / 单模块承担 ≥ 3 类不相关职责 / 同一逻辑在多个模块重复实现 |
| **high-⑧** | **复杂度超标**：系统分层 > 4 层且无充分理由 / 存在仅转发无逻辑的中间层 / 核心业务模块占比 < 60% |
| **high-⑨** | **可演进性缺陷**：新增模块需修改 ≥ 3 个既有模块 / 自顶向下递进链断裂（上层未推出下层）|
| **high-⑩** | **技术选型/数据流缺陷**：技术与场景不匹配 / 数据流明显迂回（如 A→B→C→D→E 而非 A→E）|

### medium 档（满足任一即定级 medium）

| 编号 | 客观判定准则 |
|------|------------|
| **medium-①** | 涉及 **≤ 2 个模块**且不阻断主链路的边界缺失（N=2，可程序化计数）|
| **medium-②** | 文档层级中某层缺失或**不闭环** |
| **medium-③** | 接口对齐不完整（参数名/类型对不上但**不影响主链路**）|
| **medium-④** | 命名冲突可消歧但不一致（同一对象在不同章节出现 ≥ 2 种命名）|

### low 档（满足任一即定级 low）

| 编号 | 客观判定准则 |
|------|------------|
| **low-①** | **不改语义的纯文本调整**（错别字 / 排版 / 注释补全）|
| **low-②** | 文档格式规范化（如 mermaid 语法修正）|
| **low-③** | 章节内引用链补全（如缺失交叉引用锚点，可 grep 验证）|

> 客观化要点：medium-① 量化为 "≤ 2 个模块"；low-① 删除"性能/可读性"开放词改为"不改语义"；low-② 删除"命名风格"改为"文档格式规范化"；low-③ 新增"章节内引用链补全"。

## 二、high-① 落位最小形态三要素（F2-003 修复）

"完全无落位"的客观判定：架构蓝图中**无任何模块/章节显式声明承担该 REQ-XXX**。落位的最小形态必须**同时满足以下三项**，三者齐全才算有效落位，否则判 high-①：

1. **REQ-ID 显式引用**：蓝图对应章节正文中显式出现该 REQ-ID 字符串（如"本模块承担 REQ-017"）。Gate 校验：grep 蓝图必须命中该字符串。
2. **需求要素逐条响应**：该章节必须对该 REQ 的每条 `acceptance_criteria`（L2 §3.3）逐条给出"接受 / 转化 / 反证"之一，**不能跳过**。
3. **落位结论**：该章节末尾必须有落位结论句（如"REQ-017 已由本模块 §X.Y 完整承担"）。

> Gate 校验可对每个 REQ-ID 在蓝图中检索这三类标记；任一缺失判 high-① 成立。这消除了 v2 中"加一行 REQ-ID 即文字层面落位"的伪落位空间。

## 三、severity_rationale 引用规范

### 红队引用（定级时）

红队 problem schema 的 `severity_rationale` 字段**必须**引用本表中的具体条目编号：

```
✅ 正例："依据 high-③：模块 X 的 Responsible 角色未定义，存在责任真空。"
✅ 正例："依据 medium-①：涉及 §3.2 和 §3.4 两个模块的边界缺失（N=2），但不阻断主链路。"
✅ 正例："依据 low-①：§4.2 中『架构执行者』应为『架构实现者』，错别字，不改语义。"

❌ 反例："这个问题很严重。"（未引用条目编号）
❌ 反例："依据经验判定 high。"（未引用条目编号）
❌ 反例："依据 high 档。"（未引用具体条目编号 high-①/②/...）
```

### 架构师引用（反证严重度时）

架构师若不服红队的严重度定级，可引用**同一张表**反证：

```
✅ 正例："红队标 high-⑤ 但失败模式『过载』已在蓝图 §6.4 覆盖（含触发条件 + 应对策略两段），应降级为 medium-① 或撤销。"
✅ 正例："红队标 high-① 但 REQ-017 已在 §3.2 模块清单 + §3.3 RACI + §5.2 接口中三处显式承担（grep 命中），落位最小形态三要素齐全，应撤销。"
```

### 终审引用（复核时）

终审裁决者复核严重度定级时：
- **以此表为唯一判据**，不作主观调整。
- 若红队与架构师对严重度各执一词，终审必须明确引用条目编号给出裁决（如"依据 high-③，责任真空客观存在，红队定级成立"）。

## 四、final_status 与严重度的联动

| final_status | 语义 | 适用严重度 |
|------|------|----------|
| `resolved` | 架构师**接受** problem → 蓝图已修订 → 问题清零 | 任意 |
| `accepted_with_reason` | 架构师**拒绝** + 机制级理由成立 → 红/终审认可拒绝 → problem 可接受关闭 | 任意 |
| `rejected_with_reason` | 架构师拒绝 + 理由不成立 → 终审判红队合理 → 强制修订 → 转入 resolved 路径 | 任意 |
| `escalated` | 升级到终审仲裁但**未最终判定**（中间态）| 任意（仅中间态）|

**关键约束**：
- **AC-5**：最终产出中**无未清零的高严重度问题**。所有 high problem 的 final_status 必须 ∈ `{resolved, accepted_with_reason, rejected_with_reason}`。
- **禁止以 `escalated` 收尾**：`escalated` 仅作为中间态，终审仲裁后必须转为前三者。

## 五、通过判据（§4.1 第 5 条）

**高严重度问题清零**（低 / 中严重度可保留但需标注 + 给出 mitigation）。

清零定义：所有 high problem 的 `final_status ∈ {resolved, accepted_with_reason, rejected_with_reason}`。

- 低 / 中严重度 problem 可保留，但架构师必须在响应记录中标注保留原因 + mitigation。
- 终审裁决者对保留的 medium / low problem 有一致性复核权（避免堆积）。

## 六、检查清单

### 红队自检（产出 problem 时）
- [ ] severity 是否从 {high, medium, low} 三档中取值？
- [ ] severity_rationale 是否引用了本表的具体条目编号（如 high-③）？
- [ ] high-① 落位判定是否校验了最小形态三要素（REQ-ID 显式引用 + 需求要素逐条响应 + 落位结论）？
- [ ] final_status 首次产出是否填 `escalated`？

### 架构师自检（响应 problem 时）
- [ ] 接受的 problem 是否标注修订位置（章节 + 行号或锚点）？
- [ ] 拒绝的 problem 是否给出机制级理由（引用蓝图已有设计反证 / 引用 L2 REQ 边界反证 / 指出红队违反本准则，三类之一）？
- [ ] 拒绝时是否引用本表反证严重度（如"应降级 medium"）？

### 终审自检（复核严重度时）
- [ ] 是否以此表为唯一判据，未作主观调整？
- [ ] 裁决时是否明确引用条目编号（如 high-③）？
- [ ] 是否禁止以 `escalated` 收尾？

## 七、示例

### 示例 1：high-① 落位三要素缺失

红队 problem：
```json
{
  "problem_id": "P-R1-007",
  "dimension": "需求覆盖度",
  "description": "蓝图 §3.2 模块清单中虽在『用户管理模块』行提到 REQ-017，但未对其 acceptance_criteria（共 3 条）逐条响应；章节末尾无落位结论句。",
  "severity": "high",
  "severity_rationale": "依据 high-①：REQ-017 落位最小形态三要素中，要素 1（REQ-ID 显式引用）存在，但要素 2（需求要素逐条响应）和要素 3（落位结论）缺失，三要素不齐全判 high-① 成立。",
  "ref_requirement_id": "REQ-017",
  "final_status": "escalated"
}
```

架构师响应（接受）：
```json
{
  "problem_id": "P-R1-007",
  "response": "accepted",
  "revision_location": "§3.2 模块清单『用户管理模块』行 + 新增 §3.2.7『REQ-017 落位说明』子章节",
  "final_status": "resolved"
}
```

### 示例 2：架构师反证 high-⑤

红队 problem：
```json
{
  "problem_id": "P-R3-002",
  "dimension": "演化与失败",
  "description": "蓝图 §6.4 缺失『过载』失败模式。",
  "severity": "high",
  "severity_rationale": "依据 high-⑤：缺失关键失败模式『过载』。",
  "ref_requirement_id": "REQ-009",
  "final_status": "escalated"
}
```

架构师响应（拒绝 + 反证）：
```json
{
  "problem_id": "P-R3-002",
  "response": "rejected",
  "mechanism_reason": "拒绝依据：蓝图 §6.4.3 已覆盖『过载』失败模式（触发条件：QPS > 1000；应对策略：令牌桶限流 + 排队降级）。红队标 high-⑤ 不成立，应撤销。引用严重度准则表 high-⑤ 条目反证：三类失败模式齐全，无缺失。",
  "final_status": "accepted_with_reason"
}
```

> 后续：若红队复审不服 → 触发 §5.6 轮内迭代升级；终审仲裁时以此表为唯一判据。
