# 轮内迭代与终审判定

## 目的
本文档为架构设计师与终审裁决者提供**终审判定规则、REQ-ID 增补流程、死循环兜底、全局一致性校验、归档要求**的完整规则。这是整个 app 的"治理核心"——决定对抗何时收敛、何时强制终止。

> **v5 变更**：原 §一 轮内迭代机制、§二 升级判据、§三 终审轮内仲裁已删除——4 红队并行执行 + 红队校验角色统一判断后，不再存在单红队轮内迭代与升级。本档聚焦终审的终态判定与 REQ-ID 审批。

## 适用角色
- 架构设计师（承担重跑义务、申请 REQ-ID 增补、执行 M2 归档）
- 终审裁决者（行使全局一致性校验、死循环兜底、REQ-ID 审批、C2 Gate 自检、M1 terminated emit）

## 一、4 红队并行 + 红队校验角色（v5 新增）

### 1.1 拓扑结构
架构设计师校验 `blueprint_v1_ready` 后，FORK 扇出到 4 个红队（结构/边界/极限/质量）并行执行。4 个红队全部完成后 JOIN 汇入**红队校验角色**。

### 1.2 红队校验角色的职责
- 读取 4 个红队的问题清单
- 检查每份清单的 `findings` 数组
- 4 份全部为空 → `all_passed` → 终审裁决者
- 任一非空 → `issues_found` → 回退架构设计师（carries 携带 4 红队原产出）

### 1.3 红队的职责
红队不再判断路由 verdict，只产问题清单（无条件汇入红队校验角色）。

## 二、架构师重跑机制（v5 替代原"轮内迭代"）

### 2.1 触发条件
红队校验角色输出 `issues_found` → 架构设计师收到 carries（含 4 红队原产出物）→ 阅读问题清单修订蓝图。

### 2.2 重跑内容
架构师修完蓝图后，重新走 `架构设计师 → 架构设计师校验 → 4 红队并行 → 红队校验角色` 全链路。4 个红队**每次全跑**，不区分上次哪几个提了意见。

### 2.3 无 max_executions 上限
红队校验角色 → 架构设计师 这条边**不设 max_executions**，由主AGENT通过上下文感知自动识别死循环并终止。

> 设计理由：避免人为上限阻碍多次人工微调；架构设计可能需要多轮迭代才能收敛。

## 三、终审全局一致性校验（从红队校验角色 all_passed 进入）

### 3.1 触发条件
红队校验角色输出 `all_passed`（4 个红队的 findings 全部为空）→ JOIN 汇入终审裁决者 → 终审做全局一致性校验。

### 3.2 校验范围

终审裁决者读取 L2 + 蓝图 + 红队综合裁决：

- **蓝图内部一致性**：章节之间是否一致（如 §3 模块清单与 §5 文档层级冲突）
- **需求-蓝图追溯完整性**：L2 中每条 REQ-ID 是否在蓝图中有对应落位
- **红队综合裁决合理性**：红队校验角色的 all_passed 判定是否有遗漏（抽查关键章节）
- **严重度合理性**：复核严重度定级（以《严重度判定准则》为唯一判据）
- **M2 修复-跨轮对比**：可参考 `outputs/archive/run-{N}/` 历史归档目录做跨轮对比（架构师跨层回退时归档）

### 3.3 终审终态 verdict

| verdict | 触发条件 | 路由 |
|---------|---------|------|
| `passed` | 全局一致性 OK | → 完成（输出最终架构蓝图，AC-6）|
| `consistency_defect` | 发现一致性缺陷（**非死循环性质**）| → 架构设计师重跑（max_executions: 2）|
| `terminated` | 累计 ≥ 2 次一致性回退（死循环兜底）**OR** M1 修复（max_executions 耗尽）| → 完成（强制终止，标注未决项）|

> **§9 终审权限边界**：终审**无降级输出权**——要么全过输出、要么回退架构师重跑、要么强制终止。

## 四、终审一致性回退（§9.2）

### 4.1 触发条件

4 红队全过后终审发现**全局一致性缺陷**（非死循环性质，如蓝图章节之间不一致、需求-蓝图追溯断裂）。

### 4.2 处置（含 M2 修复-归档）

终审输出 `consistency_defect` → 架构设计师重跑。

回退路径：`终审 → 架构设计师 → 校验 → 4 红队 → 红队校验角色 → 终审`。

**重跑约束**：
- **M2 修复-归档强制**：架构师在覆盖固定路径前**必须**先将当前产出（`outputs/架构蓝图.md` + `process/outputs/R{1,2,3,4}-问题清单.json`）复制到 `outputs/archive/run-{N}/`（N 为全局重跑轮次计数，从 1 开始）。归档由架构师**主动执行**（skill 强制），保证 AC-4 severity 变化链可追溯。
- **不保留前次 v1**，从空白蓝图重起（§9.2）。
- **max_executions: 2**（§9.3 死循环兜底）。

> 理由：终审看的是全局一致性，单轮修订无法解决跨章节问题。

## 五、终审死循环兜底（§9.3）

### 5.1 触发条件

**累计发生 ≥ 2 次终审一致性回退**（即架构师重跑 2 次后终审仍发现一致性缺陷）。

**M1 修复补充触发**：检测到 dispatch.schema_constraints.verdict_enum 已不含 `consistency_defect`（即 max_executions 耗尽）。

### 5.2 处置

终审裁决者行使 **死循环兜底终止权**：
- 输出 `outputs/终审裁决书.json`，`verdict = terminated`。
- 标注所有未决项。
- 路由：→ 完成（强制终止）。

### 5.3 终审人工介入接口

本 app 不追求 100% 全自动，保留终审人工介入接口（终审裁决者可主动追加人工意见）。但 terminated 触发条件是**客观信号**（累计 ≥ 2 次一致性回退 OR M1 修复 max_executions 耗尽），不是主观判定。

## 六、REQ-ID 增补流程（§9.4）

### 6.1 问题背景

L2 在对抗期冻结，但红队做演化推理时可能发现"架构蓝图推演出的隐含边界约束"未在 L2 注册 REQ-ID。若无机制级路径，架构师只能强行塞入已有 REQ-ID（失真）或静默忽略（漏需求）。

### 6.2 增补流程

**入口**：架构设计师校验输出 `unregistered_requirement` → 终审裁决者（restrict_verdict 限 reqid_*）。

**业务流程**：
1. 架构师提交增补请求（在响应记录中标注 unregistered_requirement + 推演依据）
2. 终审审批（二选一）：
   - 批准 `reqid_approved` → 需求分析师执行 L2 解冻-写入-重冻结 → 架构师重跑全链路（F3-001）
   - 驳回 `reqid_rejected` → 架构师按现有 L2 完成 response
3. 路由与信号机制详见《SDK陷阱规避》

### 6.3 增补次数限制（M1 修复后）

**整个对抗期最多 2 次** REQ-ID 增补（避免 L2 反复变更）。

max_executions: 2（终审 → 需求分析师 的 `reqid_approved` 边）。

**M1 修复**：max_executions 耗尽后，SDK_SPEC §5.6 verdict_enum 动态过滤会从 schema enum 移除 `reqid_approved`（dispatch.schema_constraints.verdict_enum 收窄），此时终审**必须 emit `terminated`**（死循环兜底强制终止）。

### 6.4 约束

- **未走增补流程的隐含需求**：架构师若在蓝图中私自添加未注册的"REQ-XXX"标记，红队可直接判 fail（§5.2 F-001 修复约束）。
- **增补审批决议可见性**：审批决议对 4 个红队**均可见**（与 §5.5 信息隔离**不冲突**——增补审批属于 L2 基准变更，不属于"其他轮问题清单"）。
- **F3-001**：增补后所有已完成轮次自动失效，从 R1 重跑。

## 七、兜底全景图（§9.6，v5 更新）

| 兜底层级 | 触发条件 | 处置 | 是否输出蓝图 |
|---------|---------|------|------------|
| 红队校验角色回退 | 4 红队中任一发现问题 | 回退架构设计师（无上限，主AGENT感知终止）| 视重跑结果 |
| 终审一致性回退 | 4 红队全过后发现一致性缺陷 | 架构师重跑全链路（max 2）+ M2 修复归档 | 视重跑结果 |
| 终审死循环终止 | 累计 ≥ 2 次一致性回退 **OR** M1 修复（max_executions 耗尽）| terminated 强制终止 | **否**（标注未决项）|
| REQ-ID 增补（§9.4）| 对抗期发现隐含需求 | 终审审批 → L2 解冻-写入-重冻结 → C1 修复 confirmed+carries 信号 | 不直接影响（流程性）|
| L1→L2 失败（§9.5）| L1 充分性四要素缺失 | 需求澄清请求 + 占位 L2 → C1 修复 fail 边 | **否**（L2 未完整产）|

## 八、终审裁决书 schema

路径：`outputs/终审裁决书.json`，schema：

```json
{
  "result": {
    "verdict": "passed | consistency_defect | terminated",
    "summary": "终审结论摘要（≥ 50 字符，C2 Gate 自检要求）",
    "global_consistency_check": {
      "blueprint_section_consistency": "OK | DEFECT",
      "requirement_blueprint_traceability": "OK | DEFECT",
      "red_team_verdict_review": "OK | DEFECT",
      "severity_review": "OK | DEFECT"
    },
    "fallback_trigger_record": {
      "consistency_fallback_count": 0,
      "reqid_supplement_count": 0
    },
    "unresolved_items": []
  }
}
```

**verdict 取值约束**：
- `passed`：全局一致性 OK，输出最终架构蓝图（AC-6）。
- `consistency_defect`：发现一致性缺陷（非死循环），回退架构师重跑。
- `terminated`：死循环兜底，强制终止。

> 终审在 REQ-ID 审批路径中输出 `reqid_approved` / `reqid_rejected` 等中间态 verdict，但终态只能是上述三者之一。

## 九、C2 修复-终审 Gate 自检项

> **C2 根本原因已机制级修复**：原担心 _required_files 强制 3 个文件存在导致 fail 边死循环，现已在 orchestrator.py `_check_required_files` 中修复——`type=process` 的产出物不再强制存在性校验，多路径角色（如终审）各路径可独立产出。
>
> **APP 层补充自检**（仍保留，作为防御性双保险）：
> - `result.verdict` ∈ 当前上下文 restrict_verdict 范围
> - `result.summary` ≥ 50 字符
> - 必需字段齐全
>
> 详见 skill 自检项 "输出前" 段。

## 十、M1 修复-max_executions 耗尽后强制 emit terminated

> 详见《SDK陷阱规避》§跨层回退信号识别 M1 检测机制。本档不重复。

**业务要求概述**：终审每次执行时检测 `dispatch.schema_constraints.verdict_enum`，若已不含 `consistency_defect` 或 `reqid_approved`（max_executions 耗尽），立即 emit `terminated`。

## 十一、M2 修复-跨层回退归档流程

> **归档操作的唯一权威源：《归档策略》**（含完整 cp 命令清单、run-{N} 计数规则、AC-4 保证机制）。本档不重复。

**业务要求概述**：跨层回退触发时，架构师在覆盖固定路径前必须先将当前产出归档到 `outputs/archive/run-{N}/`。终审在第二次一致性校验时可读取 archive 目录做跨轮对比。

## 十二、检查清单

> **去重说明**：架构师自检详见 skill.md 自检项段；终审自检详见《SDK陷阱规避》§自检清单 + skill.md 自检项段。本节只列出**本文档独有的关键点**，不重复 skill 自检内容。

### 架构师关键点（重跑时）
- [ ] 收到 `consistency_defect` 重跑时，是否**先归档**（详见《归档策略》）再从空白蓝图重起？
- [ ] 收到 dispatch 注入的 REQ-ID增补审批.json 时，是否识别为"增补后重跑"并从 R1 重跑（F3-001）？
- [ ] 收到红队校验角色 `issues_found` 时，是否读取 4 红队原产出物了解所有问题？

### 终审关键点
- [ ] M1 修复：是否检测 verdict_enum 收窄 → emit terminated？
- [ ] M2 修复：是否参考 `outputs/archive/run-{N}/` 做跨轮对比（如归档存在）？
- [ ] 红队综合裁决合理性抽查（all_passed 是否有遗漏）？

### 示例 3（M2 修复）：跨层回退归档

终审第 1 次输出 `consistency_defect` → 架构师执行归档：

```bash
# 归档当前产出（M2 修复强制）
mkdir -p outputs/archive/run-1
cp outputs/架构蓝图.md outputs/archive/run-1/
cp process/outputs/R1-问题清单.json outputs/archive/run-1/
cp process/outputs/R2-问题清单.json outputs/archive/run-1/
cp process/outputs/R3-问题清单.json outputs/archive/run-1/
cp process/outputs/R4-问题清单.json outputs/archive/run-1/
# 然后从空白蓝图重起，覆盖固定路径
```

终审第 2 次一致性校验时，可读取 `outputs/archive/run-1/` 做跨轮对比（如对比第 1 轮的 severity 变化链是否在新轮中保留）。
