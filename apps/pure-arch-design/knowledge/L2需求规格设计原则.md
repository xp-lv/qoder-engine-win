# L2 需求规格设计原则

> 本文档是「需求分析师」角色产出 L2 需求规格文档时的设计指导。校验角色不读本文档，改读《L2需求规格校验清单》。
>
> 每条原则后的"Why"提示这条原则背后的需求工程依据。完整论证详见《L2需求规格编写指南》。

---

## 1. L2 必须含 REQ-XXX ID 体系

每条 requirement 必填六字段（req_id / title / description / acceptance_criteria / priority / source_ref），字段约束、正则、唯一性、可证伪要求详见《L2需求规格编写指南》§二（唯一权威源）。

**Why**：L2 是 Build 资产，下游反复查询。REQ-ID 是追溯的身份证号——丢了或重复了，架构师/红队/终审交流时指向不明，整个对抗机制失效。

## 2. L1 充分性四要素检查必做

目标 / 角色（≥2 个含 1 个产出角色）/ 边界 / 验收（≥1 条可证伪信号）。四要素任一缺失即判 L1 不充分，**禁止强行产出完整 L2**。

**Why**：L1 信息量不足时，L2 无论怎么写都包含虚构内容——需求分析师不是用户，不能凭空猜测业务意图。强行产出 = 把猜测当作需求 = 后期发现错误时代价极高。

## 3. L1 不充分时的处置

产出需求澄清请求（Markdown）+ 占位 L2。用户在 manual confirm 时看到"缺什么"→ reject → 回到自身补充 L1。

**Why**：返回机制比猜测机制便宜。澄清成本 = 用户 5 分钟，猜测成本 = 下游数小时的修正。

## 4. L2 完整性自检必做

六字段齐全 + req_id 唯一 + acceptance_criteria 每条可证伪 + source_ref 双向可追溯 + 无模糊词。自检不通过禁止提交。

**Why**：L2 质量差的代价不对称——L2 上省 1 小时，下游要付 100 小时。自检是最后的便宜防线。

## 5. 禁用模糊词

禁用"合理""适当""较快""足够""良好"等无法量化的词；acceptance_criteria 每条必须可证伪。

**Why**：模糊词 = 不可证伪 = 主观解读 = 红队/架构师各说各话。Karl Popper："无法被证伪的命题是信仰，不是科学。"

## 6. L2 对抗期冻结

confirmed 后冻结不可改。REQ-ID 增补须经终审审批走增补流程（L2 解冻-写入-重冻结，版本号 +1），整个对抗期最多 2 次增补。

**Why**：对抗机制依赖"基准 vs 设计"比较。基准动 = 所有测量失效。冻结 = 保证参考系不动。让解冻"昂贵"（需终审审批）= 防止随手解冻。

## 7. REQ-ID 增补 L2 更新规则

仅在 requirements 数组追加新 REQ-ID（现有最大编号 +1），**不可修改既有 REQ**。L2 版本号 +1。

**Why**：修改既有 REQ = 破坏下游所有引用（架构师/红队/响应记录都引用原 REQ）。不变性原则（Immutability）保证历史追溯链不断裂。

## 8. 推断字段必标

L1 含歧义但充分时，基于合理推断补全 L2 必须在 front-matter 标注 `inferred_fields`（列出推断字段及推断依据）。

**Why**：未标注的推断 = 被当成事实。后期发现错误时无法分辨"这是用户要求的"还是"这是需求分析师猜的"。标注 = 让推断可被质疑。

## 9. L2 ↔ L1 双向可追溯

每条 requirement 的 source_ref 必须引用 L1 中的具体章节且能在 L1 中 grep 命中。

**Why**：正向防"需求丢失"，反向防"需求分析师虚构需求"。这是 IEEE 830-1998 需求标准的核心要求。
