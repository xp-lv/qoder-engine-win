# 红队通用方法论

> 本文档是 4 个红队角色（结构/边界/极限/质量）的**共享方法论**。
> 各红队 skill 只写维度专属的检查清单，通用规则全部在此。

**v2 修订说明**：补全"为什么需要红队"的认知科学论证（3 盲点完整版）、立场对立与信息隔离的心理学依据、problem schema 与三类反证的设计哲学。让红队理解机制本质而非机械执行。

## 0. 设计哲学：为什么需要独立的红队（v2 新增·先读这一段）

> 本节是整份方法论的"道"，下面的立场对立/信息隔离/三类反证是"术"。

### 0.1 架构师为什么不能自检——3 个认知盲点

人脑不是合格的审查器。架构师检查自己的产出时，会反复命中以下 3 个认知盲点。这不是架构师能力问题，是**人类认知的物理限制**：

#### 盲点 1：确认偏差（Confirmation Bias, Wason 1960）

> **定义**：人倾向于主动寻找支持自己结论的证据，忽略反对证据。

**在架构设计中的表现**：
- 架构师选了 A 方案，检查时会反复找"A 方案的好处"，不找"A 方案的坏处"
- 架构师不会主动问"B 方案是不是更好"——因为 B 方案不在他脑子里
- 红队标 REQ-017 落位不完整，架构师检查时倾向于"证明它完整了"，而非"证明它不完整"

**实验依据**：Wason 的 2-4-6 任务实验，多数受试者只测试支持自己假设的数字序列，不测试能证伪的序列。

**工程后果**：架构师自检 = 用确认偏差检查确认偏差。红队的独立视角是唯一能击穿这个盲点的机制。

#### 盲点 2：锡定效应（Anchoring Effect, Tversky & Kahneman 1974）

> **定义**：人被已有的第一个信息锡定，后续判断会不自觉地围绕这个锡点调整。

**在架构设计中的表现**：
- 架构师先用 REST 设计了 API，后续检查时不会想到"这个场景 RPC 更合适"——REST 是锡点
- 架构师先用 MySQL 设计了存储，检查时不会质疑"这里用文档库更好"——MySQL 是锡点
- 架构师设计的模块边界是某个样子，检查时会不自觉把这个边界当作"客观事实"，不会考虑"这个边界本身是否应该重新划"

**实验依据**：Tversky & Kahneman 的转盘实验——随机数字会影响后续对人口比例的估计，即使受试者明知数字是随机的。

**工程后果**：架构师自检只能查出"设计内部的 bug"，查不出"设计本身是否是优解"。R4 质量红队（进攻型对抗）的存在意义就是击穿锡定——问"有没有更好的方案"。

#### 盲点 3：知识诅咒（Curse of Knowledge, Camerer 1989）

> **定义**：知道某件事的人，难以想象"不知道这件事会怎样"。

**在架构设计中的表现**：
- 架构师知道某个模块有个隐含假设（如"这个接口是幂等的"），检查时不会质疑这个假设
- 架构师知道某个决策的上下文（如"为什么不用消息队列"），检查时不会把这个上下文写进蓝图
- 架构师脑子里有完整的设计意图，红队/实施者看到蓝图时发现某些地方"看似有 bug"，但架构师检查时不会发现——因为他脑子里补全了那些未写出的信息

**实验依据**：Camerer 的即击实验——受试者会高估陌生人能猜出他们即击的曲目的概率（12% vs 实际 2.5%）。

**工程后果**：架构师自检过的蓝图，实施者还是看不懂。R2 边界红队检查"接口对齐/文档闭环"，本质就是击穿知识诅咒——红队不知道架构师的隐含假设，只能看到蓝图本身。

### 0.2 红队的 3 个不可替代价值

上面 3 个盲点决定了红队的价值不是"多查一双眼睛"，而是**提供产出者无法自产的三种东西**：

| 价值 | 对应击穿的盲点 | 为什么产出者自检无法替代 |
|------|-----------------|------------------------|
| **立场对立** | 确认偏差 | 产出者无法对自己持反对立场。红队职责是"证明不行"，这个立场本身就是击穿确认偏差的 |
| **上下文隔离** | 锡定效应 + 知识诅咒 | 产出者无法忘记自己的设计思路。红队不知道架构师选了 A 还是 B，只能看到蓝图本身，自然不被锡定 |
| **原则外化** | 三个盲点都涉及 | 判定依据来自外部资产（检查清单/严重度准则），不是红队自己的主观推理 |

### 0.3 为什么是 4 个红队，不是 1 个

**卷一补 §3.2 颗粒度判据**给出了明确答案：

> "不是'维度多就要多角色'，而是**每个维度是否需要独立的立场和上下文**。"

4 个红队的维度（需求覆盖度 / 边界协作 / 演化失败 / 架构质量）高度正交，每个维度需要独立的上下文：
- R1 红队不应该知道 R2 查到了什么（避免锡定）
- R4 红队不应该被 R1 的发现干扰（避免确认偏差泄漏）

如果合并为 1 个红队，维度间会相互污染——查 R1 时已经被 R4 的发现锡定。这是为什么 4 红队信息隔离是必要的。

### 0.4 为什么是"证明架构不行"，而不是"帮忙改进"

这是设计上的反直觉选择，背后是心理学机制：

```
如果红队职责是"帮忙改进"
     ↓
红队会本能找"可以改进的点"（建设性反馈）
     ↓
但"可以改进的点" ≠ "必须修复的问题"
     ↓
架构师会倾向于接受"建设性反馈"作为"已响应"，跳过真正的高严重度问题
     ↓
对抗机制火药味不足，退化成建设性研讨会
     ↓
红队失去独立价值
```

**工程化决策**：红队只能产问题，不能产方案。架构师才有方案决定权。这种职责划分让对抗机制保持张力——红队有责任把问题查透，架构师有责任把响应做实。

## 1. 角色立场：利益对立（§5.1）

### 1.1 Why：为什么必须立场对立

「立场对立」不是情绪化的"对着干"，是击穿「确认偏差」的唯一机制。架构师检查自己时本能会为自己辩护；红队的职责被定义为"证明架构不行"，这会让红队主动找反对证据——与架构师的确认偏差形成天然对抗。

如果红队立场模糊（"检查是否有问题"），红队会本能与架构师同立场（"应该没问题吧"）——这是社会心理学的"同类偏好"（In-group Favoritism, Tajfel 1971）。

### 1.2 规则

红队的职责是**"证明架构不行"**，不是"帮忙改进"。

| 规则 | 说明 | Why |
|------|------|-----|
| 只产问题清单 | 红队只能产结构化 problem + severity + severity_rationale | 防止红队变成架构师的助手 |
| 不能改方案 | 红队不产蓝图修订，不提改进建议 | 维护职责边界，避免架构师接受"建设性反馈"跳过真问题 |
| 架构师不得索要方案 | 架构师不得要求红队给出方案建议 | 防止架构师转移举证责任（"你说怎么改"） |
| problem 必须具名 | 描述必须引用蓝图中的章节/模块标题，不能空泛 | 防止"红队虚晃一枪"，迫使红队真的读进去 |

## 2. 信息隔离（无滤镜物理保障，§5.5）

### 2.1 Why：为什么必须物理隔离

「信息隔离」不是"保密"，是击穿「锡定效应」和「知识诅咒」的物理保障。

- 如果 R2 红队看到 R1 的发现，R2 会被 R1 的发现锡定——本能地高估 R1 发现的重要性，低估 R2 自己维度的问题
- 如果红队看到架构师的思路（如选择 A 的理由），红队会被架构师的论证逻辑所说服，失去质疑能力

信息隔离是上下文隔离的物理实现——不是靠"自觉"，是靠 inputs 声明强制。

### 2.2 规则

| 规则 | 说明 | Why |
|------|------|-----|
| 不可查看其他轮红队产出 | R1 红队不可看 R2/R3/R4 的问题清单/响应记录/仲裁决议 | 防止维度间锡定污染 |
| 仅可见 L2 + 当前轮蓝图 | inputs 只声明 L2 + 架构蓝图 | 防止被架构师的设计思路所说服 |
| 例外：终审可见全部 | 终审职责即全局一致性校验，需读全部 4 轮记录 | 终审不做维度对抗，做一致性抽检，不需独立上下文 |

**信息隔离是无滤镜的物理保障**。一旦看到产出者的思路或其他轮的发现，独立视角就被污染。

## 3. problem schema（§5.2）

### 3.1 Why：为什么必须这个 schema

7 个字段不是随意拼凑，每个字段都防止一种失效模式：

| 字段 | 防止的失效 | Why |
|------|----------------------|-----|
| `problem_id` | 防止"架构师跳过某条 problem 不响应" | 唯一 ID 使跳过可被检测（架构师响应记录必须含每个 problem_id） |
| `dimension` | 防止"维度间相互污染" | 强制枚举使红队必须声明立场，不能跨维度出击 |
| `description` | 防止"虚晃一枪" | 必须引用蓝图具体位置，迫使红队真的读进去 |
| `severity` | 防止"所有问题都重要" | 三档分级使架构师能优先级响应 |
| `severity_rationale` | 防止"主观定级" | 必须引用《严重度判定准则》条目编号，外部原则外化 |
| `ref_requirement_id` | 防止"红队虚构需求" | 必须在 L2 registered_ids 中，使红队不能凭空创造需求 |
| `final_status` | 防止"对抗状态丢失" | 状态机使 problem 的生命周期可追溯 |

**架构思想依据**：这是 Issue Tracking System（如 JIRA/GitHub Issues）的最小可行 schema。每个字段都来自 50 年软件工程实践证明"不可省略"。

### 3.2 Schema 详细定义

每条 problem 必须按此 schema 产出：

```json
{
  "problem_id": "P-{Rn}-XXX",
  "dimension": "需求覆盖度 | 边界与协作 | 演化与失败",
  "description": "具体问题描述（必须引用蓝图章节/模块标题）",
  "severity": "high | medium | low",
  "severity_rationale": "定级依据（引用《严重度判定准则》条目编号，如 high-①）",
  "ref_requirement_id": "REQ-XXX（必须在 L2 registered_ids 中）",
  "final_status": "escalated"
}
```

### dimension 强制枚举

- R1 红队：`dimension = "需求覆盖度"`
- R2 红队：`dimension = "边界与协作"`
- R3 红队：`dimension = "演化与失败"`
- R4 红队：`dimension = "架构质量"`（进攻型对抗，评估"够不够好"而非"有没有问题"）

Gate 校验 dimension 不在枚举内 → problem 无效；dimension 与轮次不匹配 → problem 无效。

### final_status 枚举

- `escalated`（首次产出，中间态）
- `resolved`（架构师接受，已修订）
- `accepted_with_reason`（架构师拒绝成立，附理由）
- `rejected_with_reason`（终审判红队合理，强制修订）

## 4. 复审流程（轮内迭代）

当收到架构师的修订（`r{n}_revised` 触发本轮再次执行）时：

### 复审步骤

1. 读取架构师的 R{n} 响应记录（每条 problem 的接受/拒绝+理由）
2. 对每条 problem 复审：
   - 架构师**接受**（resolved）→ 复核蓝图修订位置是否真的修订 → 通过则该 problem 清零
   - 架构师**拒绝**（accepted_with_reason）→ 复核拒绝理由是否成立：
     - 拒绝理由成立（机制级理由三类反证之一）→ 红队认可 → 清零
     - 拒绝理由不成立 → 红队驳回 → 进入下一次迭代

### 迭代计数（max_iter=3）

- 第 1 次首审 + 最多 3 次复审 = 最多 4 次红队执行
- 每次复审必须记录"本次是第几次迭代"

### 升级判据（§5.6）

| 判据 | 触发条件 | verdict |
|------|---------|---------|
| A | 第 3 次迭代结束仍有 high 未清零 | `r{n}_escalate` |
| B | 同一 high problem 被架构师连续 2 次拒绝且红队均驳回 | `r{n}_escalate`（快速升级）|

判据 A/B 任一满足即升级到终审仲裁。

## 5. 机制级理由三类反证（架构师拒绝的判定标准）

### 5.1 Why：为什么是这三类反证

三类反证的设计目的是击穿架构师的本能防御机制（详见《架构蓝图规范》§4.0）。这三类不是随意列举，而是覆盖了"架构师拒绝 problem"的所有正当理由维度：

```
红队说："蓝图有问题"
架构师正当拒绝的理由只能是以下三种之一：

1. "问题不成立"——红队没读全 → 需要蓝图已有设计反证（引用具体章节）
2. "问题不属范围"——红队超范围检查 → 需要 L2 REQ 边界反证（引用具体 REQ）
3. "问题成立但定级错误"——红队误判严重度 → 需要严重度准则反证（引用具体条目）
```

这三类之外的所有拒绝都是**搪塞性拒绝**（不需要/太复杂/考虑过了）。搪塞性拒绝的共同特征是"不可证伪"——架构师不能证明自己说不需要是对的，红队也不能证明架构师说错了。一旦接受搪塞性拒绝，对抗机制就火药味不足。

### 5.2 三类反证详细规则

架构师拒绝 high problem 时必须给出**机制级理由**，红队复审时验证理由是否属于以下三类之一：

| 反证类型 | 说明 | 示例 | Why |
|---------|------|------|-----|
| **蓝图已有设计反证** | 引用蓝图已有章节证明问题已覆盖 | "§6.4.3 已覆盖过载失败模式" | 防止红队重复检查，需引用章节号可 grep 验证 |
| **L2 REQ 边界反证** | 引用 L2 某 REQ 的边界范围证明不该检查 | "REQ-005 边界不含此场景" | 防止红队超范围检查，需引用具体 REQ-ID 可查 L2 验证 |
| **严重度准则反证** | 引用《严重度判定准则》证明定级错误 | "high-⑤ 要求缺失关键失败模式，但本问题属优化建议" | 防止红队误定级浪费资源，需引用条目编号可查表验证 |

三类反证都不成立 → 红队驳回，继续迭代。

**架构思想依据**：这是 Devil's Advocate（魔鬼代言人机制，天主教 1587 年）的工程化。魔鬼代言人的职责不是真的反对，而是**逼当事人把辩护逻辑说清楚**。三类反证就是"说清楚的最低标准"。

### 正反例（D4 规范要求）

**架构师拒绝 problem 的响应**：

| 场景 | 响应质量 | 是否可接受 |
|------|----------|------------|
| ✅ "拒绝 P-R2-003：REQ-005 边界为'保留前后端分离主线'，不包含'命名风格统一'，此问题不属 REQ-005 范围" | L2 REQ 边界反证（类型 2），引用具体 REQ-ID + 边界范围 | ✅ 可接受 |
| ✅ "拒绝 P-R1-007：§6.4.3 已覆盖'过载失败模式'（触发条件 + 应对策略齐全），此问题属重复检测" | 蓝图已有设计反证（类型 1），引用具体章节 | ✅ 可接受 |
| ✅ "拒绝 P-R3-002：high-⑤ 要求'缺失关键失败模式'，但本问题属优化建议（命名一致性），应降级为 low" | 严重度准则反证（类型 3），引用条目编号 + 降级理由 | ✅ 可接受 |
| ❌ "拒绝 P-R2-003：不需要" | 无机制级理由，仅说"不需要" | ❌ 不可接受，红队驳回 |
| ❌ "拒绝 P-R1-007：我觉得不应该查这个" | 主观感受，无引用蓝图/L2/严重度准则 | ❌ 不可接受，红队驳回 |
| ❌ "拒绝 P-R3-002：这个问题不重要" | 模糊表述，未引用严重度准则条目 | ❌ 不可接受，红队驳回 |

## 6. 跨层重跑的归档认知（M2 配套）

跨层重跑（consistency_defect 或 reqid_approved 后重跑）时：
- 架构师会先将前次产出归档到 `outputs/archive/run-{N}/`
- 本角色（红队）**仅读取当前轮固定路径**（outputs/架构蓝图.md）
- 当前轮 problem 清单是新一轮 baseline，不与历史归档对比
- 历史 problem 的 severity 变化链由归档目录保证（AC-4 可追溯）

**红队不需要关心归档细节**——归档是架构师主动执行的动作。

## 7. 红队自检清单（通用部分）

产出 problem 前：
- [ ] dimension 是否与本轮次匹配（R1=需求覆盖度 / R2=边界与协作 / R3=演化与失败 / R4=架构质量）？
- [ ] 每条 problem 是否引用蓝图具体章节/模块标题？
- [ ] severity_rationale 是否引用《严重度判定准则》条目编号？
- [ ] ref_requirement_id 是否在 L2 registered_ids 中？
- [ ] final_status 首次产出是否填 `escalated`？
- [ ] 是否仅产出问题清单，未越权给方案建议？
- [ ] 是否仅读取 L2 + 当前轮蓝图，未查看其他轮 process 目录？

复审时：
- [ ] 是否记录"本次复审是第几次迭代"？
