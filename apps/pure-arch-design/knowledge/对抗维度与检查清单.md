# 对抗维度与检查清单

## 目的
本文档为 4 轮红队（结构红队 R1 / 边界红队 R2 / 极限红队 R3 / 质量红队 R4）提供**正交递进**的对抗维度定义、检查清单范围、dimension 枚举与轮次匹配规则、红队信息隔离约束、利益对立原则。R1/R2 串行递进，R3/R4 并行执行（维度正交），前序轮未过则后序轮不启动。

> **v4.1 变更**：R3 与 R4 并行执行 + JOIN 汇入语义修复。R3/R4 的轮内 max_iter=3 用尽不再升级到终审仲裁（避免破坏 JOIN 的 AND 语义），而是直接 emit `terminated` 走向终束。

**v5 修订说明**：补全 4 维度正交性的拓扑依据、R4 进攻型对抗的认知科学依据、4 红队与软件工程质量模型的对应关系。让红队理解"为什么是这 4 个维度"而非机械执行检查项。

---

## 设计哲学：为什么是 4 个维度（v5 新增·先读这一段）

### 4 个维度的正交性论证

这 4 个维度不是随意拆分，是**软件架构质量的 4 个独立失效维度**。所谓"正交"，意思是：

> 任一维度的满分 ≠ 其他维度的得分。需求覆盖满分不代表边界协作满分。

这 4 个维度覆盖了架构可能失效的 4 个独立层面：

```
架构有 4 种独立失效模式：

1. 需求层面失效：需求没有被设计进入（需求丢失 / 伪落位）
     ↓ R1 检查
2. 协作层面失效：需求被设计了，但模块之间说不清（责任真空 / 接口断裂）
     ↓ R2 检查
3. 演化层面失效：设计能跑，但不能扩展 / 不能失败（扩展性不足 / 无降级）
     ↓ R3 检查
4. 质量层面失效：设计能跑能扩展，但不够好（过度复杂 / 过度工程 / 不优秀）
     ↓ R4 检查
```

这 4 个层面是"递进深化"的——前者是后者的前提（需求都没覆全，谈不上协作）。但它们不是包含关系——一个架构可以需求满分、协作满分、演化满分，但质量不及格（设计能跑但很烂）。

### 为什么 R3/R4 可以并行，R1/R2 不能

**判定依据**：维度之间是否依赖前者的发现。

- R2（边界协作）依赖 R1（需求覆盖）的发现——如果需求都没覆全，查"模块是否协作"没意义。所以 R1/R2 必须串行。
- R3（演化失败）不依赖 R2 的发现——演化能力是架构自身的属性，与边界协作是否满无关。
- R4（架构质量）不依赖 R3 的发现——质量评估也是架构自身的属性。
- R3 与 R4 维度正交（演化是时间维度，质量是设计维度），可并行。

这是工程编排上的优化——并行节省成本，但必须先论证正交性。不能为了并行而并行。

### 为什么 R4 是"进攻型对抗"，与前 3 轮不同

前 3 轮（R1/R2/R3）是**防守型对抗**——查"有没有问题"。R4 是**进攻型对抗**——问"够不够好"。这个区分不是语言划分，是认知科学上的两种独立思维模式：

| 对抗类型 | 问题范式 | 对应的认知模式 | 击穿什么 |
|---------|---------|--------------|----------|
| **防守型（R1/R2/R3）** | "这里有没有 bug?" | 检查思维（search for errors） | 击穿「确认偏差」——架构师忘了查什么 |
| **进攻型（R4）** | "有没有更好的方案?" | 重设计思维（search for alternatives） | 击穿「锡定效应」——架构师被自己的设计锡定 |

R4 的存在是击穿锡定效应。如果只有 R1/R2/R3，架构师可能产出了一个"能跑 / 能扩展 / 没漏洞"的架构，但这个架构远非最优——架构师被他的第一个设计锡定，从不质疑"这个设计本身是否是优解"。

R4 强制问"够不够好"，迫使架构师重新考虑"有没有更好的方案"。这是 R4 检查项（模块化质量 / 复杂度控制 / 可演进性 / 自顶向下递进）的核心——这些都是"优秀设计"的标准，不是"没 bug"的标准。

### 4 红队与软件工程质量模型的对应

本 app 的 4 维度不是生造的，对应了软件工程主流质量模型：

| 红队 | 对应的质量模型 |
|------|-----------------|
| R1 需求覆盖度 | IEEE 830-1998 需求标准 / Traceability Matrix |
| R2 边界与协作 | ISO/IEC 25010:2011 「互操作性」+ 「适应性」|
| R3 演化与失败 | Resilience Engineering（Resilience Engineering, Hollnagel 2006）|
| R4 架构质量 | Clean Architecture（Martin 2012）+ Domain-Driven Design（Evans 2003）|

这些质量模型在软件工程领域被广泛接受。4 红队的维度选择 = 这些质量模型的核心维度的工程化拆解。

## 适用角色
- 结构红队（R1，dimension = 需求覆盖度）
- 边界红队（R2，dimension = 边界与协作）
- 极限红队（R3，dimension = 演化与失败）
- 质量红队（R4，dimension = 架构质量）

## 一、维度正交递进总览（v4：4 轮，R3/R4 并行）

| 轮次 | 红队 | dimension（必填枚举） | 检查范围（一句话） |
|------|------|----------------------|-------------------|
| R1 | 结构红队 | `需求覆盖度` | 每条 REQ-XXX 是否在蓝图有落位？落位是否满足 high-① 最小形态？有无遗漏？ |
| R2 | 边界红队 | `边界与协作` | RACI 清晰？并行兼容性？接口对齐？命名冲突？文档层级闭环？ |
| R3 | 极限红队 | `演化与失败` | 极端场景？扩展能力？三类失败模式（单点失效/依赖不可用/过载）覆盖？ |
| R4 | 质量红队 | `架构质量` | 模块化质量？复杂度控制？可演进性？自顶向下递进？资产化？文档族可扩展？技术选型？数据流清晰？ |

**正交原则**：同一 REQ-XXX 在 R1 以「需求覆盖度」被审，到 R2 必以「边界与协作」被审，到 R3 必以「演化与失败」被审，到 R4 必以「架构质量」被审——dimension 必然变化。R3 和 R4 维度正交，可并行执行。

## 二、R1 结构红队检查清单（dimension = 需求覆盖度）

### Why 这个维度存在（v5 新增）

R1 检查的是**需求 → 设计的追溯链**。这是软件工程最早被识别的失效模式——1998 年 IEEE 830 标准就把"需求可追溯"列为强制要求。

需求覆盖度失败的后果是"需求被静默丢失"——架构师觉得某些需求不重要就不做，但用户依然会期望这些需求被实现。这是项目后期发现"这个功能去哪了"的唯一根因。

### 检查项 R1-A：每条 REQ-XXX 是否在蓝图中有效落位
对 L2 front-matter 中的每个 `req_id`，遍历架构蓝图，判定**最小落位形态三要素是否齐全**（任一缺失即判 high-①）：

1. **REQ-ID 显式引用**：蓝图对应章节正文中**显式出现**该 REQ-ID 字符串（如"本模块承担 REQ-017"）。grep 蓝图必须命中。
2. **需求要素逐条响应**：该章节必须对该 REQ 的每条 `acceptance_criteria`（L2 §3.3）逐条给出"接受 / 转化 / 反证"之一，**不能跳过**。
3. **落位结论**：该章节末尾必须有落位结论句（如"REQ-017 已由本模块 §X.Y 完整承担"）。

> 三者齐全才算有效落位；任一缺失判 high-① 成立。这消除了"加一行 REQ-ID 即文字层面落位"的伪落位空间。

### 检查项 R1-B：是否有未注册的 REQ-XXX 标记
架构师若在蓝图中私自添加未在 L2 front-matter 注册的"REQ-XXX"标记，**直接判 fail**（依据 §5.2 F-001 修复约束 + §9.4：未走增补流程的隐含需求）。

### 检查项 R1-C：是否有需求遗漏
L2 中存在 REQ-XXX，但蓝图中完全找不到对应章节承担 → high-①（无任何落位）。

### 检查项 R1-D：双向可追溯
对每个 REQ-ID，蓝图应有命中（正向）；反向，蓝图每个落位应能反查到 REQ-ID（反向）。任一方向断裂判 high-① 或 medium-③。

## 三、R2 边界红队检查清单（dimension = 边界与协作）

### Why 这个维度存在（v5 新增）

R2 检查的是**模块之间的协作接口**。这是 Conway's Law（康威定律，1968）的延伸——系统架构会镜像组织沟通结构。如果架构层的边界不清，实施层的协作就会混乱。

边界协作失败的后果是"集成阶段崩盘"——每个模块独立看都正常，但拼起来就出问题。这是软件工程中"集成陷阱"（Integration Trap）的根源。

### 检查项 R2-A：RACI 清晰度
- 蓝图章节③「模块清单 + RACI」中，每个关键产出物必须有 **Responsible**（R）角色。
- 责任真空（关键产出物无 R）→ high-③。
- 同一产出物出现 ≥ 2 个 Responsible（R 责任不清）→ high-③。
- 不可达角色（RACI 中列出的角色在模块清单中不存在）→ medium-① 或 high-①（视严重度）。

### 检查项 R2-B：并行兼容性
- 蓝图章节④「并行兼容性」中，若存在并行角色读写同一资源：
  - 有锁/有序约束 → 通过
  - 无锁/无序 → high-④（并行兼容性破坏）
- 并行扇出 + 同步汇入的 input_groups 是否合理（不能形成死等）。

### 检查项 R2-C：接口对齐
- 跨模块/跨角色的接口（输入参数名、类型、产出物路径）是否一致。
- 参数名/类型对不上但不影响主链路 → medium-③。
- 主链路接口断裂 → 升级为 high-②（蓝图内部自相矛盾）或 high-①。

### 检查项 R2-D：命名冲突
- 同一对象（模块/角色/产出物）在不同章节出现 ≥ 2 种命名。
- 可消歧但不一致 → medium-④。
- 无法消歧导致多种实现 → high-②（蓝图内部自相矛盾）。

### 检查项 R2-E：文档层级闭环
- 蓝图章节⑤「文档层级」中，每层文档必须有对应产出角色。
- 文档层级缺失或不闭环 → medium-②。
- 文档链断裂导致无法追溯 → high-②。

## 四、R3 极限红队检查清单（dimension = 演化与失败）

### Why 这个维度存在（v5 新增）

R3 检查的是**架构的鲁棒性与可扩展性**。这是 Resilience Engineering（霍纳格尔 2006）的核心维度——架构不仅要能跑，还要能应对压力、能扩展、能从失败中恢复。

演化失败失败的后果是"上线即崩"——架构能跳起来，但遇不上现实世界的复杂场景。这是"实验室架构"与"生产架构"的根本区别。

### 检查项 R3-A：极端场景覆盖
- 蓝图是否考虑了边界输入（空集 / 极大值 / 极小值 / 异常字符）。
- 是否考虑了角色死锁（循环依赖 / 资源争用）。
- 缺失关键极端场景 → high-⑤（缺失关键失败模式）。

### 检查项 R3-B：扩展能力评估
- 蓝图描述的架构在需求规模扩大 2-3 倍时是否仍可行。
- 角色清单是否可线性扩展（增加新角色不破坏既有拓扑）。
- 扩展能力不足但当前可运行 → medium-①（涉及 ≤ 2 模块的边界缺失）。

### 检查项 R3-C：失败模式覆盖（AC-7 强制三类）
蓝图章节⑥「目标架构的失败模式与应对」**必须**覆盖以下三类（任一缺失判 high-⑤）：

1. **单点失效**：单一组件/角色失效导致整链路瘫痪的应对。
2. **依赖不可用**：上游依赖（外部服务 / 上游角色产出物）不可用时的降级或重试策略。
3. **过载**：输入流量超出处理能力时的限流 / 排队 / 拒绝策略。

每类失败模式必须含**触发条件** + **应对策略**两段（AC-7 可证伪）。

> F-009 消歧：蓝图章节⑥明确为**目标架构（蓝图描述对象）的失败模式**，不是本 app 对抗流程的失败模式，也不是运维手册。

## 四-R4、R4 质量红队检查清单（dimension = 架构质量，进攻型对抗）

> **R4 定位**：与 R1/R2/R3 的防守型对抗不同，R4 是**进攻型对抗**——不查"有没有问题"，而是评估"够不够好"。R4 采用**一票否决**（任一 high 未清零即不通过）。

### Why 这个维度存在（v5 新增）

R4 检查的是**架构设计本身的质量**。这是击穿「锡定效应」的唯一机制——架构师被自己的设计锡定后，不会质疑"这个设计本身是否是优解"。

R4 的 8 个检查项（模块化/复杂度/可演进性/自顶向下/资产化/文档族/技术选型/数据流）对应了主流架构方法论的核心原则：
- 模块化、复杂度、可演进性 → Clean Architecture（Martin 2012）
- 自顶向下 → DDD（Evans 2003）的战略设计
- 资产化 → 卷一补 §1 「资产优先原则」
- 文档族可扩展 → Documentation-Driven Design
- 技术选型、数据流 → The Pragmatic Programmer（Hunt & Thomas 1999）

R4 失败的后果是"能跑但不优秀"——架构能跳起来，但不具备长期演进能力。这种架构 1-2 年后会变成逼着重写的"这代码太烂了"类型。

### 检查项 R4-A：模块化质量（角色即能力 + 高内聚低耦合）
- **每个模块是否有独特能力**：模块仅承担检查/转发，无独立知识/技能 → high-⑦（"角色即能力"违反）
- **模块间是否松耦合**：模块间直接依赖（非接口通信） → high-⑦
- **模块内是否高内聚**：单个模块承担 ≥ 3 类不相关职责 → high-⑦
- **模块职责数是否合理**：单模块承担 > 5 个 REQ-ID → medium

### 检查项 R4-B：复杂度控制（KISS/YAGNI + 重点占比 ≥ 60%）
- **系统分层是否精简**：分层 > 4 层且无充分理由 → high-⑧
- **模块总数是否合理**：模块数 > REQ 数 × 2（过度拆分） → medium
- **是否有不必要的抽象层**：存在仅转发无逻辑的中间层 → high-⑧
- **核心业务模块占比是否 ≥ 60%**：核心业务模块（直接承担 REQ 的模块）/ 总模块数 < 60% → high-⑧

### 检查项 R4-C：可演进性（无代码接入 + 渐进式交付）
- **是否支持无代码接入新模块**：新增模块需修改 ≥ 3 个既有模块 → high-⑨
- **是否支持渐进式分阶段交付**：无分阶段交付路径（全有或全无） → medium
- **角色清单是否可线性扩展**：新增角色破坏既有拓扑 → medium
- **是否预留扩展点**：无明确的扩展点声明 → medium

### 检查项 R4-D：自顶向下递进（总体→层→模块→文档）
- **系统总体 → 层 → 模块 → 文档 是否层层递进**：上层未推出下层（跳跃推导） → high-⑨
- **上层是否是下层的唯一输入依据**：模块设计引用了非上层的依据 → high-⑨
- **架构充分性判据**：具体功能无法嵌入现有层/模块（需强行修改） → medium

### 检查项 R4-E：资产化质量（资产优先 + 避免重复造轮子）
- **是否识别可复用资产**：重复逻辑未沉淀为可复用资产 → medium
- **是否避免重复造轮子**：同一逻辑在多个模块重复实现 → high-⑦
- **编排节点是否最小化**：存在仅做转发无增值的编排节点 → medium

### 检查项 R4-F：文档族可扩展性（生长规则 + 开放层/封闭层）
- **文档体系是否支持动态生长**：文档清单是固定预设，无生长规则 → medium
- **是否有开放层/封闭层声明**：未明确哪些层可扩展、哪些不可变 → low
- **是否有文档生长规则**：无"新增模块 → 确定维度 → 分配编号 → 声明依赖"规则 → low

### 检查项 R4-G：技术选型合理性
- **技术是否匹配场景**：选用的技术与问题场景不匹配（如用关系数据库做全文搜索） → high-⑩
- **是否过度工程**：引入了当前规模不需要的技术复杂度 → medium
- **是否有替代方案评估**：关键选型无替代方案对比 → low

### 检查项 R4-H：数据流清晰性
- **数据流向是否无迂回**：数据从 A→B→C→D→E 而非 A→E → high-⑩
- **是否有数据转换浪费**：多次不必要的数据格式转换 → medium
- **数据所有权是否明确**：每条数据流有明确的产出者和消费者 → medium

## 五、dimension 强制枚举与轮次匹配规则（§5.2 / §5.5）

problem 的 `dimension` 字段**必须**从以下枚举中取值，Gate 校验非法值直接判 problem 无效：

| 枚举值 | 允许的轮次 |
|--------|----------|
| `需求覆盖度` | 仅 R1（结构红队）|
| `边界与协作` | 仅 R2（边界红队）|
| `演化与失败` | 仅 R3（极限红队）|
| `架构质量` | 仅 R4（质量红队）|

**轮次匹配规则**：
- R1 红队产 problem 时，dimension 字段必须填 `需求覆盖度`
- R2 红队产 problem 时，dimension 字段必须填 `边界与协作`
- R3 红队产 problem 时，dimension 字段必须填 `演化与失败`
- R4 红队产 problem 时，dimension 字段必须填 `架构质量`
- Gate 校验：dimension 值不在枚举内 → problem 无效；dimension 值与红队轮次不匹配 → problem 无效。

> 这一约束使 §5.6 轮内升级判据"同一 high problem"的 dimension 比较具备稳定字符串基础。

## 六、红队信息隔离约束（§5.5 / AC-8）

3 轮红队之间**信息隔离**：

- R1 / R2 / R3 红队**不允许查看其他轮的问题清单、响应记录、仲裁决议**（避免互相抄袭、避免锚定偏差）。
- 每轮红队仅可见：L2（唯一对照源）+ 当前轮架构师交付的蓝图版本。
- 工作区运行时保证 3 轮红队的 process 目录互相隔离。

**例外**：终审裁决者可见全部 3 轮对抗记录（其职责即全局一致性校验，§5.5 明文例外）。

**信息隔离的副作用处置**：R2 红队若误填 dimension=`需求覆盖度`（属 R1 范围），Gate 校验直接判 problem 无效。即用 dimension 枚举 + 轮次匹配规则保证维度正交，**而非靠信息传递**。

## 七、红队利益对立原则（§5.1）

- 红队职责是 **"证明架构不行"**，**不是** "帮忙改进"。
- 红队只能产出 **结构化问题清单 + 严重度 + 严重度依据**，**不能改方案**。
- 架构师**不得**要求红队给出方案建议。
- 红队的 problem 描述必须**引用蓝图中的章节/模块标题**，不能空泛。
