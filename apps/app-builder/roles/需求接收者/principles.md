# 需求接收者 原则

## 设计原则
1. 需求文档必须包含：目标、核心能力、角色清单、协作流程、对抗性要求、验收标准、文档层级
2. 角色清单至少 2 个角色，至少 1 个 producer 入口
3. 验收标准必须具体可证伪（可客观判定通过/不通过）
4. 对抗性要求需明确是否引入红队/审计者角色
5. 文档层级需明确目标 APP 需要哪些文档类型及其细化关系

## 文档层级要素（第七要素）
需求文档中必须描述目标 APP 的文档分层结构，包括：
- **文档类型列表**：列出需要的文档类型及其职责
  - 推荐四层：需求文档 → 层分解文档 → 层模块文档 → 实现规格文档
  - 简单项目可裁剪为两层：需求文档 → 实现规格文档
- **文档间依赖关系**：上层文档是下层文档的唯一输入依据
- **文档层与执行层的关系**：执行者以哪份文档为实现依据
- **每层是否需要审阅/对抗角色**

示例：
```
## 文档层级
1. 需求文档（本文档）→ 产出者：需求文档作者
2. 层分解文档 → 产出者：层分解文档作者，输入：需求文档
3. 层模块文档 → 产出者：层模块文档作者，输入：层分解文档
4. 实现规格文档 → 产出者：实现规格文档作者，输入：层模块文档
5. 执行层以实现规格文档为依据，FORK 扇出为前端/后端/测试执行者
6. 需求文档需人工确认，其余文档 auto
```

## 校验清单
- [ ] 角色清单完整（>=2 个角色，含 producer 入口）
- [ ] 协作流程清晰（线性/并行/迭代）
- [ ] 验收标准可证伪
- [ ] 无内部矛盾
- [ ] 文档层级明确（列出文档类型及依赖关系）
- [ ] 文档层与执行层界限分明
