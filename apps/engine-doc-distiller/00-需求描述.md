# 需求文档：引擎文档族提炼 app

## 1. 目标

构建一个多角色 app（名称：`engine-doc-distiller`），用于**系统地从引擎源代码和运行数据中提炼信息，产出结构化的文档族**。

该 app 替代当前手动撰写引擎文档族的方式（已有引擎文档族 v1/2.0/3 三版，均为人工逐篇编写），通过多角色协作 + 对抗性审阅，实现以下**五大核心目标**：

1. **代码→文档的可追溯性**：每篇文档必须明确引用其信息来源（engine/ 目录下的具体文件/函数/类）
2. **文档族结构化**：按引擎三层架构（稳定层/组织层/功能层）分层组织，有全局导航、编号注册表、术语词典
3. **引擎特性清晰陈列**：文档族不只是列举组件，必须提炼和展示引擎的核心设计特性（如指令周期、多角色协作、状态机、对抗闭环、并行调度、回退修复机制等），让读者能从"设计哲学"层面理解引擎
4. **引擎全貌完整描述**：文档族必须完整覆盖引擎的运作全貌——从用户视角（app.yaml 定义）、到编译视角（compiler → ROUTER.json → registry.json）、到运行视角（orchestrator → router → gate → step）、到扩展视角（sdk），读者可仅凭文档族理解引擎的**端到端完整运作方式**
5. **优化点发现与已知局限标注**：文档族必须包含"优化建议"和"已知局限"维度，帮助读者和引擎维护者识别引擎可以改进的点，促进引擎持续演进

> **增量更新能力**：引擎代码变更后，能识别受影响的文档并定向更新（作为设计约束保留）

## 2. 核心能力

| 能力域 | 描述 |
|--------|------|
| **引擎代码分析** | 读取 `engine/sdk/`、`engine/scripts/`、`engine/schemas/` 中的代码与规范文件，提取架构信息（组件职责、接口签名、数据流、状态机、控制流） |
| **信息提炼与结构化** | 将代码级信息提炼为概念级知识（如"指令周期 = dispatch→execute→post_execute→BLOCKING→post_confirm"），去除实现细节噪声 |
| **引擎特性提炼** ⭐ | 从分散的代码组件中提炼出引擎的**核心设计特性**，形成可独立阅读的"特性档案"——包括：指令生命周期、多角色协作模型、状态机与状态迁移、对抗闭环（challenged/loop/major_rework）、并行调度（FORK/JOIN）、回退修复机制（fix.py）、编译管线、SDK 扩展接口。每项特性需有：定义、设计动机、运作机制图示、代码对应关系 |
| **引擎全貌编排** ⭐ | 设计文档族使得读者可以按多条阅读路径理解引擎全貌：(a) 新手入门路径——从"引擎是什么"到"如何跑一个 app"；(b) 架构理解路径——从三层架构到每个组件的职责；(c) 运维路径——如何诊断/修复/扩展引擎。文档族需有一篇"引擎总览"文档，串联所有组件形成端到端运作叙述 |
| **文档族编排** | 设计文档族的分层结构（全局/功能层/组织层/稳定层/应用接口层/管理），分配编号，建立交叉引用导航地图 |
| **文档撰写** | 逐篇撰写 Markdown 文档，每篇包含：概述、核心概念、代码对应关系、Mermaid 图（拓扑/状态图）、与其他文档的交叉引用。文档涉及组件时需标注该组件的**已知局限** |
| **优化点识别** ⭐ | 通过对引擎代码和文档的结构化分析，主动发现引擎可优化的点——包括：架构层面（耦合度、扩展性）、设计层面（状态机复杂度、边界条件）、工程层面（错误处理、命名规范、重复逻辑），产出结构化优化建议清单 |
| **质量审阅** | 检查文档的准确性（与代码一致）、完整性（无遗漏组件）、一致性（术语统一、交叉引用有效）、**特性完整性（核心特性是否全部覆盖）** |
| **对抗性验证** | 红队角色从"新读者"视角挑战文档：是否可仅凭文档理解引擎？是否有断链？是否有歧义？**是否覆盖了引擎的核心特性？是否能让读者建立引擎全貌认知？** |
| **知识沉淀** | 输出一份构建报告（人类可读），记录文档族的完整结构、覆盖率统计、引擎特性清单、已知局限汇总、优化建议优先级排序、已知 Gap |

> ⭐ 标记的能力为本次新增，响应用户的三个关键补充诉求

## 3. 角色清单

| # | 角色名 | type | confirm | 职责 |
|---|--------|------|---------|------|
| 1 | **需求接收者** | producer | manual | 接收用户的文档族范围指令（目标读者、覆盖深度、优先组件），整理为结构化需求 |
| 2 | **引擎分析师** | standard | auto | 读取 `engine/` 全部代码，提炼组件清单、接口签名、数据流、依赖关系；**提炼引擎核心设计特性**（指令周期、多角色协作、状态机、对抗闭环、并行调度、回退修复等），产出结构化分析报告 |
| 3 | **文档架构师** | standard | auto | 基于分析报告设计文档族分层结构、编号体系、导航地图、交叉引用矩阵；**设计"引擎特性展示"专题章节和"引擎全貌"总览文档的编排方案**；规划已知局限标注规范 |
| 4 | **文档撰写者** | standard | auto | 按文档架构逐篇撰写 Markdown 文档，每篇标注代码来源引用；**撰写引擎特性档案、引擎总览文档**；在每篇组件文档中标注已知局限 |
| 5 | **文档审阅者** | standard | auto | 审阅文档的准确性（代码可追溯）、完整性（覆盖率）、一致性（术语/格式/交叉引用）、**特性完整性（核心特性是否全部覆盖且描述准确）** |
| 6 | **文档红队** | standard | auto | 从新读者视角对抗性挑战：可理解性、断链检测、歧义检测、关键信息遗漏检测、**全貌可理解性（读者能否建立引擎完整认知）、特性覆盖完整性** |
| 7 | **优化分析师** ⭐ | standard | auto | **基于引擎分析报告和文档族，系统性地发现引擎可优化的点**：分析架构耦合、扩展性、状态机复杂度、错误处理、重复逻辑等，产出结构化优化建议清单和已知局限汇总，按优先级排序 |
| 8 | **知识管理者** | standard | auto | 汇总审阅结果和优化建议，裁决是否通过/迭代，产出构建报告（含优化建议整合），管理文档族版本 |

**约束说明：**
- producer 入口角色：**需求接收者**（confirm: manual，需用户确认范围后启动）
- 共 **8 个角色**，覆盖提炼→编排→撰写→审阅→对抗→优化→沉淀的完整闭环
- 所有 standard 角色均为 auto confirm，保证流水线自动推进
- ⭐ 优化分析师为本次新增角色，响应用户"发现优化点"诉求

## 4. 协作流程

**流程类型：迭代式（含审阅回路 + 对抗回路 + 并行优化分析）**

```
需求接收者 (producer, manual)
    │ confirmed
    ▼
引擎分析师 ──────────── 读取 engine/ 代码，提炼组件清单 + 引擎核心特性
    │ confirmed
    ▼
文档架构师 ──────────── 设计文档族结构（含特性展示编排 + 全貌总览 + 局限标注规范）
    │ confirmed
    ▼
文档撰写者 ──────────── 逐篇撰写文档（含特性档案、引擎总览、已知局限标注）
    │ confirmed
    ▼
[FORK] 并行扇出 ────────┬──→ 文档审阅者   (维度：准确性/完整性/一致性/特性完整性)
                        ├──→ 文档红队     (维度：可理解性/断链/歧义/遗漏/全貌认知/特性覆盖)
                        └──→ 优化分析师 ⭐ (维度：优化点识别/已知局限/改进建议/优先级排序)
                              │              │              │
                              ▼              ▼              ▼
                         [JOIN] 汇入 → 知识管理者
    │
    ├─ confirmed  → 整合优化建议 → 产出构建报告 → 完成
    ├─ loop       → 回退文档撰写者（修复质量问题）  max_executions: 5
    └─ major_rework → 回退文档架构师（结构需重设计） max_executions: 3
```

**回路设计：**
- **质量回路**：知识管理者判定 loop → 文档撰写者修复，上限 5 次
- **结构回路**：知识管理者判定 major_rework → 文档架构师重设计，上限 3 次
- **对抗回路**：文档红队 challenged → 知识管理者复核，决定是否触发 loop
- **优化回路**：优化分析师的优化建议不触发回退，而是整合进最终构建报告；若发现严重设计缺陷（如核心组件缺失错误处理），标记为"高优先级"纳入构建报告的"已知局限"章节

**并行设计：**
- 文档审阅者、文档红队、优化分析师三者**并行执行**（FORK + JOIN），提升效率
- 优化分析师基于引擎分析报告 + 已完成的文档族进行分析，与审阅/对抗同时进行，不阻塞主线

## 5. 对抗性要求

**明确引入对抗角色：文档红队（第 6 号角色）**

文档红队的对抗维度：

| 维度 | 挑战内容 | 判定标准 |
|------|---------|---------|
| **可理解性** | 一个不了解引擎的读者，能否仅凭文档理解该组件的设计？ | 关键概念有定义、有示例、有图示 |
| **全貌认知** ⭐ | 读者阅读完文档族后，能否建立引擎的**完整运作认知**（从 app.yaml 到运行产出）？ | 存在"引擎总览"文档，端到端串联所有核心组件，有完整的运作流程叙述 |
| **特性覆盖** ⭐ | 文档族是否覆盖了引擎的**全部核心设计特性**？是否每项特性都有独立的"特性档案"？ | 引擎核心特性清单 ≥ 6 项（指令周期/多角色协作/状态机/对抗闭环/并行调度/回退修复），每项有独立文档或章节 |
| **断链检测** | 文档间交叉引用是否有死链/悬空引用？ | 所有 `→` 引用的目标文档实际存在 |
| **歧义检测** | 同一术语在不同文档中是否有不同定义？ | 术语词典覆盖所有专有名词，定义唯一 |
| **遗漏检测** | 是否有引擎核心组件没有对应文档？ | 覆盖率 = 已文档化组件 / 引擎全部组件 ≥ 95% |

**对抗路径：**
- 文档红队 verdict = `confirmed`（通过）→ 汇入知识管理者
- 文档红队 verdict = `challenged`（挑战成立）→ 知识管理者复核，复核 upheld 则触发 loop 回退文档撰写者

## 7. 文档层级（第七要素）

本节描述目标 app（engine-doc-distiller）产出的**文档族分层结构**。需明确文档类型、依赖关系、文档层与执行层的关系，以及每层的审阅/对抗安排。

> 概念区分：**文档层** = 目标 app 运行后产出的 Markdown 文档族（给引擎读者看的成果物）；**执行层** = app 自身的角色流水线（需求→分析→架构→撰写→审阅→对抗→优化→沉淀，给 app 自身用的协作骨架）。本节目的就是划清这两层并描述它们的映射关系。

### 7.1 文档类型列表

目标 app 产出的文档族按职责分为 **5 大文档类型**：

| 类型代号 | 文档类型 | 典型文档 | 说明 |
|---------|---------|---------|------|
| **L0 导航类** | 全局导航 / 索引 | README.md、95-导航地图.md、编号注册表、99-术语词典.md | 文档族的"入口"，读者经此进入；注册全部文档编号与术语 |
| **L1 总览类** | 引擎全貌 | 引擎总览.md（端到端串联） | 一篇串联所有核心组件的端到端运作叙述，覆盖用户/编译/运行/扩展/修复视角 |
| **L2 特性档案类** | 引擎核心设计特性 | 指令生命周期、多角色协作模型、状态机、对抗闭环、并行调度、回退修复（≥6 篇） | 每项特性独立成篇/成章，含定义/动机/机制图示/代码对应 |
| **L3 组件文档类** | 分层组件说明 | 稳定层 / 组织层 / 功能层 / 应用接口层 各组件文档 | 按引擎三层（稳定/组织/功能）+ 应用接口层组织，每篇含代码可追溯引用与已知局限 |
| **L4 管理类** | 报告与建议 | 构建报告.md、优化建议清单、已知局限汇总 | 人类可读总览：覆盖率统计、特性清单、优化建议（按优先级）、已知 Gap |

### 7.2 文档间依赖关系

文档类型之间存在**有向依赖**（上游被下游引用，下游依赖上游产出）。依赖关系呈**有向无环图（DAG）**，不存在循环引用：

```
L0 导航类 (导航地图/编号注册表/术语词典)
    ▲ 依赖（索引全部文档，最晚定稿）
    │
    ├── L1 总览类 (引擎总览) ────── 依赖 L2/L3（串联引用所有特性档案与组件文档）
    │         │
    │         │
    ├── L2 特性档案类 ────────────── 依赖 L3 对应组件文档（引用代码细节）
    │         │                        │
    │         ▼                        ▼
    └── L3 组件文档类 ────────────── 依赖 SDK_SPEC.md / 术语词典（L0）
              │
              ▼
         L4 管理类 (构建报告/优化建议) ── 依赖 L0~L3 全部 + 审阅/对抗/优化报告
```

**依赖规则（供文档架构师落地为交叉引用矩阵）：**
- **导航地图（L0）**必须索引全部 L1~L4 文档，最后定稿
- **引擎总览（L1）**必须交叉引用全部特性档案（L2）和组件文档（L3）
- **特性档案（L2）**必须引用其涉及的组件文档（L3）作为代码佐证
- **组件文档（L3）**必须引用术语词典（L0）定义的术语，并标注已知局限
- **构建报告（L4）**汇总覆盖率统计、特性清单、优化建议，依赖 L0~L3 + 执行层报告
- **禁止循环依赖**：若 A 引用 B，则 B 不得再引用 A（避免读者陷入死循环）

### 7.3 文档层与执行层的关系

文档层（产物）与执行层（角色流水线）之间存在明确的**产出映射**：

| 执行层角色 | 产出物（执行层） | 喂养/对应的文档层内容 |
|-----------|----------------|---------------------|
| 引擎分析师 | 引擎分析报告.json | → L3 组件文档的素材 + L2 特性档案的素材 |
| 文档架构师 | 文档族架构.json | → 定义 L0~L4 的分层结构、编号、导航地图、交叉引用矩阵 |
| 文档撰写者 | *.md 文档族正文 | → 直接产出 L0 导航类、L1 总览类、L2 特性档案类、L3 组件文档类文档 |
| 文档审阅者 | 审阅报告.json | → 校验 L0~L3 文档的准确性/完整性/一致性（不产出文档，但决定文档是否通过） |
| 文档红队 | 对抗分析报告.json | → 对抗性验证 L1 全貌认知、L2 特性覆盖、L3 可理解性（不产出文档，但触发文档修复） |
| 优化分析师 | 优化建议清单.json | → 产出 L4 管理类的优化建议与已知局限汇总 |
| 知识管理者 | 构建报告.md | → 整合产出 L4 管理类（构建报告），管理文档族版本 |

**界限原则（文档层与执行层界限分明）：**
1. **执行层角色产出物 ≠ 文档层文档**：分析报告/架构设计/审阅报告等是"过程产物"，存在 `outputs/` 目录；文档层 Markdown 存放在 workspace 根目录的分层目录中，是"最终成果"
2. **文档层由撰写者角色统一产出**：所有给读者看的 Markdown（L0~L3）均由文档撰写者产出，审阅者/红队/优化分析师只产出执行层报告（不直接改写文档层），保证文档层风格一致
3. **执行层报告驱动文档层迭代**：审阅者/红队的 findings 通过知识管理者裁决后，回退到撰写者修改文档层，而非由审阅者直接改

### 7.4 每层审阅 / 对抗角色安排

不同文档类型的质量风险不同，对应不同的审阅/对抗重点：

| 文档层 | 主要风险 | 审阅角色 | 对抗角色 | 验证重点 |
|--------|---------|---------|---------|---------|
| **L0 导航类** | 编号冲突、断链、术语不一致 | 文档审阅者 | 文档红队（断链检测） | 编号唯一、引用有效、术语覆盖 |
| **L1 总览类** | 全貌叙述断裂、视角缺失 | 文档审阅者 | **文档红队（全貌认知，重点对抗）** | 端到端串联完整、四视角无缺 |
| **L2 特性档案类** | 特性遗漏、机制描述错误 | 文档审阅者（特性完整性） | **文档红队（特性覆盖，重点对抗）** | ≥6 项特性、每项四要素齐全 |
| **L3 组件文档类** | 代码不可追溯、已知局限缺失 | 文档审阅者（准确性） | 文档红队（可理解性） | 代码引用有效、每篇含已知局限 |
| **L4 管理类** | 建议无优先级、覆盖率造假 | 知识管理者（自检） | —（由知识管理者整合三方报告） | 建议按优先级排序、覆盖率可核验 |

**对抗强度说明：**
- L1 总览类与 L2 特性档案类是**对抗重点**（红队的全貌认知/特性覆盖维度专门针对这两层）
- L3 组件文档类对抗强度中等（红队从可理解性/断链角度挑战）
- L0 导航类与 L4 管理类以**审阅者校验**为主（结构化校验，对抗强度低）

## 8. 验收标准

以下标准均为**可证伪的二元判定**（通过/不通过）：

### 8.1 结构完整性

- [ ] **AC-1**：文档族包含 ≥ 5 个分层目录（如 00-全局、10-功能层、20-组织层、30-稳定层、40-应用接口层）
  - **证伪方式**：计算目录层级数 < 5 → 不通过
- [ ] **AC-2**：存在一份导航地图文档（README.md 或 95-导航地图.md），列出全部文档及其阅读路径
  - **证伪方式**：导航地图不存在，或未列出全部文档 → 不通过
- [ ] **AC-3**：存在一份编号注册表，每篇文档有唯一编号且无冲突
  - **证伪方式**：编号重复或缺失 → 不通过

### 8.2 代码可追溯性

- [ ] **AC-4**：每篇文档的"代码对应关系"章节明确引用 `engine/` 下的具体文件路径
  - **证伪方式**：任一篇文档无代码引用，或引用路径不存在 → 不通过
- [ ] **AC-5**：引擎分析师的分析报告中，列出 engine/scripts/ 全部 .py 文件，无遗漏
  - **证伪方式**：diff(engine/scripts/*.py 列表, 分析报告中列出的文件) ≠ ∅ → 不通过

### 8.3 覆盖率

- [ ] **AC-6**：文档覆盖率 ≥ 95%（已文档化组件数 / 引擎核心组件总数）
  - **核心组件清单**：orchestrator, router, gate, step, compiler, fix, set_state, init, switch, mode, sdk, schemas（共 12 个组件域）
  - **证伪方式**：覆盖率 < 95% → 不通过

### 8.4 引擎特性陈列 ⭐

- [ ] **AC-7**：文档族包含 ≥ 6 项引擎核心特性的"特性档案"（独立文档或独立章节）
  - **必含特性清单**：①指令生命周期 ②多角色协作模型 ③状态机与状态迁移 ④对抗闭环（challenged/loop/major_rework） ⑤并行调度（FORK/JOIN） ⑥回退修复机制（fix.py）
  - **每项特性档案需包含**：定义、设计动机、运作机制（含 Mermaid 图示）、代码对应关系
  - **证伪方式**：特性档案数量 < 6，或任一必含特性缺失，或任一档案缺少上述四要素 → 不通过

### 8.5 引擎全貌完整性 ⭐

- [ ] **AC-8**：存在一篇"引擎总览"文档，端到端串联引擎核心组件形成完整运作叙述
  - **总览需覆盖**：用户视角（app.yaml → compiler → ROUTER.json）、运行视角（orchestrator → router → gate → step → switch）、扩展视角（sdk → custom roles）、修复视角（fix.py → gate）
  - **证伪方式**：总览文档不存在，或缺少上述任一视角的叙述 → 不通过

### 8.6 优化点发现 ⭐

- [ ] **AC-9**：优化分析师产出的优化建议清单包含 ≥ 5 条结构化优化建议
  - **每条建议需包含**：优化维度（架构/设计/工程）、问题描述、影响评估、建议方案、优先级（高/中/低）
  - **证伪方式**：建议数量 < 5，或任一建议缺少上述五要素 → 不通过
- [ ] **AC-10**：文档族中每篇组件文档包含"已知局限"章节（可标注"暂无已知局限"）
  - **证伪方式**：任一组件文档无"已知局限"章节 → 不通过
- [ ] **AC-11**：构建报告中包含"优化建议汇总"章节，按优先级排序，且整合了优化分析师的建议
  - **证伪方式**：构建报告无优化建议章节，或未按优先级排序 → 不通过

### 8.7 交叉引用完整性

- [ ] **AC-12**：所有文档内的交叉引用（如 `→ 20-组织层/10-指令周期设计`）指向的目标文档实际存在
  - **证伪方式**：任一交叉引用的目标文件不存在 → 不通过

### 8.8 文档层级完整性 ⭐（第七要素新增）

- [ ] **AC-16**：文档族架构设计中明确列出 5 大文档类型（L0 导航类 / L1 总览类 / L2 特性档案类 / L3 组件文档类 / L4 管理类），且实际产出的文档均可归入这 5 类
  - **证伪方式**：架构设计缺少任一类型定义，或存在无法归类的"孤儿文档" → 不通过
- [ ] **AC-17**：文档间依赖关系为有向无环图（DAG），无循环依赖；导航地图（L0）可索引全部文档（L0~L4）
  - **证伪方式**：存在循环引用（A→B→A），或导航地图遗漏任一文档 → 不通过

### 8.9 DAG 与运行正确性

- [ ] **AC-13**：app.yaml 编译后 DAG 可达（所有角色从 producer 入口可达，所有角色可达"完成"终态）
  - **证伪方式**：存在孤立角色或不可达终态 → 不通过
- [ ] **AC-14**：所有角色的 verdict 在 edges 的 `when:` 中有对应出边（无悬空 verdict）
  - **证伪方式**：角色输出的 verdict 无对应边 → 不通过

### 8.10 对抗性

- [ ] **AC-15**：文档红队角色存在，且其 verdict 空间包含 `confirmed`/`challenged` 两条出边
  - **证伪方式**：无红队角色，或缺少任一出边 → 不通过

## 10. 设计约束

1. **app_name**：`engine-doc-distiller`
2. **producer 入口**：需求接收者，confirm: manual（用户确认文档族范围后启动流水线）
3. **文档撰写为迭代过程**：撰写者可能被审阅者/红队触发多轮修复
4. **三方并行审阅**：文档审阅者、文档红队、优化分析师并行执行（FORK + JOIN），提升效率
5. **角色数 = 8**，覆盖完整知识闭环（含优化分析）
6. **对抗性角色数 = 1**（文档红队），优化分析师虽非对抗角色但提供独立的改进视角
7. **引擎特性档案为必含产出**：≥ 6 项核心特性，每项有独立文档或章节
8. **优化建议为必含产出**：优化建议清单 ≥ 5 条，构建报告必须整合优化建议汇总
9. **已知局限标注为强制规范**：每篇组件文档必须包含"已知局限"章节
10. **文档层级为强制要素**（第七要素）：文档族架构必须定义 L0~L4 五大文档类型、有向无环依赖关系，并明确文档层（成果物）与执行层（角色流水线）的界限——所有给读者的 Markdown 由文档撰写者统一产出，审阅/红队/优化角色只产出执行层报告
