# v9.2 引擎机制速查

本文档是 engine-test app 的知识注入文件，供关键角色理解所在步骤验证的引擎机制。

## 1. Gate 两阶段校验

- **Layer 0（信封校验）**：校验 role-executor 返回的 JSON 格式契约（step/verdict/status/outputs）。失败 → ENVELOPE_FAIL → BLOCKING（协议违规，不走 fail 边）。
- **Layer 1（文件校验）**：校验磁盘产出物的存在性 + 非空 + 二进制短路 + 可选 contract。失败 → FAIL → 走 fail 边（质量问题，可重试）。

## 2. verdict 权威源

- verdict 合法性权威源是 `ROUTER.json` 的 `transitions keys`。
- 角色输出的 verdict 必须在该 step 的 transitions 中，否则 Gate Layer 0 拒绝。
- `confirmed` 是无显式条件边的默认 verdict（编译器自动生成）。

## 3. 条件路由与 restrict_verdict

- **条件路由**：step 根据 verdict 值选择不同下游（如 pass→正路径，revise→回退路径）。
- **restrict_verdict**：边级声明，限制下游角色可输出的 verdict 范围。例如路由者→正路径 的 restrict_verdict=[approved, rejected] 表示正路径只能输出 approved 或 rejected。

## 4. carries 物料传递

- **carries**：边级显式声明物料传递。例如 物料生产者→物料消费者 carries=[outputs/测试报告.json]。
- carries 文件在 dispatch 时自动注入到下游 role 的 inputs 列表。

## 5. max_executions 循环上限

- **max_executions**：边级声明，控制循环次数上限。例如 循环者→循环者 max_executions=2 表示最多自循环 2 次。
- 超过上限后边被"掐断"，不再调度（edge_counts 计数器机制）。

## 6. FORK-JOIN 拓扑

- **FORK**：一个 step confirmed 后扇出到多个目标（并行执行）。
- **JOIN**：多个来源 step 的 confirmed 边指向同一目标，目标等待所有来源完成后才执行（input_groups AND 条件）。

## 7. manual 阻塞确认

- **confirm: manual** 的角色在产出后会进入 awaiting_confirmation 状态。
- 需要用户回复 confirmed（通过）或 fail（拒绝）才能继续。
- fail 会触发 backward 边回退重试，feedback 注入到重试 dispatch 的 inputs 中。

## 8. terminal_state 判定

- 完成判定的权威源是 router.py 的"可达集闭合检查"。
- 从 entry 出发 BFS 沿已完成 step 的 verdict 边遍历，可达集中无未完成 step → 判定完成。
- 条件路由中未选择的分支路径不影响完成判定。
