> **改造轮次**：第二轮增量改造（在第一轮 17 角色架构基础上）
> **改造时间**：2026-07-13 ~ 2026-07-14
> **最终结论**：✅ 审计通过（upheld），改造完成

## 一、改造概要

| 项目 | 说明 |
|------|------|
| **目标 APP** | `lxp-eng-planning`（精力管理系统·前后端分离架构） |
| **改造类型** | 新增角色 + 调整编排 + 修复缺陷 |
| **改造前角色数** | 17 角色（需求确认 → 需求分流 → 前后端文档 → 交叉审核 → 设计组 → 前后端实现 → 合并校验 → 部署 → 终审） |
| **改造后角色数** | 19 角色（新增「预览部署者」+「前端预览确认」） |
| **改造前人工确认节点** | 2 个（需求接收者、综合裁决者） |
| **改造后人工确认节点** | 3 个（新增「前端预览确认」） |
| **最终裁决** | 综合裁决 `confirmed` → 审计 `upheld` |

### 改造核心目标

让用户在**前端代码写完后**，能够通过浏览器查看**实际渲染的网页效果**，满意了再继续后续流程，不满意可以打回前端设计组修改。这填补了原来「代码写完直接合并校验」时用户无法提前看到前端成果的体验缺口。

改造后的前端链路：

```
前端设计组(3角色) → 前端实现者 → 前端实现者(校验) → 预览部署者 → 前端预览确认(人工)
   → 满意(confirmed) → 合并校验者
   → 不满意(needs_revision) → [交互逻辑设计师, 美学布局设计师] 修改 → 重新预览（最多3轮）
```

## 二、新增角色说明

### 2.1 预览部署者（standard / auto）

| 属性 | 值 |
|------|-----|
| **类型** | standard（标准角色，自动确认） |
| **核心职责** | 将前端代码部署到本地预览环境（localhost），启动预览服务，输出可访问的预览地址 |
| **输入** | 前端代码（由前端实现者产出） |
| **输出** | 预览部署报告（含预览 URL、部署状态、技术栈信息） |
| **在流程中的位置** | 前端实现者（校验）通过 → 预览部署者 → 前端预览确认 |

**执行步骤要点**：识别技术栈（React/Next.js/Vite 等）→ 安装依赖 → 启动预览服务 → 验证服务可用（HTTP 请求检查）→ 生成预览部署报告（含预览地址如 `http://localhost:3000`）。

### 2.2 前端预览确认（standard / manual）

| 属性 | 值 |
|------|-----|
| **类型** | standard（标准角色，**人工确认** — 全局第 3 个 manual 节点） |
| **核心职责** | 用户通过浏览器打开预览地址，查看实际渲染效果，对美学设计、交互体验、页面布局进行人工判断 |
| **输入** | 预览部署报告 |
| **输出** | 前端预览确认报告（含 verdict + 具体反馈意见/修改建议） |
| **在流程中的位置** | 预览部署者 → **前端预览确认（用户决策）** → 合并校验者 / 回退设计组 |

**为什么需要人工确认**：美学与交互是高度主观的领域，仅靠自动校验无法替代人类审美判断。用户需要实际看到渲染后的网页——配色是否协调、拖拽是否流畅、布局是否舒适，这些只有打开浏览器看了才知道。

**用户评估维度**：
- 🎨 **美学设计**：配色方案、字体层次、视觉风格
- 🖱️ **交互体验**：拖拽编排流畅度、点击响应、操作反馈
- 📐 **页面布局**：组件排列、间距、响应式适配
- ⚡ **功能呈现**：树状视图、拖拽编排、每日清单、周视图的实际渲染效果

## 三、反馈循环机制

这是本次改造最核心的新增能力——**预览 → 反馈 → 修改 → 再预览** 的闭环。

### 3.1 循环路径

```
前端预览确认(verdict = needs_revision)
   ↓ 携带「前端预览确认报告（含用户具体反馈意见）」
   → [交互逻辑设计师, 美学布局设计师] 重新设计
   → 前端实现者 (JOIN 累计计数·接口翻译师无需重做)
   → 前端实现者(校验)
   → 预览部署者 重新部署
   → 前端预览确认 (用户再次查看)
```

### 3.2 关键设计决策：回退目标为什么只有 2 个角色？

| 角色 | 是否回退 | 原因 |
|------|---------|------|
| 交互逻辑设计师 | ✅ 是 | 负责交互逻辑设计，美学/交互修改的核心 |
| 美学布局设计师 | ✅ 是 | 负责美学布局设计，视觉调整的核心 |
| 接口翻译师 | ❌ 否 | 前端美学/交互的修改通常不需要重新翻译接口逻辑 |

> **为什么不回退接口翻译师？** 这是第一轮改造发现的 **MAJOR1 问题**（后端级联风险）。如果接口翻译师被重新触发，其 `confirmed` 出边会级联触发 API 契约设计师 → 后端设计组 JOIN → 后端实现者 → 后端实现者校验——**整个后端链路都会无谓重跑**。通过将接口翻译师排除在回退目标之外，并利用 JOIN 的累计计数语义（接口翻译师上一轮已完成贡献持续满足），级联链在数据结构层面被物理斩断。

### 3.3 循环上限

- **max_executions: 3** — 最多 3 轮「预览 → 反馈 → 修改 → 再预览」循环
- 超过 3 次后该回退边被路由器掐断，用户可选择 `confirmed`（接受当前效果）继续后续流程

## 四、人工确认节点说明

改造后系统共有 **3 个人工确认节点**（confirm: manual），这是你（用户）在流程中的三次关键决策点：

| 序号 | 节点 | 何时出现 | 你需要做什么 | 你的选择 |
|------|------|---------|-------------|---------|
| ① | **需求接收者** | 流程最起点 | 确认需求描述是否准确、完整 | confirmed（需求清晰） / loop（补充修改） |
| ② | **前端预览确认** ⭐新增 | 前端代码写完并部署后 | 打开浏览器查看实际网页效果 | confirmed（满意，继续） / needs_revision（不满意，打回修改） |
| ③ | **综合裁决者** | 全流程末端 | 对最终交付物做全局终审 | confirmed（通过交付） / challenged（打回重审） / deploy_doc_defect（部署文档返工） |

> ⭐ **前端预览确认**是本次改造新增的第 3 个人工确认节点，它让你在代码合入主流程之前就能看到真实的前端效果，避免"等代码都写完了才发现前端不符合预期"的尴尬。

## 六、迭代历程

本次改造经历了 **2 轮审阅迭代**，第 1 轮发现问题后在第 2 轮全部修复确认。

### 6.1 第 1 轮：发现问题

第 1 轮三路审阅（改造红队 / 结构审阅者 / 合规审阅者）共发现 **5 项 findings**：

| 级别 | ID | 问题描述 | 严重性说明 |
|------|----|---------|-----------|
| 🔴 CRITICAL | blocking_mode 注释污染 | app.yaml 中 `confirm: manual` 字段行内注释被编译器嵌入 registry.json 的 blocking_mode 值，导致引擎精确匹配失败，**人工确认闸门静默失效** | 用户的前端预览确认节点可能被自动放行，失去人工把关能力 |
| 🟠 MAJOR1 | 后端级联风险 | needs_revision 回退目标含接口翻译师，接口翻译师 confirmed 出边会级联触发 API 契约设计师 → 后端设计组 → 后端实现者，**整个后端链路无谓重跑** | 用户对前端美学不满意会意外触发后端全部重做，严重浪费资源 |
| 🟠 MAJOR2 | 可选输入遗漏 | 交互逻辑设计师、美学布局设计师的 app.yaml inputs 遗漏「前端预览确认报告[可选输入]」声明 | 回退时角色无法接收用户反馈意见，修改缺乏依据 |
| 🟡 MINOR1 | 硬编码路径 | 预览部署者 skill.md 中硬编码了 `outputs/src/frontend/` 路径 | 违反路径规范，应使用 dispatch 注入语义 |
| 🟡 MINOR2 | fail carries 缺失 | 因 CRITICAL 导致编译器未识别 manual 模式，fail 回退路径缺少 feedback.json | 用户反馈无法携带回退（CRITICAL 修复后自动连带解决） |

### 6.2 修复执行

针对上述 5 项 findings 逐一修复：

| Finding | 修复操作 | 验证结果 |
|---------|---------|---------|
| **CRITICAL** | confirm 字段注释移到独立行，值只写纯 `manual` | ✅ registry.json blocking_mode = `"manual"` 纯值（3 处 manual 节点全部纯值，0 处注释污染） |
| **MAJOR1** | needs_revision 回退目标从 3 角色缩减为 2 角色（移除接口翻译师） | ✅ 三件套（app.yaml / ROUTER.json / registry.json）回退目标一致，级联链物理斩断 |
| **MAJOR2** | 交互逻辑设计师 / 美学布局设计师 inputs 新增前端预览确认报告[可选输入] | ✅ 输入声明完整，skill.md 引用与 app.yaml 契约一致 |
| **MINOR1** | 预览部署者 skill.md 删除硬编码路径 | ✅ 现为「读取 dispatch 注入的输入文件（前端代码目录）」纯语义描述 |
| **MINOR2** | CRITICAL 修复后编译器自动补充 fail carries feedback.json | ✅ ROUTER.json fail carries 包含 feedback.json（连带修复） |

### 6.3 第 2 轮：全部修复确认

第 2 轮三路审阅均返回 `confirmed`：

- **改造红队（对抗维度）**：4 项关键攻击全部失败（0/4 击穿），6 项附加探测均通过或为 LOW 级观察

**第 2 轮合并结果**：0 critical / 0 major / 3 low（均为非阻塞性文档/清洁度建议）
