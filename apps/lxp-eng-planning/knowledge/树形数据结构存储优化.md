# 树形数据结构存储优化

## 目的

本文档为 lxp-eng-planning 应用的三层任务树（长期目标→短期任务→可执行任务）提供 SQLite + Prisma 场景下的存储方案与查询优化指南。解决的核心问题是：如何高效存储、查询、级联删除三层树形结构。

## 适用角色

- 技术文档管理者（设计数据模型时参考）
- 代码生成者（实现数据访问层时参考）

## 1. 树形存储方案对比

| 方案 | 查询子树 | 插入 | 移动子树 | 删除子树 | 适用场景 |
|------|---------|------|---------|---------|---------|
| **邻接表（parent_id）** | 需递归（CTE 或 N+1） | O(1) | O(1) | 需递归 | 层级固定且浅（≤3层），写入频繁 |
| **物化路径（path）** | `LIKE '1.2.%'` 前缀匹配 | O(1) | 需更新所有子孙 path | O(1) | 层级深、需快速查子树 |
| **嵌套集（lft/rgt）** | `lft BETWEEN ?` 范围查询 | O(n)（需重排） | O(n) | O(n) | 读多写少、子树查询极频繁 |

### 选型判据

lxp-eng-planning 的任务树具有以下特征：
- **层级固定 3 层**（长期目标 level=1 / 短期任务 level=2 / 可执行任务 level=3）
- **增删改查频繁**（用户持续编排任务）
- **无需深递归查询**（最多 3 层，可用固定层数 JOIN 替代递归）

**推荐方案**：**邻接表（parent_id 自引用）**——层级浅（≤3 层）时递归代价可控，写入/移动成本最低。

## 2. Prisma Schema 中 parent_id 自引用模型定义

```prisma
model Task {
  id            String   @id @default(cuid())
  title         String
  description   String?
  
  // 层级控制
  level         Int      // 1=长期目标, 2=短期任务, 3=可执行任务
  parent_id     String?  // 自引用外键，level=1 时为 null
  
  // 精力管理
  energy_level  EnergyLevel  // HIGH / MEDIUM / LOW 枚举
  energy_points Float        // 派生值：HIGH=4.0, MEDIUM=2.0, LOW=1.0
  
  // 计划编排
  scheduled_date    DateTime?   // 排程日期（仅 level=3 可排程）
  scheduled_period  String?     // 时段：morning / afternoon / evening
  
  // 状态
  is_archived   Boolean  @default(false)
  is_completed  Boolean  @default(false)
  completed_at  DateTime?
  incomplete_reason String?  // 精力不足 / 时间不够 / 被打断
  
  // 时间戳
  created_at    DateTime @default(now())
  updated_at    DateTime @updatedAt
  
  // 自引用关系
  parent   Task?   @relation("TaskTree", fields: [parent_id], references: [id], onDelete: Cascade)
  children Task[]  @relation("TaskTree")
  
  @@index([parent_id])
  @@index([scheduled_date])
  @@index([level])
}

enum EnergyLevel {
  HIGH
  MEDIUM
  LOW
}
```

### 关系映射说明

- `parent`：指向父节点（多对一），`onDelete: Cascade` 表示删除父节点时自动删除所有子节点
- `children`：反向关系（一对多），通过 `@relation("TaskTree")` 关联
- `parent_id` 为 null 时表示根节点（level=1 的长期目标）

## 3. 层级深度限制策略

### 3.1 数据库层校验

SQLite 本身不支持 CHECK 约束限定枚举值（需要应用层校验），但可以添加约束：

```sql
-- 确保 level 在 1-3 范围内
ALTER TABLE Task ADD CONSTRAINT chk_level CHECK (level IN (1, 2, 3));
```

### 3.2 应用层双重校验

```typescript
// Prisma 中间件或服务层校验
async function createTask(data: CreateTaskInput) {
  if (data.parent_id) {
    const parent = await prisma.task.findUnique({ where: { id: data.parent_id } });
    if (!parent) throw new Error('父任务不存在');
    
    const newLevel = parent.level + 1;
    if (newLevel > 3) {
      throw new Error('任务层级超过3层限制：长期目标(1)→短期任务(2)→可执行任务(3)');
    }
    
    // level=3 的任务才可排程
    if (newLevel < 3 && data.scheduled_date) {
      throw new Error('只有可执行任务(level=3)可以排程到日历');
    }
  }
  
  return prisma.task.create({ data: { ...data, level: data.parent_id ? newLevel : 1 } });
}
```

## 4. 级联删除与归档策略

### 4.1 Prisma onDelete: Cascade

```prisma
parent Task? @relation("TaskTree", fields: [parent_id], references: [id], onDelete: Cascade)
```

删除一个长期目标（level=1）时，Prisma + SQLite 自动级联删除其下所有短期任务和可执行任务。

### 4.2 软删除（归档）方案

用户需求明确：删除需确认，支持归档。因此实际使用软删除而非物理删除：

```typescript
async function archiveTask(taskId: string) {
  // 递归归档该任务及其所有子孙
  await prisma.task.updateMany({
    where: {
      OR: [
        { id: taskId },
        // 通过 path 或递归查询找到所有子孙
      ]
    },
    data: { is_archived: true }
  });
}
```

**归档 vs 删除决策**：
- 用户选择"归档"：设置 `is_archived = true`，任务从树中隐藏但数据保留
- 用户选择"删除"：级联删除（`onDelete: Cascade`），数据永久移除
- 删除操作需二次确认（前端弹出确认对话框）

## 5. 树形结构递归查询优化

### 5.1 SQLite CTE 递归查询

```sql
-- 查询某节点的所有子孙（递归 CTE）
WITH RECURSIVE TaskTree AS (
  -- 基础查询：起始节点
  SELECT id, title, level, parent_id FROM Task WHERE id = ?
  UNION ALL
  -- 递归查询：所有子节点
  SELECT t.id, t.title, t.level, t.parent_id
  FROM Task t
  JOIN TaskTree tt ON t.parent_id = tt.id
)
SELECT * FROM TaskTree ORDER BY level, id;
```

### 5.2 固定层数 JOIN（3 层树优化）

由于层级固定为 3 层，可以用固定层数 JOIN 替代递归，查询性能更优：

```typescript
// 一次性查询整棵树（3 层 JOIN）
async function getFullTree() {
  const roots = await prisma.task.findMany({
    where: { level: 1, is_archived: false },
    include: {
      children: {
        where: { is_archived: false },
        include: {
          children: {
            where: { is_archived: false }
          }
        }
      }
    },
    orderBy: { created_at: 'asc' }
  });
  return roots;
}
```

### 5.3 消除 N+1 查询

**❌ 错误（N+1 查询）**：
```typescript
const roots = await prisma.task.findMany({ where: { level: 1 } });
for (const root of roots) {
  root.children = await prisma.task.findMany({ where: { parent_id: root.id } });
  // 每个子任务又查询一次 → N+1 问题
}
```

**✅ 正确（一次查询 + include）**：
```typescript
const roots = await prisma.task.findMany({
  where: { level: 1 },
  include: { children: { include: { children: true } } }
});
// 总共 1 + 1 + 1 = 3 次查询（每层一次），与树大小无关
```

## 6. energy_points 派生字段计算

### 6.1 energy_level 到 energy_points 的映射

| energy_level | energy_points | 耗时范围 |
|-------------|--------------|---------|
| HIGH | 4.0 | ≥ 4 小时 |
| MEDIUM | 2.0 | 2-4 小时（含2，不含4） |
| LOW | 1.0 | < 2 小时 |

**边界归属**：耗时恰好 2 小时 → MEDIUM；恰好 4 小时 → HIGH。

### 6.2 存储策略

**方案 A（推荐）**：创建时计算并存储 `energy_points` 字段

```typescript
const ENERGY_MAP = { HIGH: 4.0, MEDIUM: 2.0, LOW: 1.0 };

async function createTask(data: CreateTaskInput) {
  return prisma.task.create({
    data: {
      ...data,
      energy_points: ENERGY_MAP[data.energy_level]
    }
  });
}
```

**方案 B**：使用 Prisma 计算字段（Prisma 不支持计算列，需在查询后映射）

### 6.3 更新联动

当用户修改任务的 `energy_level` 时，同步更新 `energy_points`：

```typescript
async function updateEnergyLevel(taskId: string, level: EnergyLevel) {
  return prisma.task.update({
    where: { id: taskId },
    data: { energy_level: level, energy_points: ENERGY_MAP[level] }
  });
}
```

## 7. 执行记录与精力日志表设计

### 7.1 执行记录表（ExecutionLog）

```prisma
model ExecutionLog {
  id          String   @id @default(cuid())
  task_id     String   // 关联可执行任务
  task        Task     @relation(fields: [task_id], references: [id])
  
  completed   Boolean  @default(false)
  completed_at DateTime?
  incomplete_reason String?  // 精力不足 / 时间不够 / 被打断
  
  scheduled_date DateTime   // 计划日期
  scheduled_period String   // morning / afternoon / evening
  
  created_at  DateTime @default(now())
  
  @@index([task_id])
  @@index([scheduled_date])
}
```

### 7.2 精力日志表（EnergyLog）

```prisma
model EnergyLog {
  id          String   @id @default(cuid())
  log_date    DateTime  // 日志日期
  log_type    String    // daily / weekly
  
  total_points Float    // 当日/当周总精力消耗
  budget       Float    // 当日/当周预算
  is_overload  Boolean  // 是否超载
  
  breakdown   String    // JSON: { HIGH: count, MEDIUM: count, LOW: count }
  completion_rate Float // 完成率
  
  created_at  DateTime @default(now())
  
  @@index([log_date])
  @@index([log_type])
}
```

### 7.3 三表关联设计

```
Task (任务树)
  │
  ├── 1:N → ExecutionLog (执行记录)
  │           每次任务完成/标记未完成生成一条记录
  │
  └── 间接 → EnergyLog (精力日志)
              由每日/每周统计汇总生成，记录精力消耗与完成率
```
