# 全栈启动验证最佳实践

## 目的

本文档为后端启动验证者、全栈联调验证者、合并校验者、后端实现者和前端实现者提供运行时验证方法论指导，确保产出的代码真正可运行而非仅"看起来正确"。

## 适用角色

- 后端启动验证者：后端服务启动 + API 端点测试
- 全栈联调验证者：前后端联调 + 端到端用户流程测试
- 合并校验者：审查联调验证报告，确认运行时集成验证维度
- 后端实现者：产出可运行的后端代码（含启动脚本、健康检查端点、CORS 配置）
- 前端实现者：产出可运行的前端代码（含环境变量配置、proxy 转发、Error Boundary）

## 一、Prisma 数据库初始化标准流程

### 1.1 三条命令的区别

| 命令 | 用途 | 适用场景 |
|---|---|---|
| `npx prisma generate` | 生成 Prisma Client（TypeScript 类型） | 每次修改 schema.prisma 后必须执行 |
| `npx prisma db push` | 将 schema 直接推送到数据库（创建/修改表结构） | 开发环境快速同步，不生成迁移历史 |
| `npx prisma migrate dev` | 生成迁移文件并应用到数据库 | 生产环境需迁移历史时使用 |

### 1.2 验证流程中的推荐用法

后端启动验证者使用 `db push`（开发环境快速初始化），不使用 `migrate dev`（避免迁移文件管理复杂性）。

### 1.3 常见错误

- **Schema 语法错误**：`npx prisma validate` 可提前校验
- **数据库文件锁定**：确保没有其他进程占用 SQLite 文件
- **Prisma Client 未生成**：`generate` 必须在 `db push` 之前执行

## 二、Node.js 后端服务启动验证方法

### 2.1 健康检查轮询模式

后端服务启动是异步过程，不能假设 `npm run dev` 后立即可用。采用轮询模式：

```
1. 启动后端服务（后台进程）
2. 每 2 秒请求一次 GET /api/health
3. 收到 200 响应 → 服务就绪
4. 超过 30 秒未就绪 → 判定启动失败
```

### 2.2 健康检查端点要求

后端实现者必须实现 `GET /api/health` 端点：
- 返回 200 状态码
- 响应体包含 `{ "status": "ok" }` 或类似标识
- 不依赖数据库连接（纯存活检查）

### 2.3 后台进程管理

- 启动时记录进程 PID
- 测试完成后通过 PID 或端口杀掉进程
- 确保端口释放，避免下次启动冲突

## 三、API 端点自动化测试方法论

### 3.1 CRUD 完整测试

对每个资源端点，按 CRUD 顺序测试：

| 操作 | 方法 | 验证要点 |
|---|---|---|
| Create | POST | 201 状态码 + 响应体含新资源 ID + 数据库中可查询到新记录 |
| Read | GET | 200 状态码 + 响应体数据结构正确 |
| Update | PUT/PATCH | 200 状态码 + 数据库中记录已更新 |
| Delete | DELETE | 200/204 状态码 + 数据库中记录已删除 |

### 3.2 持久化验证

不能仅依赖 HTTP 响应状态码判断操作成功。每个写入操作（POST/PUT/PATCH/DELETE）后，必须：
1. 通过数据库查询（如 `npx prisma studio` 或直接 SQL 查询）确认数据确实变更
2. 或通过 GET 请求查询该资源，确认数据一致

### 3.3 树形结构特殊验证

lxp-eng-planning 使用邻接表方案（parentId 自引用）存储任务树：
- 创建子任务时验证 parentId 正确关联
- 查询任务树时验证层级关系正确
- 删除父任务时验证子任务的处理策略（级联删除或拒绝删除）

## 四、前后端联调测试模式

### 4.1 连通性检查清单

| 检查项 | 方法 | 预期结果 |
|---|---|---|
| API base URL 配置 | 检查前端环境变量（VITE_API_BASE_URL） | 指向后端地址或通过 proxy 转发 |
| CORS 配置 | 检查后端 CORS 中间件配置 | 允许前端域（localhost:5173 等） |
| Vite proxy 配置 | 检查 vite.config.ts 的 server.proxy | /api 请求转发到后端端口 |
| 实际请求测试 | 从前端 dev server 发起 API 请求 | 成功到达后端并返回数据 |

### 4.2 端到端用户流程测试

按 F1-F4 功能逐个测试完整用户流程：

1. **F1 树状视图**：创建任务 → 创建子任务 → 查询任务树 → 验证树形结构
2. **F2 拖拽编排**：拖拽任务更新排程 → 验证数据库排程已更新 → 重新加载确认持久化
3. **F3 每日清单**：选择日期 → 查询当日任务 → 验证精力标签显示
4. **F4 周视图**：查询周视图 → 验证预算聚合计算 → 验证精力可视化

### 4.3 错误处理测试

| 场景 | 请求 | 预期响应 | 前端预期表现 |
|---|---|---|---|
| 无效数据 | POST 提交空标题/非法日期 | 4xx + 错误信息 | 显示错误提示，不崩溃 |
| 不存在的资源 | GET /api/tasks/99999 | 404 | 显示"未找到"提示 |
| 后端服务异常 | 模拟后端错误 | 5xx | Error Boundary 生效，不白屏 |

## 五、常见启动失败排查清单

### 5.1 端口占用

- **现象**：`EADDRINUSE` 错误
- **排查**：`lsof -i :端口号` 查看占用进程
- **解决**：杀掉占用进程或更换端口

### 5.2 依赖缺失

- **现象**：`Cannot find module 'xxx'` 错误
- **排查**：检查 package.json dependencies 是否完整
- **解决**：重新执行 `npm install`

### 5.3 Prisma Schema 错误

- **现象**：`npx prisma generate` 或 `db push` 失败
- **排查**：执行 `npx prisma validate` 检查语法
- **解决**：修正 schema.prisma 语法错误

### 5.4 CORS 拒绝

- **现象**：浏览器控制台报 `Access-Control-Allow-Origin` 错误
- **排查**：检查后端 CORS 中间件配置
- **解决**：在 CORS 配置中添加前端域（如 `http://localhost:5173`）

### 5.5 数据库连接失败

- **现象**：`Can't reach database server` 错误
- **排查**：检查 DATABASE_URL 配置 + 数据库文件路径
- **解决**：修正连接字符串或确保数据库文件存在

### 5.6 环境变量缺失

- **现象**：前端 API 请求发到错误地址或后端读取不到配置
- **排查**：检查 .env 文件 + process.env 读取
- **解决**：确保 .env 文件存在且变量名正确
