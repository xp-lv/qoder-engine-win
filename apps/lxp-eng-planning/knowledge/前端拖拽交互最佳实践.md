# 前端拖拽交互最佳实践

## 目的

本文档为 lxp-eng-planning 应用的拖拽看板（周计划/日计划视图）提供前端交互设计方案与性能优化实践。解决的核心问题是：如何在 200+ 任务量级下实现流畅的拖拽体验（FPS ≥ 30，延迟 < 200ms）。

## 适用角色

- 技术文档管理者（设计技术方案时参考）
- 代码生成者（实现拖拽功能时参考）

## 1. 拖拽库选型对比

| 库 | 生态 | 性能 | 灵活性 | 推荐场景 |
|----|------|------|--------|---------|
| **React DnD** | React 生态最成熟 | 中等（基于 HTML5 DnD API） | 高（可自定义 drag source/drop target） | 需要复杂拖拽逻辑（如跨容器排序） |
| **@dnd-kit** | 新兴，React | 高（基于 Pointer Events，支持传感器抽象） | 高（Sortable/Preset 模块化） | 推荐首选——性能好、API 现代、内置无障碍支持 |
| **Vue Draggable (vuedraggable)** | Vue 生态 | 中等（基于 SortableJS） | 中等 | Vue 项目快速集成 |

**选型建议**：React 项目首选 `@dnd-kit/core` + `@dnd-kit/sortable`，其传感器抽象（Mouse/Touch/Keyboard）和动画系统在 200+ 元素场景下性能表现最优。

## 2. 拖拽事件生命周期与状态管理

### 2.1 事件流

```
dragstart → dragover（持续触发） → drop → dragend
```

### 2.2 状态管理方案

拖拽过程中需维护以下状态：

```typescript
interface DragState {
  activeTask: Task | null;        // 当前拖拽的任务
  activeContainer: string | null; // 当前悬停的容器（列/时段）
  overContainer: string | null;   // 即将放置的容器
  dragOffset: { x: number; y: number }; // 拖拽偏移量
}
```

**状态更新策略**：
- `dragstart`：设置 `activeTask`，冻结拖拽元素的原始位置
- `dragover`：高频更新 `overContainer`，**使用 requestAnimationFrame 节流**避免每帧重渲染
- `drop`：执行任务的位置变更（更新 `scheduled_date` / `scheduled_period` 字段），持久化到后端
- `dragend`：清理所有拖拽状态

### 2.3 React 状态更新优化

```typescript
// ❌ 错误：直接修改数组引用
tasks[index].scheduled_date = newDate;
setTasks([...tasks]);

// ✅ 正确：使用不可变更新
setTasks(prev => prev.map(t =>
  t.id === activeId ? { ...t, scheduled_date: newDate } : t
));
```

## 3. 200+ 任务场景下的性能优化

### 3.1 虚拟列表渲染

当任务总数超过 100 时，应使用虚拟列表只渲染可视区域内的任务卡片：

```typescript
import { useVirtualizer } from '@tanstack/react-virtual';

// 只渲染视口内的任务卡片，DOM 节点数恒定（约 10-15 个）
const rowVirtualizer = useVirtualizer({
  count: tasks.length,
  getScrollElement: () => scrollRef.current,
  estimateSize: () => 60, // 每张任务卡片高度
});
```

**注意事项**：
- 拖拽中的元素需要从虚拟列表中"提取"到独立层渲染，避免虚拟列表滚动导致拖拽元素位置跳跃
- 使用 `position: fixed` + 坐标映射让拖拽元素脱离滚动容器

### 3.2 requestAnimationFrame 节流

```typescript
const rafRef = useRef<number>();

const handleDragOver = useCallback((event: DragOverEvent) => {
  if (rafRef.current) return; // 已有待执行的 rAF，跳过
  rafRef.current = requestAnimationFrame(() => {
    const overId = event.over?.id;
    setActiveContainer(getContainerId(overId));
    rafRef.current = undefined;
  });
}, []);
```

**效果**：将 dragover 事件的高频回调限制在 60fps（约 16ms 一次），避免同步布局抖动。

### 3.3 DOM diff 最小化

- 为每个任务卡片设置稳定的 `key={task.id}`（不要用数组索引）
- 拖拽过程中只更新受影响的容器内容，不重新渲染整个看板
- 使用 React.memo 包裹 TaskCard 组件，只在 `task` 属性变化时重渲染

```typescript
const TaskCard = React.memo(({ task }: { task: Task }) => {
  return <div>{task.title} - {task.energy_points}点</div>;
}, (prev, next) => prev.task.id === next.task.id
  && prev.task.scheduled_date === next.task.scheduled_date);
```

### 3.4 性能基准

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 拖拽 FPS | ≥ 30 | Chrome DevTools Performance 面板 |
| 拖拽放置响应延迟 | < 200ms | 从 drop 事件到 UI 更新完成的时间 |
| 首次渲染时间 | < 1s | 200 个任务的初始渲染 |
| 重渲染范围 | ≤ 2 个容器 | 拖拽从 A 列拖到 B 列，只重渲染 A 和 B |

```
│ 8.0/8.0 │12.0/8.0 │...  │...  │...  │...  │ ...  │ ← 每列负荷
├─────────┼─────────┼─────┼─────┼─────┼─────┼──────┤
│ [任务A] │ [任务C] │     │     │     │     │      │
│ [任务B] │         │     │     │     │     │      │
└─────────┴─────────┴─────┴─────┴─────┴─────┴──────┘
```

### 4.2 列容器（Droppable）

每个日期列是一个 droppable 容器：

```typescript
function DayColumn({ date, tasks }: { date: string; tasks: Task[] }) {
  const { setNodeRef } = useDroppable({ id: `day-${date}` });
  const dayLoad = tasks.reduce((sum, t) => sum + t.energy_points, 0);
  const isOverloaded = dayLoad > dailyBudget;

  return (
    <div ref={setNodeRef} className={isOverloaded ? 'column-overload' : ''}>
      <span>{dayLoad} / {dailyBudget}</span>
      {tasks.map(t => <TaskCard key={t.id} task={t} />)}
    </div>
  );
}
```

### 4.3 任务卡片（Draggable）

```typescript
function TaskCard({ task }: { task: Task }) {
  const { attributes, listeners, setNodeRef, transform } = useDraggable({
    id: task.id,
  });
  // transform 控制拖拽时的视觉偏移
  return (
    <div ref={setNodeRef} {...listeners} {...attributes}>
      {task.title} [{task.energy_level}]
    </div>
  );
}
```

### 5.2 时段分类标签

时段（上午/下午/晚上）仅作为**分类标签**，不影响精力点计算口径。`daily_load` = 当日所有时段任务的 `energy_points` 之和。

每个时段是一个独立的 droppable 容器，ID 格式为 `period-{date}-{morning|afternoon|evening}`。

### 6.2 拖拽时实时更新

拖拽过程中（dragover），预览放置后的负荷变化：

```typescript
function calculatePreviewLoad(tasks: Task[], activeTask: Task, targetDate: string) {
  const targetDayTasks = tasks.filter(t => t.scheduled_date === targetDate);
  return targetDayTasks.reduce((sum, t) => sum + t.energy_points, 0) + activeTask.energy_points;
}
```

### 6.3 超载预警 UI

- **日负荷超载**：`daily_load > daily_budget` → 列高亮橙色
- **周负荷超载**：`weekly_load >= overload_threshold (56.0)` → 顶部横幅红色 + 文案「⚠ 本周精力负荷 58.0 / 56.0，已超载，建议调整」

## 7. 拖拽放置响应延迟 < 200ms 保证策略

1. **乐观更新**：drop 时立即更新前端状态，异步持久化到后端
2. **批量更新**：连续拖拽操作合并为一次 API 请求
3. **后端响应优化**：CRUD API 响应时间 < 100ms（SQLite 本地查询）
4. **避免布局抖动**：拖拽元素使用 `transform` 而非 `top/left`（不触发 reflow）
5. **动画 GPU 加速**：`will-change: transform` 让浏览器将元素提升到合成层
