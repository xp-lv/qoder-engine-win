# 校验标准手册

## 目的

本文档为 lxp-eng-planning 应用的校验者与综合裁决者提供统一审查标准、检查清单、严重级别定义与 verdict 路由规则。解决的核心问题是：如何系统地、可证伪地审查代码交付质量，并给出明确的 verdict 裁决，且精准定位缺陷层级以触发正确的回退路由。

## 适用角色

- 校验者（质量层校验时参考）
- 综合裁决者（最终裁决时参考）

---

## 1. 文档一致性检查方法

### 1.1 核心原则

代码实现与技术文档（API 签名/数据模型字段/业务逻辑）必须**逐项比对**，不一致率必须为 **0%**。

### 1.2 检查维度

| 维度 | 检查内容 | 比对来源 |
|------|---------|---------|
| API 接口签名 | 路径、HTTP 方法、请求体字段、响应体字段、状态码 | API 文档 vs 实际路由定义 |
| 数据模型字段 | 字段名、类型、约束、关系 | 数据库设计文档 vs Prisma schema |
| 业务逻辑 | 精力点计算、层级约束、预算聚合规则 | 技术设计文档 vs 实际业务代码 |
| 能量值映射 | HIGH=4.0, MEDIUM=2.0, LOW=1.0 | 精力管理领域知识 vs 实际映射代码 |
| 需求覆盖 | F1-F4 功能点是否全部实现 | 需求规格文档 vs 代码功能模块 |
| 验收标准对照 | CODE-X / DOC-X / DEPLOY-X 是否满足 | 验收标准清单 vs 实际交付物 |

### 1.3 检查方法

```
1. 列出 API 文档中所有接口定义
2. 逐一与代码中的路由定义比对
3. 记录不一致项（字段缺失/类型不匹配/路径不一致）
4. 统计不一致率 = 不一致项数 / 总检查项数
5. 不一致率必须为 0%，否则根据缺陷层级选择 verdict（见 §6.2 和 §9）
```

## 6. 严重级别定义与 verdict 路由

### 6.1 严重级别

| 级别 | 标签 | 判定标准 |
|------|------|---------|
| **critical** | 阻断核心功能 | 核心功能无法运行（如 CRUD 失败、数据库初始化失败、拖拽不可用） |
| **major** | 功能不完整 | 某功能缺失或行为不正确但不阻断其他功能（如预警不显示、报告不弹出） |
| **minor** | 体验缺陷 | 功能可用但体验不佳（如 UI 对齐、颜色不一致、文案错误） |
| **info** | 优化建议 | 改进建议（如性能优化、代码重构） |

### 6.2 校验者 verdict 四维路由判定规则

校验者有四个 verdict，实现需求§4.2 第7条回退规则 2+3 的精准路由：

```
判定流程：
1. 若所有维度通过，无 critical/major 问题：
   → verdict = confirmed（前进至部署文档管理者）

2. 若存在代码层缺陷（代码不符合技术文档规格）：
   → verdict = BLOCKING（回退至代码生成者 R7，max:3）
   触发条件示例：API 路由缺失/CRUD 返回错误/DB 初始化失败/拖拽不可用/FPS<30

3. 若代码严格按文档实现但技术文档本身有误：
   → verdict = DOC_DEFECT（回退至技术文档管理者 R5，max:3）
   触发条件示例：API 文档签名笔误/数据库设计文档字段遗漏/技术设计文档逻辑矛盾

4. 若需求规格/验收标准/用户故事有误：
   → verdict = REQ_DOC_DEFECT（回退至需求文档管理者 R4，max:3）
   触发条件示例：验收标准遗漏关键功能/用户故事描述矛盾/需求规格不完整

5. 多种缺陷并存时：选择最严重的缺陷层级作为 verdict
```

**禁止模糊结论**：不允许出现"基本通过""大体可以"等模糊表述，必须明确 verdict。

### 6.3 综合裁决者 verdict 四维路由判定规则

综合裁决者有四个 verdict，实现需求§3「做最终 verdict，处理升级与回退决策」：

```
判定流程：
1. 若全部交付物合格，校验者 confirmed，无遗留 critical：
   → verdict = confirmed（→ 完成）

2. 若发现架构层根本缺陷：
   → verdict = challenged（→ 架构师 R3，max:2）
   触发条件：DAG 拓扑无法支撑需求/角色定义缺失关键能力/架构方案根本性矛盾

3. 若发现需求层根本缺陷：
   → verdict = requirement_defect（→ 需求文档管理者 R4，max:1）
   触发条件：F1-F4 功能域定义缺失/验收标准无法证伪/用户故事与需求规格矛盾

4. 若发现部署文档缺陷：
   → verdict = deploy_doc_defect（→ 部署文档管理者 R6，max:2）
   触发条件：启动命令无法执行/配置项缺失/CHANGELOG 未记录关键变更

5. 多种缺陷并存时：按影响面排序（架构>需求>部署），选择最大影响面的 verdict
```

## 7. 文档质量检查维度

### 7.1 可追溯性

每个功能点（F1-F4）在以下文档中均可追溯：
- 需求规格文档：有对应的功能描述
- 技术设计文档：有对应的实现方案
- 代码实现：有对应的代码模块
- 验收标准清单：有对应的验收条目

### 7.2 完整性

| 文档 | 完整性要求 |
|------|-----------|
| 需求规格文档 | 覆盖 F1-F4 全部功能域 |
| API 文档 | 接口数 ≥ 6（CRUD 5 + 统计 1+） |
| 数据库设计 | 三张表均定义完整 |
| 部署指南 | 含完整启动命令 |
| 运维手册 | 含配置项清单 + 故障排查 |

### 7.3 时效性

- 文档内容与最新代码实现一致
- 无过时的接口定义或字段描述
- CHANGELOG 记录了所有变更

### 7.4 可读性

- 文档结构清晰（有目录、有标题层级）
- 术语统一（如 energy_points / energy_level 全文一致）
- 示例充分（API 有请求/响应示例）

## 8. 校验报告格式要求

校验报告（JSON）必须包含以下结构：

```json
{
  "result": {
    "verdict": "confirmed | BLOCKING | DOC_DEFECT | REQ_DOC_DEFECT",
    "summary": "校验概述",
    "dimensions": {
      "前端交互": {
        "status": "pass | fail",
        "findings": [
          {
            "severity": "critical | major | minor | info",
            "item": "CODE-1",
            "description": "具体问题描述",
            "evidence": "佐证信息",
            "defect_layer": "code | doc_tech | doc_req"
          }
        ]
      },
      "后端API": { ... },
      "数据模型": { ... },
      "文档一致性": {
        "status": "pass | fail",
        "consistency_rate": "100%",
        "inconsistencies": []
      },
      "性能": { ... }
    }
  }
}
```

### 8.1 具名引用要求

校验报告中每个 finding 必须包含**至少 3 个具名引用**：
- 引用功能编号（如 F1, F2）
- 引用验收标准编号（如 CODE-1, CODE-6）
- 引用具体代码文件或文档章节

### 8.2 缺陷层级标注要求

每个 finding 必须标注 `defect_layer`，用于辅助 verdict 路由判定：
- `code`：缺陷根因在代码实现（→ BLOCKING）
- `doc_tech`：缺陷根因在技术文档（→ DOC_DEFECT）
- `doc_req`：缺陷根因在需求文档（→ REQ_DOC_DEFECT）

## 9. 缺陷层级定位方法

### 9.1 定位流程

校验者和综合裁决者在发现缺陷后，需准确定位缺陷根因层级，以触发正确的回退路由：

```
缺陷发现
  │
  ├─ 代码不符合技术文档规格？
  │    ├─ 是 → defect_layer = code（代码实现缺陷）
  │    │        校验者 verdict = BLOCKING → 回退至代码生成者 R7
  │    │
  │    └─ 否 → 继续判断文档问题
  │              │
  │              ├─ 技术文档（API/数据库设计/技术设计）本身有误？
  │              │    ├─ 是 → defect_layer = doc_tech（技术文档缺陷）
  │              │    │        校验者 verdict = DOC_DEFECT → 回退至技术文档管理者 R5
  │              │    │
  │              │    └─ 否 → 继续判断需求文档问题
  │              │              │
  │              │              ├─ 需求规格/验收标准/用户故事有误？
  │              │              │    ├─ 是 → defect_layer = doc_req（需求文档缺陷）
  │              │              │    │        校验者 verdict = REQ_DOC_DEFECT → 回退至需求文档管理者 R4
  │              │              │    │
  │              │              │    └─ 否 → defect_layer = code（默认归为代码缺陷）
  │              │              │
  │              │              └─ 综合裁决者层级判定：
  │              │                   架构层根本缺陷 → challenged → 架构师 R3
  │              │                   需求层根本缺陷 → requirement_defect → 需求文档管理者 R4
  │              │                   部署文档缺陷 → deploy_doc_defect → 部署文档管理者 R6
```

### 9.2 层级判定示例

**示例 1**：API 文档定义 `POST /api/tasks` 请求体含 `energy_level` 字段，但代码中路由未校验该字段。
- 代码不符合文档 → `defect_layer = code` → 校验者 verdict = **BLOCKING**

**示例 2**：API 文档定义 `POST /api/tasks` 响应体含 `energy_points` 字段，代码也返回了该字段，但 API 文档中 `energy_points` 的类型写成了 `int` 而非 `float`。
- 代码按文档实现，文档本身有误 → `defect_layer = doc_tech` → 校验者 verdict = **DOC_DEFECT**

**示例 3**：验收标准清单中遗漏了 F2 拖拽性能要求（FPS ≥ 30），导致代码生成者未实现性能优化。
- 根因在需求文档 → `defect_layer = doc_req` → 校验者 verdict = **REQ_DOC_DEFECT**

**示例 4**：综合裁决者发现 app.yaml 的 DAG 结构中缺少文档管理者的回退路径。
- 架构层根本缺陷 → 综合裁决者 verdict = **challenged** → 回退至架构师

### 9.3 综合裁决者与校验者的职责区分

| 角色 | 审查范围 | 回退触发条件 | 回退目标 |
|------|---------|-------------|---------|
| 校验者 | 代码 + 技术文档 + 需求文档一致性 | 代码缺陷 / 技术文档缺陷 / 需求文档缺陷 | R7 代码生成者 / R5 技术文档管理者 / R4 需求文档管理者 |
| 综合裁决者 | 全部交付物 + 架构层 + 部署层 | 架构根本缺陷 / 需求层根本缺陷 / 部署文档缺陷 | R3 架构师 / R4 需求文档管理者 / R6 部署文档管理者 |

## 检查清单

校验者执行校验后应自查：

- [ ] 文档一致性检查完成，不一致率为 0%
- [ ] 前端交互（F1/F2/F3）逐项检查
- [ ] 后端 API CRUD + 统计接口全部验证
- [ ] 数据模型三表结构 + Prisma migration 验证
- [ ] 性能指标（FPS ≥ 30, 延迟 < 200ms）验证
- [ ] 每个发现（finding）有明确严重级别
- [ ] 每个发现（finding）有缺陷层级标注（code/doc_tech/doc_req）
- [ ] verdict 明确为 confirmed / BLOCKING / DOC_DEFECT / REQ_DOC_DEFECT 之一，无模糊结论
- [ ] 校验报告中包含至少 3 个具名引用（功能编号/验收标准/代码文件）
- [ ] 报告格式符合 JSON 结构要求

综合裁决者执行最终审查后应自查：

- [ ] 已审查全部 12 份上游产出物
- [ ] §8 验收标准清单逐项核对完成
- [ ] 四维度（可追溯性/完整性/时效性/可读性）检查完成
- [ ] verdict 明确为 confirmed / challenged / requirement_defect / deploy_doc_defect 之一
- [ ] 若非 confirmed，已注明回退目标角色和回退理由
