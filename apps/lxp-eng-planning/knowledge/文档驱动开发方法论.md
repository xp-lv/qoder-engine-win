# 文档驱动开发方法论

## 目的

本文档为 lxp-eng-planning 应用的文档驱动开发（Documentation-Driven Development, DDD）提供方法论指导与各文档管理者的工作规范。解决的核心问题是：如何通过"文档先行、文档即契约、文档持续维护"三大原则，确保需求→架构→编码→部署全生命周期的文档一致性。

## 适用角色

- 需求文档管理者（维护需求层文档）
- 技术文档管理者（维护技术层文档）
- 部署文档管理者（维护部署层文档）

---

## 1. 文档先行原则

### 1.1 核心定义

每个阶段的代码/方案生成前，对应文档必须先由文档管理者产出。文档是下游工作的**唯一输入依据**，不可跳层或凭空创造。

### 1.2 文档层级递进

```
L0 需求文档（需求接收者产出）
  ↓ 需求审计通过后
L1 需求规格文档（需求文档管理者产出）
  ↓
L2 架构设计文档（架构师产出）
  ↓ 架构审计通过后
L3 技术设计文档（技术文档管理者产出）
  ↓ 文档即契约
L4 代码实现（代码生成者产出）
  ↓ 质量校验通过后
L5 部署文档（部署文档管理者产出）
```

### 1.3 阻塞模式

| 层级转换 | 阻塞条件 | 阻塞含义 |
|---------|---------|---------|
| L0 → L1 | 需求审计通过 | R4 需求文档管理者必须等待需求审计通过后才启动 |
| L2 → L3 | 架构审计通过 | R5 技术文档管理者必须等待架构审计通过后才启动 |
| L4 → L5 | 质量校验通过 | R6 部署文档管理者必须等待质量校验通过后才启动 |

---

## 2. 文档即契约原则

### 2.1 核心定义

L3 技术设计文档是代码实现的**唯一契约**。代码生成者必须严格遵循文档定义：
- API 接口签名（路径、方法、请求体、响应体、状态码）必须与 API 文档一致
- 数据模型字段（类型、约束、关系）必须与数据库设计文档一致
- 业务逻辑必须与技术设计文档描述一致

### 2.2 契约偏差处理

如果代码生成者发现技术文档存在问题：
1. **不可自行修改代码偏离文档**——必须标记为"文档偏差"
2. 在校验报告中记录偏差详情
3. 由校验者判定是否 BLOCKING
4. 若需修改文档，回退至技术文档管理者更新

---

## 3. 文档持续维护策略

### 3.1 核心定义

迭代中任何方案/代码变更，对应文档管理者必须**同步更新**相关文档，保持文档与实际一致。

### 3.2 变更触发规则

| 变更类型 | 触发的文档更新 | 执行角色 |
|---------|-------------|---------|
| 需求变更 | L1 需求规格文档 + 用户故事 + 验收标准清单 | 需求文档管理者 |
| 架构调整 | L2 架构说明 + L3 技术设计文档 | 技术文档管理者 |
| 代码修复（不改接口） | 无需更新文档 | — |
| 代码修复（改接口） | L3 API 文档 + L5 变更日志 | 技术文档管理者 + 部署文档管理者 |
| 部署配置变更 | L5 部署指南 + 运维手册 | 部署文档管理者 |

## 4. 需求规格文档标准模板

### 4.1 文档结构

```markdown
# 需求规格文档

## 1. 概述
- 1.1 项目目标
- 1.2 核心价值
- 1.3 部署形态

## 2. 功能规格
### F1 目标分解（三层拆解）
- 功能编号：F1
- 优先级：P0
- 功能描述
- 输入/输出
- 约束条件
- 验收标准（可证伪）

### F2 计划编排（拖拽看板）
（同上格式）

### F3 每日执行（清单 + 完成）
（同上格式）

### F4 精力管理（追踪 + 可视化 + 预警）
（同上格式）

## 3. 非功能需求
- 3.1 性能要求
- 3.2 安全要求
- 3.3 可用性要求

## 4. 追溯链
| 需求编号 | 上游需求来源 | 下游技术文档 |
|---------|------------|------------|
| F1 | 00-需求描述.md §2.1 | 技术设计文档 §F1 |
```

### 4.2 功能编号规范

- 功能域编号：F1, F2, F3, F4（对应目标分解/计划编排/每日执行/精力管理）
- 子功能编号：F1.1, F1.2（如目标分解下的增删改查）
- 验收标准编号：AC-F1-1, AC-F1-2

### 4.3 验收标准可证伪写法

每条验收标准必须满足：
- **具体**：有明确的操作步骤
- **可证伪**：有明确的通过/不通过判定条件
- **可追溯**：关联到具体功能编号

**示例**：
```
AC-F2-1：前端拖拽编排可运行
  验证方式：鼠标拖拽任务到目标列后位置实际变更
  判定标准：拖拽放置后任务 scheduled_date 字段更新，UI 刷新
  优先级：P0
```

## 5. 技术设计文档标准模板

### 5.1 架构概述

```markdown
# 技术设计文档

## 1. 系统架构
- 1.1 技术栈（前端 React + 后端 Node.js + 数据库 SQLite/Prisma）
- 1.2 部署架构（单服务器 + URL token 鉴权）
- 1.3 模块划分

## 2. 数据模型设计
### 2.1 ER 图
### 2.2 表结构定义（Prisma schema）
### 2.3 索引设计
```

### 5.2 API 接口契约模板

每个 API 接口必须包含以下字段：

```markdown
### POST /api/tasks
**功能**：创建任务
**请求体**：
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 任务标题 |
| level | int | 是 | 层级 1-3 |
| parent_id | string | 否 | 父任务 ID |
| energy_level | enum | 是 | HIGH/MEDIUM/LOW |

**响应体**（200）：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 任务 ID |
| title | string | 任务标题 |
| energy_points | float | 精力点（4.0/2.0/1.0） |

**错误响应**：
| 状态码 | 场景 |
|-------|------|
| 400 | 参数校验失败 |
| 404 | parent_id 不存在 |

**追溯**：关联 F1（目标分解 CRUD）
```

## 6. 部署文档标准模板

### 6.1 环境要求矩阵

| 项目 | 要求 | 说明 |
|------|------|------|
| Node.js | ≥ 18.0 | 后端运行时 |
| SQLite | 内置 | 无需单独安装 |
| 内存 | ≥ 512MB | 单用户场景 |

### 6.2 启动命令序列

```bash
# 1. 安装依赖
npm install

# 2. 初始化数据库
npx prisma migrate deploy

# 3. 生成 Prisma Client
npx prisma generate

# 4. 启动服务
npm run start

# 5. 验证访问
curl http://localhost:3000/health
```

### 6.3 故障排查决策树

```
服务无法启动？
  ├─ 端口被占用？ → 修改 .env 中 PORT 配置
  ├─ 数据库初始化失败？ → 删除 prisma/dev.db 后重新 migrate
  └─ 依赖安装失败？ → 删除 node_modules 后重新 npm install
```

## 7. 文档间追溯链编写方法

### 7.1 追溯链格式

每个规格条目必须关联上层需求编号，形成完整追溯链：

```
F1（目标分解）
  → AC-F1-1（验收标准：树状 CRUD）
    → SPEC-API-1（技术规格：POST /api/tasks）
      → CODE: src/backend/taskService.ts（代码实现）
        → TEST: taskService.test.ts（测试用例）
```

### 7.2 追溯矩阵

| 功能编号 | 需求规格 | API 接口 | 代码实现 | 验收标准 |
|---------|---------|---------|---------|---------|
| F1 | 需求规格文档 §F1 | API文档 POST/GET/PUT/DELETE /api/tasks | src/backend/ | AC-F1-1~4 |
| F2 | 需求规格文档 §F2 | API文档 PUT /api/tasks/schedule | src/frontend/ | AC-F2-1~2 |
| F3 | 需求规格文档 §F3 | API文档 POST /api/tasks/complete | src/backend/ | AC-F3-1 |
| F4 | 需求规格文档 §F4 | API文档 GET /api/stats/energy | src/backend/ | AC-F4-1 |

## 检查清单

各文档管理者产出文档后应自查：

### 需求文档管理者
- [ ] 需求规格文档覆盖 F1-F4 全部功能域
- [ ] 每个功能有优先级标注（P0/P1/P2）
- [ ] 每条验收标准可证伪（有具体验证方式和判定标准）
- [ ] 用户故事使用 As a / I want / So that 格式
- [ ] 追溯链中每条需求关联到上游需求来源

### 技术文档管理者
- [ ] 技术设计文档覆盖架构概述 + 模块划分
- [ ] API 文档覆盖全部 CRUD + 统计接口（≥ 6 个接口）
- [ ] 每个 API 包含路径/方法/请求体/响应体/状态码
- [ ] 数据库设计包含 ER 图 + Prisma schema + 三张表
- [ ] 追溯链中每条 API 关联到功能编号 F1-F4

### 部署文档管理者
- [ ] 部署指南含完整启动命令（可复制执行）
- [ ] 运维手册含配置项清单 + 故障排查决策树
- [ ] CHANGELOG 记录所有版本变更
- [ ] 环境要求矩阵完整
- [ ] 追溯链中部署文档关联到实际代码实现
