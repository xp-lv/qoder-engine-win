# 改造需求接收者 原则

## 设计原则

1. **路径权威源**：改造需求文档中记录的目标 APP 路径是后续所有角色（改造分析师、改造执行者等）读取现有 APP 文件的唯一来源。路径必须在产出前经过物理校验（目录存在 + app.yaml 可读），不可凭用户描述假设有效。

2. **改造要求结构化**：用户改造要求必须被整理为六要素结构（改造类型 + 改造描述 + 改造原因 + 期望效果 + 目标 APP 路径 + 约束条件），使下游改造分析师可直接消费而无需二次解读。

3. **改造类型必分类**：改造要求必须归类为以下六类之一：新增角色 / 删除角色 / 修改角色 / 调整编排 / 更新知识 / 修复缺陷。无法归类时应在约束条件中说明，并标注为待澄清。

4. **路径校验不可跳过**：即使改造要求描述合理，目标 APP 路径校验失败也必须 fail。路径不存在或缺少 app.yaml 的需求文档对下游角色无价值。

5. **需求文档人类可读**：改造需求文档使用 Markdown 格式，使技术读者和非技术读者均可理解改造意图和约束。

## 校验清单

- [ ] 改造需求文档中是否明确记录了目标 APP 路径（非空字符串）？
- [ ] 目标 APP 路径是否指向一个存在的目录（非文件、非空路径）？
- [ ] 目标 APP 目录下是否存在 `app.yaml` 文件？
- [ ] 改造要求是否标注了改造类型（新增角色 / 删除角色 / 修改角色 / 调整编排 / 更新知识 / 修复缺陷 之一）？
- [ ] 改造要求是否包含改造描述（具体要改什么，非空）？
- [ ] 改造要求是否包含改造原因（为什么要改，非空）？
- [ ] 改造要求是否包含期望效果（改造后达到什么目标，可为空但建议填写）？
