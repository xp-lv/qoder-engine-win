# 改造执行规范

## 目的

本文档定义改造执行者的增量修改原则与操作规范，确保只修改方案涉及的文件，改动可追溯，编译可验证。

## 适用角色

- 改造执行者：执行改造方案时参照本规范操作

## 核心内容

### 一、增量修改核心原则

- **只修改改造方案中涉及的文件**，不重写未触及的文件
- **每处改动必须在改造方案中有对应条目**（改动原因 + 影响范围）
- **方案即契约**——只改方案涉及的文件，不自行扩大范围

### 二、app.yaml 编辑操作规范

#### YAML 格式保持
- 缩进风格保持一致（与现有文件对齐）
- 引号使用保持一致
- 注释风格保持一致
- 设计约定注释块（架构级注释）需同步更新

#### roles 段增删改
- 角色定义字段完整性：type / confirm / inputs / outputs
- 新增角色必须声明 outputs（必填字段）
- producer 角色必须标记 type=producer

#### edges 段增删改
- 边的语法正确性（四种原子模式）
- when 表达式格式：`when: result.verdict == "xxx"`
- max_executions 声明（如有 backward 边）
- 同步汇入语法：`[A, B, C] → D`

#### knowledge 段调整
- inject_to 角色名与 roles 定义一致
- inject 路径与实际文件路径一致

### 三、skill.md 编写规范

#### 无硬编码路径原则
- **不写具体文件路径**——路径由 dispatch 注入到 task_prompt 的输入文件和产出物路径中
- skill.md 中写路径会与 dispatch 注入产生重复权威源，改路径时容易遗漏

#### 语义描述为主
skill.md 应包含以下语义内容：
- **角色定位**：这个角色是什么，做什么
- **执行步骤**：具体操作步骤（不是"分析需求"，而是"按知识文档中的方法论逐项分析"）
- **verdict 判定规则**：什么情况产出什么 verdict
- **设计约束**：不可逾越的边界
- **自检项**：产出前应自查的检查项

#### 可选输入标注约定
- `#[可选输入]` 标注仅在特定条件时存在的输入
- 例如：`#[可选输入] 综合裁决书：回退时存在`

### 四、schema.json 调整规范

#### verdict enum
- 必须与 edges when 表达式完全一致（编译器自动同步但手动修改时需检查）
- fail 为系统保留词，不写入 enum

#### _required_files
- 与 outputs 声明对齐
- 列出该角色必须产出的文件名称

#### format 约束定义
- `result.verdict`：必填字符串，enum 值从 edges when 提取
- `result.summary`：必填字符串
- `result` 下的 properties 根据角色需要定义

### 五、改造类型执行指南

#### 新增角色完整流程
1. 在 roles 中定义角色（type/confirm/inputs/outputs）
2. 在 edges 中添加连接边（上游→新角色→下游）
3. 声明 when 条件（如有条件路由）
4. 设置 max_executions（如有 backward 边）
5. 生成 skill.md + schema.json

#### 删除角色完整流程
1. 从 roles 中删除角色定义
2. 清理 edges 中所有引用该角色的边（上游边和下游边）
3. 检查 knowledge inject_to 是否有该角色引用并清理
4. 验证 DAG 无孤儿引用（BFS 检查）

#### 修改角色完整流程
1. 更新对应 skill.md（执行逻辑/步骤）
2. 更新对应 schema.json（如 verdict 变化需同步 enum）
3. 如 verdict 变化，同步修改 edges when 表达式
4. 如 inputs/outputs 变化，同步修改 app.yaml roles 段

### 六、compiler.py 调用规范

修改完成后必须调用 compiler.py 重新编译目标 APP：
1. 检查三个编译产物均生成且内容非空：
   - ROUTER.json：路由表
   - registry.json：物料注册表
   - manifest.json：目录模板
2. 编译失败则记录错误信息
3. 编译成功后检查 ROUTER.json 中的 transitions 是否与预期一致

### 七、checksum 比对验证方法

验证增量修改原则的方法（不依赖 git）：
1. **改造前扫描**：目标 APP 目录所有文件计算 SHA-256
2. **改造后重新计算**：同方法
3. **对比**：checksum 变化文件清单 vs 改造方案预期修改清单
4. **一致率必须 100%**：实际变更文件集 = 预期变更文件集

## 示例

### 示例：修改一个角色的 skill.md（修改角色类型）

**改造方案条目**：
- 目标文件：roles/模拟验证者/skill.md
- 改动类型：修改
- 改动内容摘要：增加向后兼容性检查的具体步骤
- 改动原因：需求要求验证向后兼容性

**执行步骤**：
1. 读取现有 skill.md
2. 在执行步骤中增加"向后兼容性检查"章节
3. 保存修改
4. 计算 SHA-256 对比，确认只有该文件变更
5. 编译目标 APP，检查产物完整

### 示例：新增角色

**执行步骤**：
1. 在 app.yaml roles 段添加角色定义
2. 在 app.yaml edges 段添加连接边
3. 创建 roles/新角色/skill.md
4. 创建 roles/新角色/schema.json
5. 编译目标 APP
6. checksum 比对确认变更文件集

## 检查清单

改造执行者可用的自检项：

- [ ] 是否只修改了方案涉及的文件？
- [ ] app.yaml 修改后 YAML 格式是否正确？
- [ ] 新增角色是否同时生成了 skill.md + schema.json？
- [ ] 删除角色是否清理了所有边引用和 knowledge inject_to？
- [ ] schema.json verdict enum 是否与 edges when 一致？
- [ ] skill.md 是否无硬编码路径？
- [ ] 编译是否通过？三个产物是否存在？
- [ ] checksum 比对一致率是否 100%？
