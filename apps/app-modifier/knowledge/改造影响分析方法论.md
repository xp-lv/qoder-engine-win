# 改造影响分析方法论

## 目的

本文档定义从改造要求推导到受影响的文件/角色/边/verdict 集合的影响分析方法。帮助改造分析师系统性分析改造的波及范围，产出完整的改造方案。

## 适用角色

- 改造分析师：分析现有 APP 架构、设计改造方案时参照本方法论

## 核心内容

### 一、改造类型六分类及识别方法

| 改造类型 | 识别方法 | 典型需求描述 |
|----------|----------|-------------|
| 新增角色 | 需求中要求扩展能力 | "增加一个 XXX 角色"、"增加 XXX 功能" |
| 删除角色 | 需求中要求精简流程 | "移除 XXX 角色"、"简化 XXX 步骤" |
| 修改角色 | 需求中要求更新执行逻辑/格式约束 | "修改 XXX 的执行逻辑"、"调整 XXX 的输出格式" |
| 调整编排 | 需求中要求改路由/流程 | "修改 XXX 的路由"、"增加迭代回路"、"调整并行结构" |
| 更新知识 | 需求中要求增删知识文档 | "更新 XXX 方法论"、"增加 XXX 规范" |
| 修复缺陷 | 已知结构性问题 | "修复 verdict 不一致"、"消除死角色"、"修复孤儿边" |

### 二、影响范围推导方法

#### 向上游 BFS 追溯（谁给我数据）
从改动点出发，向上游追溯输入来源角色：
1. 读取改动角色的 inputs 列表
2. 对每个 input 物料，在所有角色的 outputs 中查找来源
3. 递归追溯到入口角色
4. 记录沿途经过的所有角色和边

#### 向下游 BFS 追踪（谁消费我的数据）
从改动点出发，向下游追踪输出消费者角色：
1. 读取改动角色的 outputs 列表
2. 在 edges 中查找引用该角色的边
3. 追踪到下游角色
4. 递归追踪到完成节点
5. 记录沿途经过的所有角色和边

### 三、文件影响分析矩阵

| 改造类型 | 受影响文件 |
|----------|-----------|
| 新增角色 | app.yaml roles段 + edges段 + 新 skill.md + 新 schema.json |
| 删除角色 | app.yaml（roles + edges）+ 所有引用该角色的 edges + knowledge inject_to |
| 修改角色 | 对应 skill.md + schema.json（如 verdict 变化还需修改 edges） |
| 调整编排 | app.yaml edges段 + 受影响角色的 schema.json verdict enum |
| 更新知识 | app.yaml knowledge段 + 对应 knowledge/ 文档 |
| 修复缺陷 | 视缺陷类型而定（通常涉及 app.yaml + schema.json） |

### 四、边与 verdict 影响分析

#### 新增角色
- 需要新 verdict 路由吗？（检查是否需要条件边 when 表达式）
- 新角色的上游角色是否有对应出边？
- 新角色的下游角色是否有对应入边？
- 是否需要设置 max_executions（如有 backward 边）？

#### 删除角色
- edges 中所有引用该角色的边是否清理（上游边和下游边）？
- 删除后是否有角色断连（上游角色无出边 / 下游角色无入边）？
- knowledge inject_to 是否有该角色引用需要清理？

#### 修改 verdict
- 条件边 verdict 值是否在 schema enum 中有定义？
- 新增的 verdict 是否有对应路由目标？
- 删除的 verdict 是否有残留的 when 表达式引用？

### 五、向后兼容性评估框架

1. **改造前 DAG 可达集**：BFS 从入口出发的可达节点集合
2. **改造后 DAG 可达集**：同方法计算
3. **对比原则**：
   - 原有路径（未被改造涉及的角色→完成节点）不丢失
   - 新增角色不影响原有路径
   - 删除角色后原有路径如有断裂，需修复（增加旁路边）
4. **输出**：pass（兼容）/ broken_paths（列出丢失的路径）

### 六、风险评估清单（四类风险）

| 风险类型 | 检查内容 | 判据 |
|----------|----------|------|
| 死循环风险 | backward 边 max_executions 是否合理设置 | 验证回退 max:3 / 改造迭代 max:3(R8)+2(R9) 合计5 / 全局迭代 max:2 |
| 孤儿边风险 | 删除角色后所有边引用是否清理完毕 | edges 中无指向已删除角色的边 |
| 死角色风险 | 新增角色是否有入边和出边 | 新角色至少有一条入边和一条出边 |
| verdict 不完备风险 | 条件边 verdict 是否在 schema enum 中有定义 | 所有 when verdict 在 enum 中存在 |

### 七、改造方案文档结构标准

改造方案必须包含以下结构：

```
改造方案:
  改造类型: <六分类之一>
  改动清单:
    - 目标文件: <文件路径>
      改动类型: 新增 / 删除 / 修改
      改动内容摘要: <具体改什么>
      改动原因: <为何要改>
  影响范围:
    上下游角色: <受影响角色列表>
    受影响边: <新增/删除/修改的边>
    verdict变化: <新增/删除的 verdict 值>
  向后兼容性评估:
    改造前可达集: <角色列表>
    改造后可达集: <角色列表>
    兼容性结论: pass / broken_paths
  风险评估:
    死循环风险: pass / risk_detail
    孤儿边风险: pass / risk_detail
    死角色风险: pass / risk_detail
    verdict不完备风险: pass / risk_detail
```

## 示例

### 示例：新增一个"安全审阅者"角色的影响分析

**需求**：在现有 APP 的并行审阅扇出中增加一个安全维度的审阅角色。

**影响分析**：
1. **改造类型**：新增角色
2. **上游追溯**：新角色的输入来源是改造执行报告 → 上游角色是改造执行者
3. **下游追踪**：新角色的输出消费者是综合裁决者（JOIN 同步汇入）
4. **受影响文件**：
   - app.yaml roles 段：新增"安全审阅者"角色定义
   - app.yaml edges 段：新增"模拟验证者→安全审阅者"边 + 更新 JOIN 边
   - 新增 roles/安全审阅者/skill.md
   - 新增 roles/安全审阅者/schema.json
   - 综合裁决者 schema.json 无需修改（verdict 值域仍为 confirmed/issues_found）
5. **向后兼容性**：新增角色不影响原有路径 → pass
6. **风险**：死角色风险 pass（有入边和出边）/ verdict 风险 pass（confirmed/issues_found）

## 检查清单

改造分析师可用的自检项：

- [ ] 改造类型是否正确识别（六分类之一）？
- [ ] 影响范围是否覆盖所有上下游角色和边？
- [ ] 文件影响矩阵是否列出所有受影响文件？
- [ ] 边与 verdict 影响是否分析（新增/删除/修改）？
- [ ] 向后兼容性是否评估（改造前后可达集对比）？
- [ ] 四类风险是否逐项分析？
- [ ] 改造方案文档结构是否完整（改动清单 + 影响范围 + 兼容性 + 风险）？
