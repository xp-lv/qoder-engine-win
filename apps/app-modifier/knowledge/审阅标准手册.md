# 审阅标准手册

## 目的

本文档定义治理层三路审阅角色（改造红队 / 结构审阅者 / 合规审阅者）的统一审阅标准与对抗测试规范，确保 verdict 值域统一，findings 格式一致，综合裁决者接收一致的输入格式。

## 适用角色

- 改造红队：执行对抗维度压力测试时参照本手册
- 结构审阅者：执行结构维度审阅时参照本手册
- 合规审阅者：执行合规维度审阅时参照本手册

## 核心内容

### 一、verdict 统一规范

R5 改造红队 / R6 结构审阅者 / R7 合规审阅者三路审阅 verdict 值域**统一**为：

| verdict | 语义 | 触发条件 |
|---------|------|----------|
| `confirmed` | 通过审阅，无重大问题 | 审阅维度全部 pass，无 critical/major 级 findings |
| `issues_found` | 发现需修复的问题 | 发现 critical/major/minor 级 findings |

**verdict 字段位置**：写在 `result.verdict` 中

> 三路审阅 verdict 统一确保 R8 综合裁决者接收一致的输入格式。

### 二、结构维度审阅项（R6 结构审阅者）

#### DAG 拓扑完整性
- **可达性校验**：BFS 从入口到完成节点，所有节点可达
- **死角色检测**：无角色有入边无出边且非完成节点
- **孤儿边检测**：无边指向不存在的角色
- **终态出口验证**：每条路径可达完成节点
- **同步约束验证**：fork-join 闭合，input_groups 语义正确

#### 角色定义完备性
- **outputs 字段**：每个角色都有 outputs 定义
- **type 字段**：producer 角色正确标记 type=producer
- **confirm 字段**：每个角色有 confirm 字段
- **inputs 来源**：角色 inputs 有上游 outputs 或 knowledge inject 来源

#### 边引用一致性
- **角色名对应**：edges 中角色名与 roles 定义完全对应
- **无残留引用**：删除角色后 edges 中无指向该角色的边
- **verdict 路由覆盖**：每条条件边 when 的 verdict 在 schema enum 中有定义

### 三、合规维度审阅项（R7 合规审阅者）

#### 唯一权威源检查
- **路由目标**：路由目标只在 edges 中声明，不在角色定义中
- **verdict 声明**：verdict 只在 when 表达式中声明，不在角色 verdicts 列表中
- **循环上限**：max_executions 只在边的属性中声明，不在角色级 loop 字段中
- **物料携带**：carries 由编译器自动推导，不在 registry 中手写

#### 编译期确定性检查
- **路由确定性**：所有路由由 compiler.py 编译期计算，运行时不推断
- **物料确定性**：所有物料路径编译期确定
- **同步约束确定性**：input_groups 编译期计算

#### 物料分类检查
- **deliverable**：用户可读最终产出（需求文档、架构文件、报告等）
- **process**：角色间中间报告（审阅报告、验证报告等）
- **分类合理性**：中间报告不误标为 deliverable，最终产出不误标为 process

#### skill.md 无硬编码路径检查
- **路径注入**：skill.md 中不硬编码具体文件路径
- **语义描述**：skill.md 只包含语义描述（角色定位/执行步骤/verdict 判定规则）

### 四、对抗维度审阅项（R5 改造红队）

#### 死循环检测
- backward 边 max_executions 是否合理：
  - 验证回退默认 max: 3
  - 改造迭代回路 max: 3（R8 needs_revision→R2）+ 2（R9 overturned→R2），合计 ≤ 5
  - 全局迭代回路 max: 2
- 构造可触发死循环的 verdict 组合，验证 max_executions 能否正确掐断
- 检查未被 max_executions 约束的潜在循环路径

#### 不可达路径检测
- 删除角色后是否产生断边或孤立节点
- BFS 从入口出发，检查所有角色是否可达
- 新增角色缺少入边或出边（死角色）

#### verdict 路由完备性
- 每条条件边 when 的 verdict 值在对应角色 schema enum 中有定义
- 无条件边有对应的默认 confirmed 出边
- 不存在 verdict 值声明了但没有对应路由目标的情况

#### producer 展开正确性
- producer 角色正确展开为执行 + 校验两步
- 校验角色的 confirmed 路由正确（fail 为系统保留词，由 Gate FAIL 触发，非校验角色输出）
- 校验角色 inputs 继承执行角色 outputs

#### knowledge inject_to 有效性
- inject_to 中的角色名全部存在于 roles 定义中
- inject 路径在 manifest 中有对应记录

### 五、findings 编写规范

每个 finding **必须**包含以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| 具体引用 | 角色名/边名/verdict值等具名标识 | "改造红队→综合裁决者边引用了 verdict=issues_found" |
| 严重级别 | critical=阻断性 / major=重要缺陷 / minor=建议性 | critical |
| 问题描述 | 具体问题描述 | "改造红队 schema.json 的 verdict enum 中缺少 issues_found" |
| 建议修复方案 | 具体修复建议 | "在改造红队 schema.json 的 verdict enum 中添加 issues_found" |

**不允许泛泛描述**：
- "结构有问题" → 缺乏具体引用
- "建议优化" → 缺乏具体问题和修复方案

### 六、审阅报告格式标准

```
顶层字段:
  result.verdict: confirmed / issues_found
  result.summary: "<审阅概述>"

审阅报告主体:
  审阅摘要: "<整体评价>"
  各审阅项逐项结果: [pass / fail]
  findings: [<findings 列表>]
```

**最小内容要求**：
- 正文 >= 200 字
- 含 >= 3 个具名引用（具体角色名/边名/verdict 值）

### 七、向后兼容性专项检查

改造是否破坏原有可达路径——对比改造前后 DAG 可达集：
1. 改造前可达集：BFS 从入口到完成节点的可达节点集合
2. 改造后可达集：同方法
3. 原有路径（未被改造涉及的角色→完成节点）必须仍可达
4. 输出 compatibility_check 结果（pass / broken_paths 列表）

## 示例

### 示例：结构审阅者发现 issues_found

```
result.verdict: "issues_found"
result.summary: "结构审阅发现 2 个问题：综合裁决者 schema.json verdict enum 不完整 + 模拟验证者→改造执行者 backward 边缺少 max_executions"

findings:
  - 具体引用: "综合裁决者 schema.json verdict enum"
    严重级别: major
    问题描述: "综合裁决者 schema.json 的 verdict enum 只有 [confirmed]，缺少 needs_revision 和 requirements_revision_needed，但 edges 中有对应的 when 表达式"
    建议修复方案: "在综合裁决者 schema.json 的 verdict enum 中添加 needs_revision 和 requirements_revision_needed"

  - 具体引用: "模拟验证者→改造执行者 backward 边"
    严重级别: minor
    问题描述: "验证回退边缺少显式的 max_executions 声明（编译器默认 3，但建议显式声明）"
    建议修复方案: "在 edges 中为该边添加 max_executions: 3"
```

## 检查清单

改造红队/结构审阅者/合规审阅者可用的自检项：

- [ ] verdict 是否在 {confirmed, issues_found} 范围内？
- [ ] 每个 finding 是否包含具名引用 + 严重级别 + 问题 + 修复方案？
- [ ] 报告正文是否 >= 200 字，含 >= 3 个具名引用？
- [ ] 审阅维度是否全部检查（各自负责的维度）？
- [ ] 向后兼容性是否检查？
- [ ] result.verdict 和 result.summary 是否填写？
