# 七维模拟验证方法论

## 目的

本文档定义对改造后 APP 执行七维度 dry-run 校验的逐步方法论，以及向后兼容性检查的标准流程。确保改造后的 APP 在结构、数据流、同步约束等维度均正确无误。

## 适用角色

- 模拟验证者：执行七维校验 + 向后兼容性检查时参照本方法论逐维度操作
- 改造红队：执行对抗压力测试时参照本方法论理解校验维度和极端场景

## 核心内容

### 维度一：DAG 可达性校验

**目的**：确保所有节点可达完成节点，无死角色、无孤儿边、无有向环。

**操作步骤**：
1. 从入口（producer 角色）出发执行 BFS 遍历
2. 记录所有可达节点
3. 检测死角色：有入边但无出边且非完成节点的角色
4. 检测孤儿边：edges 中指向不存在角色的边
5. 验证 backward 边不构成有向环（backward 边指向的上游角色不可通过 normal 边回到当前角色）

**判据**：所有角色可达完成节点 → pass；存在死角色/孤儿边/有向环 → fail

### 维度二：verdict 完备性校验

**目的**：确保每条条件边有明确的 verdict 值，verdict 值在 schema enum 中有定义。

**操作步骤**：
1. 提取所有 when 表达式中的 verdict 值
2. 与各角色 schema.json 的 verdict enum 逐一对比
3. 无条件边默认 verdict=confirmed，检查是否有对应出边
4. fail 为系统保留词，Gate FAIL 自动生成 backward 边，不写入 schema enum

**判据**：所有 when verdict 值在 schema enum 中有定义且一致率 100% → pass

### 维度三：数据流完整性校验

**目的**：确保每个角色的输入物料有上游来源。

**操作步骤**：
1. 对每个角色 inputs 中的每个物料项：
   - 在上游角色 outputs 中查找来源
   - 在 knowledge inject 中查找来源
2. 无来源则标记 data_flow_broken
3. 可选输入（标注 `#[可选输入]`）不强制校验

**判据**：所有必需 inputs 有来源 → pass；存在 data_flow_broken → fail

### 维度四：同步约束正确性校验

**目的**：确保 input_groups 的 AND/OR 语义正确，fork-join 闭合。

**操作步骤**：
1. 验证 input_groups 语义：
   - 组内 AND：同一组内所有来源全部完成才能调度
   - 组间 OR：任一组满足即可调度
2. fork 扇出必须有对应 join 汇聚
3. 检测悬挂 fork（扇出后无汇聚）
4. 检测悬挂 join（无对应 fork 的同步汇入）

**判据**：所有 fork-join 闭合、input_groups 语义正确 → pass

### 维度五：producer 展开正确性校验

**目的**：确保 producer 角色正确自动展开为执行 + 校验两步。

**操作步骤**：
1. 检查 producer 角色是否自动生成 {角色名}（校验）角色
2. 校验角色 inputs 继承执行角色 outputs
3. 校验角色 outputs 自动生成 {角色名}-validation.json
4. 校验边 confirmed→下游、fail→回退执行角色

**判据**：producer 展开符合规范 → pass

### 维度六：knowledge 注入正确性校验

**目的**：确保 knowledge 段的 inject_to 映射与实际角色名一致。

**操作步骤**：
1. 检查 knowledge 段 inject_to 列表中的每个角色名是否在 roles 定义中存在
2. 检查 inject 路径（knowledge/xxx.md）是否在 manifest 中有对应记录
3. 缺省 inject_to 则不注入

**判据**：所有 inject_to 角色名存在且路径有效 → pass

### 维度七：物料分类正确性校验

**目的**：确保 deliverable / process 类型分配合理。

**操作步骤**：
1. deliverable（用户可读最终产出：需求文档、架构文件、报告等）
2. process（角色间中间报告：审阅报告、验证报告等）
3. 检查中间报告是否误标为 deliverable
4. 检查最终产出是否误标为 process

**判据**：物料分类合理 → pass

### 向后兼容性检查方法

**目的**：确保改造未破坏原有可达路径。

**操作步骤**：
1. **计算改造前 DAG 可达集**：BFS 从入口出发的可达节点集合
2. **计算改造后 DAG 可达集**：同方法
3. **对比两个集合**：未被改造涉及的角色→完成节点的路径必须仍可达
4. **输出结果**：pass（兼容）/ broken_paths（列出丢失的路径）

**判据**：原有路径不丢失 → pass；原有路径丢失 → fail

## 示例

### 示例：验证一个删除角色后的兼容性

假设目标 APP 原有角色 [A, B, C, D, E]，改造删除了角色 C：

1. 改造前可达集：{A, B, C, D, E, 完成}
2. 改造后可达集：{A, B, D, E, 完成}
3. 对比：角色 C 被删除，这是预期行为
4. 检查：A→B→D→E→完成 路径仍可达？
   - 如果 B→D 边存在 → pass
   - 如果 B→D 边不存在（删除 C 时未修复 B 的下游） → broken_paths: [B 无法到达 D]

### 示例：检测 verdict 不完备

假设某角色有出边 `when: result.verdict == "custom_verdict"`，但该角色 schema.json 的 verdict enum 只有 `["confirmed"]`：

- verdict "custom_verdict" 不在 enum 中 → 维度二 fail
- 修复：在 schema.json 中添加 "custom_verdict" 到 enum，或在 edges 中修正 when 表达式

## 检查清单

模拟验证者/改造红队可用的自检项：

- [ ] 维度一：是否 BFS 遍历了所有节点？死角色/孤儿边/有向环是否检测？
- [ ] 维度二：所有 when verdict 是否与 schema enum 对比？
- [ ] 维度三：所有必需 inputs 是否有上游来源？
- [ ] 维度四：fork-join 是否闭合？input_groups 语义是否正确？
- [ ] 维度五：producer 角色是否正确展开？
- [ ] 维度六：inject_to 角色名是否全部存在？
- [ ] 维度七：物料分类是否合理？
- [ ] 向后兼容性：改造前后可达集是否对比？原有路径是否不丢失？
