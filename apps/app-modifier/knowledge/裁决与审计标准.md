# 裁决与审计标准

## 目的

本文档定义综合裁决者（R8）的三路径裁决规则与裁决审计者（R9）的双路径审计规则，包括迭代回路触发判据和迭代超限处理机制。

## 适用角色

- 综合裁决者：做三路径综合裁决时参照本标准
- 裁决审计者：做双路径审计复核时参照本标准

## 核心内容

### 一、R8 综合裁决者三路径判定规则

#### confirmed（建议放行）
- **触发条件**：三路审阅（R5 改造红队 / R6 结构审阅者 / R7 合规审阅者）均无 critical/major 级 findings，或问题已在本轮修复
- **路由目标**：→ R9 裁决审计者（建议放行，需审计复核）
- **语义**：改造合格，可以放行，但需经审计者最终复核

#### needs_revision（回退修改方案）
- **触发条件**：存在需修改方案的 major 级 findings，改造方向正确但方案需调整
- **路由目标**：→ R2 改造分析师（backward, max_executions: 3）
- **语义**：改造方向正确，但具体方案有问题

#### requirements_revision_needed（全局回退需求层）
- **触发条件**：
  - 同类 findings 连续 >= 3 轮未消解
  - 或 needs_revision backward 边执行次数已达上限（3 次）
- **路由目标**：→ R1 改造需求接收者（backward, max_executions: 2）
- **语义**：改造方向/需求本身有误，需要从需求层重新定义

### 二、R8 裁决输入分析规范

#### 合并三路 findings
1. 收集 R5 改造红队 / R6 结构审阅者 / R7 合规审阅者的所有 findings
2. 按严重级别排序：critical > major > minor
3. 来源标注：每个 finding 标注来源（R5 / R6 / R7）

#### 关键规则
- critical 级 findings 直接触发 needs_revision 或 requirements_revision_needed
- 裁决需引用三路审阅的具体 findings（具名引用 R5/R6/R7 的产出）
- 不可泛泛说"有问题"，必须引用具体产出条目

### 三、R8 裁决书格式标准

```
顶层字段:
  result.verdict: confirmed / needs_revision / requirements_revision_needed
  result.summary: "<裁决理由概述>"

综合裁决书主体:
  三路审阅汇总:
    改造红队_verdict: confirmed / issues_found
    结构审阅者_verdict: confirmed / issues_found
    合规审阅者_verdict: confirmed / issues_found
  合并findings（按严重级别排序）:
    - 来源: R5 / R6 / R7
      具体引用: <具名标识>
      严重级别: critical / major / minor
  裁决依据:
    选择理由: <为何选择此 verdict>
    影响 findings: <哪些 findings 影响了裁决>
  迭代计数（如有回退）:
    改造迭代回路(R8 needs_revision边)已执行次数: N / 3
    改造迭代回路(R9 overturned边)已执行次数: N / 2（如已知）
    改造迭代回路(R8+R9合计)总执行次数: N / 5
    全局迭代回路已执行次数: N / 2
```

### 四、R9 裁决审计者审计范围严格限定

**R9 仅审计 R8 的 confirmed 裁决（放行决策）。**

- 拓扑中唯一到达 R9 的边：R8 confirmed → R9
- R8 的 needs_revision 和 requirements_revision_needed **不经 R9 审计**，直接路由至 R2/R1
- R9 **不审计回退决策**，只复核**放行决策**

### 五、R9 双路径判定规则

#### upheld（维持 confirmed 原判，同意放行）
- **触发条件**：经全链路对抗复核，确认 R8 的 confirmed 裁决合理——改造确实合格，可以放行
- **路由目标**：→ R10 改造报告生成者
- **语义**：审计者同意 R8 的放行决策，改造可以正式完成

#### overturned（推翻 confirmed 原判，认为不应放行）
- **触发条件**：从最严苛角度发现 R8 遗漏了 critical/major 问题，改造实际不合格不应放行
- **路由目标**：→ R2 改造分析师（backward, max_executions: 2）
- **语义**：审计者推翻 R8 的放行决策，改造需回退修改

### 六、R9 审计方法论

#### 全链路对抗复核原则
1. **不信任 R8 判断**：不直接接受 confirmed 判断
2. **独立重新审视**：重新审视 R5/R6/R7 三路 findings 的完整内容
3. **遗漏检测**：检查 R8 是否遗漏了 R5/R6/R7 中的 critical/major 问题
4. **降级检测**：检查 R8 是否错误地将 critical 降级为 major 或 minor
5. **独立发现**：检查是否有 R5/R6/R7 均未发现但独立发现的 critical/major 问题
6. **最严苛标准**：从最严苛角度检验改造质量，宁严勿宽

### 七、R9 审计裁决书格式标准

```
顶层字段:
  result.verdict: upheld / overturned
  result.summary: "<审计结论概述>"

审计裁决书主体:
  审计对象: "R8 综合裁决者的 confirmed 裁决"
  独立复核结果:
    R5 改造红队 findings 复核: <是否有遗漏/降级>
    R6 结构审阅者 findings 复核: <是否有遗漏/降级>
    R7 合规审阅者 findings 复核: <是否有遗漏/降级>
    独立发现的新问题: <如有>
  审计结论: upheld / overturned
  审计依据:
    结论理由: <为何 upheld 或 overturned>
    关键发现: <影响审计结论的具体发现>
  新增 findings（如有）:
    - 具体引用: <具名标识>
      严重级别: critical / major
      建议修复方案: <修复建议>
```

### 八、迭代回路触发判据与计数规则

| 迭代回路 | backward 边 | max_executions | 共享计数 |
|----------|------------|----------------|----------|
| 验证回退 | R4 defects_detected → R3 | 3 | 独立计数 |
| 改造迭代回路 | R8 needs_revision → R2 | 3 | 两条 backward 边合计 ≤ 5 |
| 改造迭代回路 | R9 overturned → R2 | 2 | 同上（3+2=5） |
| 全局迭代回路 | R8 requirements_revision_needed → R1 | 2 | 独立计数 |

#### 触发全局回退判据
- needs_revision backward 边执行次数达上限（3 次）
- 或同类 findings 连续 >= 3 轮未消解

### 九、迭代超限处理机制

当全局迭代回路（R8→R1, max_executions: 2）耗尽后：
1. 所有 backward 边被 router 掐断
2. 引擎在终止前调用 R10 改造报告生成者生成 iteration_exhausted 模式失败报告
3. 失败报告包含：
   - 迭代超限诊断
   - 已尝试方案
   - 失败原因

### 十、语义一致性声明

- R9 审计的"原判"始终是 R8 的 **confirmed**（拓扑中唯一到达 R9 的边）
- upheld = 维持该 confirmed（放行）→ R10
- overturned = 推翻该 confirmed（回退）→ R2
- 不存在审计 needs_revision 或 requirements_revision_needed 的场景

## 示例

### 示例：R8 confirmed → R9 upheld（正常放行路径）

R5/R6/R7 三路审阅均为 confirmed，无 critical/major findings：

1. R8 合并 findings：无 critical/major，只有 2 个 minor
2. R8 裁决：confirmed（建议放行）→ R9
3. R9 独立复核：确认 R8 判断合理，无遗漏
4. R9 裁决：upheld（维持放行）→ R10

### 示例：R8 confirmed → R9 overturned（审计推翻）

R5 发现了 1 个 major finding，但 R8 错误降级为 minor 并裁决 confirmed：

1. R8 合并 findings：将 R5 的 major 降级为 minor
2. R8 裁决：confirmed（错误放行）→ R9
3. R9 独立复核：发现 R8 降级了 R5 的 major finding
4. R9 裁决：overturned（推翻放行）→ R2（回退修改）

### 示例：R8 needs_revision（改造迭代回路）

R6 发现了 2 个 major findings（结构维度），R7 发现了 1 个 major（合规维度）：

1. R8 合并 findings：3 个 major（R6:2 + R7:1），0 个 critical
2. R8 裁决：needs_revision（方向正确但方案需调整）→ R2
3. R2 调整方案后重新执行

## 检查清单

综合裁决者/裁决审计者可用的自检项：

- [ ] R8：三路审阅报告是否全部读取？
- [ ] R8：findings 是否按严重级别排序并标注来源？
- [ ] R8：裁决是否引用了具体 findings（R5/R6/R7 具名引用）？
- [ ] R8：是否感知了迭代回路执行次数？
- [ ] R9：是否独立复核了三路 findings（而非直接接受 R8 判断）？
- [ ] R9：是否检查了 R8 遗漏或降级的 critical/major 问题？
- [ ] R9：审计对象是否严格限定为 R8 confirmed 裁决？
- [ ] result.verdict 和 result.summary 是否填写？
