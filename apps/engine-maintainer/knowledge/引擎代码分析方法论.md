# 引擎代码分析方法论 — 从维护需求定位受影响组件

> **本文档用途**：直接支撑引擎诊断师产出结构化诊断报告（AC-3/AC-4）。提供从维护需求定位受影响引擎组件的逐步方法论，含依赖追踪、数据流分析、影响矩阵构建方法。

---

## 1. 目的

本文档定义从维护需求到引擎代码受影响组件的系统性分析方法论。引擎诊断师按本方法论的步骤执行，确保受影响组件清单无遗漏、影响范围分析完整、问题根因定位准确。

## 2. 适用角色

- **引擎诊断师**：本方法论是诊断师的核心工作方法，按步骤执行产出诊断报告

## 4. 需求→组件映射方法

### 4.1 关键词提取

从维护需求中提取以下类型的关键词：
- **函数名**：如 `compile`, `route`, `advance`, `dispatch`
- **类名**：如 `Router`, `Orchestrator`, `Gate`
- **行为描述**：如"编译链路"、"verdict 路由"、"状态迁移"、"Gate 校验"
- **文件名**：如 `compiler.py`, `router.py`
- **概念名**：如 `input_groups`, `edge_counts`, `carries`, `max_executions`

### 4.2 搜索策略与优先级

按以下优先级在代码中搜索受影响组件：

1. **精确匹配**（最高优先级）：需求中直接提到的文件名/函数名/类名
2. **语义匹配**：行为描述对应的代码区域（如"编译链路" → compiler.py）
3. **概念匹配**：概念名在代码中的引用点（如 `input_groups` 在 compiler.py 和 orchestrator.py 中）
4. **模式匹配**（最低优先级）：模糊关键词的全文搜索

### 4.3 操作步骤

```
步骤 1：从需求文档提取关键词列表
步骤 2：按优先级在 engine/scripts/、engine/sdk/、engine/schemas/ 中搜索
步骤 3：对每个匹配结果，记录文件路径、行号范围、匹配类型
步骤 4：初步形成受影响组件清单
```

## 5. 依赖追踪技术

### 5.1 正向追踪（组件 A 调用了什么）

从受影响组件 A 出发，追踪它调用的所有其他组件：
```
A → B → C → D
A → E → F
```

方法：读取组件 A 的代码，识别所有 import、函数调用、类引用。

### 5.2 反向追踪（什么调用了组件 A）

从受影响组件 A 出发，追踪哪些组件调用了它：
```
X → A
Y → Z → A
```

方法：在全部代码中搜索组件 A 的名称，找到所有引用点。

### 5.3 递归分析

对正向和反向追踪发现的每个新组件，递归执行追踪，直到不再发现新组件。

### 5.4 追踪完整性判据

- engine/scripts/ 全部 .py 文件均已检查（无论是否受影响）
- 每个文件标注"受影响 / 不受影响"及原因
- 受影响组件的依赖链完整（无遗漏的上下游组件）

## 6. 数据流分析方法

### 6.1 变量传播路径

追踪关键变量/参数从输入到输出的传播路径：
```
输入参数 → 函数 A 处理 → 存入状态 → 函数 B 读取 → 输出
```

### 6.2 变更影响下游数据流

识别变更如何影响下游数据流：
- 修改输入参数格式 → 影响所有读取该参数的函数
- 修改输出格式 → 影响所有消费该输出的函数
- 修改中间状态 → 影响所有读取该状态的函数

### 6.3 影响范围树

构建以变更点为根的影响范围树：
```
变更点（root）
├── 直接影响：调用变更函数的代码
│   ├── 间接影响：调用"直接影响"的代码
│   └── ...
└── 直接影响：被变更函数调用的代码
    ├── 间接影响：被"直接影响"调用的代码
    └── ...
```

## 7. 影响矩阵构建方法

### 7.1 矩阵结构

| 文件 | 直接修改 | 间接影响 | 接口变更 | 行为变化 |
|---|---|---|---|---|
| compiler.py | ✓ | — | ✓ | ✓ |
| router.py | — | ✓ | — | ✓ |
| ... | ... | ... | ... | ... |

### 7.2 影响类型定义

| 影响类型 | 定义 | 示例 |
|---|---|---|
| 直接修改 | 该文件的代码需要被修改 | 修改 compiler.py 的函数实现 |
| 间接影响 | 该文件不需修改，但其行为因上游变更而改变 | router.py 不改，但 compiler 输出变了导致路由行为变 |
| 接口变更 | 该文件暴露的接口签名变化 | sdk.py 的函数签名改变 |
| 行为变化 | 该文件的运行时行为变化 | gate.py 不改，但校验逻辑因 schema 变更而变化 |

### 7.3 构建步骤

```
步骤 1：列出全部 .py 文件为矩阵行
步骤 2：对每个文件，判断四种影响类型
步骤 3：填写矩阵（✓/—）
步骤 4：对每个 ✓ 标注具体的影响描述
```

## 8. 问题根因定位方法

### 8.1 从症状到根因

```
症状（报错/异常行为）
  ↓ 沿调用链逆向
中间环节（传递了错误数据）
  ↓ 继续逆向
根因（产生错误数据的代码位置）
```

### 8.2 定位技巧

- **日志分析**：从错误信息中提取文件名、行号、函数名
- **断点推断**：在关键函数入口/出口推断数据状态
- **最小复现**：构造触发问题的最小输入
- **二分排查**：在调用链中二分定位问题引入点

### 8.3 根因报告要求

- 根因代码的文件路径和行号
- 从症状到根因的完整调用链
- 根因代码的当前行为和预期行为对比
- 修复方向建议

- [ ] engine/scripts/ 全部 .py 文件已列出（AC-3）

## 10. 示例

### 示例：需求"修复 compiler.py 中 producer 展开时校验角色 outputs 未继承的问题"

```
关键词提取：compiler.py, producer, 校验角色, outputs, 继承
搜索策略：
  1. 精确匹配：compiler.py 中 producer 展开相关代码
  2. 语义匹配：producer 展开 → _expand_producer 函数
  3. 概念匹配：outputs 继承 → 校验角色 outputs 生成逻辑

受影响组件：
  - compiler.py（直接修改）：_expand_producer 函数中 outputs 继承逻辑
  - registry.json（间接影响）：编译产物的校验角色 outputs 字段
  - schema.json（间接影响）：校验角色的 _required_files

影响矩阵：
  compiler.py: 直接修改 ✓, 接口变更 —, 行为变化 ✓
  gate.py: 直接修改 —, 间接影响 ✓（校验角色 outputs 变化影响 Gate 校验）

根因定位：
  症状：校验角色 outputs 为空
  调用链：compile → _expand_producer → 校验角色 outputs 生成
  根因：_expand_producer 中 outputs 继承逻辑遗漏了 type=process 物料
```
