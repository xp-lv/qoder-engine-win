# 变更设计方法论 — 从诊断报告推导变更方案

> **本文档用途**：支撑变更设计师产出结构化设计文档（AC-5/AC-6）。提供从诊断报告推导变更方案的逐步方法论，含变更分类、步骤设计、兼容性分析、回滚方案设计、SDK_SPEC 演进需求标注方法。

---

## 1. 目的

本文档定义从引擎诊断报告到结构化变更设计方案的方法论。变更设计师按本方法论的步骤执行，确保每个变更项包含完整要素（变更类型、步骤、影响接口、兼容性分析、回滚方案、SDK_SPEC 演进需求），可供代码实施者直接消费。

## 2. 适用角色

- **变更设计师**：本方法论是设计师的核心工作方法，按步骤执行产出变更设计文档

## 3. 变更类型分类方法

### 3.1 四种变更类型

| 类型 | 判定标准 | 设计模板特点 |
|---|---|---|
| **bug 修复** | 现有代码行为与预期不符 | 定位错误代码 → 修正 → 验证修正后行为正确 |
| **功能改进** | 现有功能需要增强或扩展 | 分析现有实现 → 设计增强方案 → 确保向后兼容 |
| **重构** | 代码结构需要优化但不改变行为 | 分析现有结构 → 设计目标结构 → 确保行为不变 |
| **性能优化** | 现有代码性能需要提升 | 分析性能瓶颈 → 设计优化方案 → 验证性能提升且行为不变 |

### 3.2 分类判据

- 代码产生了错误结果 → bug 修复
- 代码需要处理新的场景 → 功能改进
- 代码行为不变但结构需要调整 → 重构
- 代码行为不变但执行速度/内存需要改善 → 性能优化

### 3.3 注意事项

- 一个变更项可能同时涉及多种类型（如 bug 修复 + 功能改进）
- 每个变更项必须标注主要类型
- 重构和性能优化类变更必须确保行为不变（通过行为兼容性验证）

## 4. 变更步骤设计方法

### 4.1 从诊断报告提取变更目标

```
诊断报告 → 受影响组件清单 → 每个组件需要的变更操作 → 有序变更步骤
```

### 4.2 步骤依赖分析

分析步骤间的依赖关系：
- **必须先执行**：步骤 B 依赖步骤 A 的结果 → A 必须在 B 之前
- **可并行执行**：步骤 A 和 B 互不依赖 → 可同时进行
- **互斥**：步骤 A 和 B 不能同时执行 → 需要排序

### 4.3 步骤原子化原则

每个变更步骤应该是原子化的：
- **可独立验证**：每步完成后可以单独验证其正确性
- **可独立回滚**：每步可以单独回滚而不影响其他步骤
- **最小化范围**：每步只做一件事

### 4.4 步骤与代码文件的映射规范

每个步骤必须明确标注：
- 目标文件路径（如 `engine/scripts/compiler.py`）
- 操作类型（新增 / 修改 / 删除）
- 操作位置（函数名 / 类名 / 行号范围）
- 操作内容摘要

## 5. 向后兼容性分析框架（AC-6）

### 5.1 受影响 API/接口识别

从变更步骤中识别所有受影响的 API/接口：
- 函数签名变化
- 类接口变化
- 配置格式变化
- 数据结构变化

### 5.2 调用方兼容性分析

对每个受影响的 API/接口：
1. 搜索所有调用该 API 的代码（正向追踪）
2. 分析调用方是否需要修改
3. 标注兼容性判定（兼容 / 需修改 / 破坏性变更）

### 5.3 数据格式兼容性

| 兼容性类型 | 判定方法 | 判据 |
|---|---|---|
| 向后兼容 | 新代码能读旧数据 | 旧数据格式 ⊂ 新数据格式 |
| 向前兼容 | 旧代码能读新数据 | 新数据格式 ⊂ 旧数据格式 |
| 双向兼容 | 两者都兼容 | 格式不变 |

### 5.4 配置文件兼容性

- 检查配置文件格式是否变化
- 检查旧配置文件能否被新代码正确读取
- 检查新配置项是否有默认值

## 6. 回滚方案设计方法（AC-5）

### 6.1 回滚步骤设计

为每个变更项设计回滚步骤：

| 回滚方式 | 适用场景 | 操作 |
|---|---|---|
| git revert | 版本控制下的代码变更 | `git revert <commit>` |
| 手动恢复 | 部分文件修改 | 恢复原始代码 |
| 配置回退 | 配置文件变更 | 恢复原始配置 |

### 6.2 回滚后验证步骤

回滚后必须验证：
1. 编译正确性（compiler.py 无报错）
2. DAG 可达性（所有角色可达）
3. 行为正确性（核心功能正常）

### 6.3 回滚风险评估

- 回滚步骤本身的复杂度
- 回滚后是否有数据丢失风险
- 回滚是否需要停机

## 7. SDK_SPEC 演进需求标注方法（AC-15）

### 7.1 行为变化判定标准

代码变更涉及以下任一变化时，标注 SDK_SPEC 演进需求：

| 变化类型 | 具体表现 | 演进需求 |
|---|---|---|
| 接口签名变化 | 函数参数/返回值改变 | 需演进 |
| 状态迁移变化 | STATE.json 结构/语义改变 | 需演进 |
| verdict 语义变化 | verdict 值/路由规则改变 | 需演进 |
| 数据格式变化 | 编译产物格式改变 | 需演进 |
| 仅内部实现变化 | 不涉及上述任何变化 | 无需演进 |

### 7.2 标注格式

**有行为变化时**：
```
SDK_SPEC 演进需求：
  变更类型：接口签名变化
  影响接口：compiler.py compile()
  建议演进内容：新增 type 参数，默认值 deliverable
  优先级：高
```

**无行为变化时**：
```
SDK_SPEC 演进需求：无需演进（仅内部实现优化，不涉及行为变化）
```

## 8. 风险评估方法

### 8.1 风险评估矩阵

| | 低复杂度 | 中复杂度 | 高复杂度 |
|---|---|---|---|
| **大影响范围** | 中风险 | 高风险 | 高风险 |
| **中影响范围** | 低风险 | 中风险 | 高风险 |
| **小影响范围** | 低风险 | 低风险 | 中风险 |

### 8.2 风险等级判定

| 等级 | 判定标准 | 缓解措施要求 |
|---|---|---|
| 高 | 大范围 × 高复杂度 | 必须有详细回滚方案 + 分阶段实施 |
| 中 | 中范围或中复杂度 | 需有回滚方案 + 充分测试 |
| 低 | 小范围 × 低复杂度 | 标准验证即可 |

## 9. 检查清单

变更设计师在产出设计文档前自检：

- [ ] 每个变更项有变更类型分类
- [ ] 每个变更项有有序步骤（含依赖标注）
- [ ] 每个步骤映射到具体代码文件
- [ ] 向后兼容性分析覆盖受影响接口（AC-6）
- [ ] 每个变更项有回滚方案（AC-5）
- [ ] SDK_SPEC 演进需求已标注（AC-15）
- [ ] 风险评估有等级和缓解措施
- [ ] 变更范围不超出诊断报告

## 10. 示例

### 示例：修复 compiler.py 中 verdict 提取遗漏 when: 中的 max_executions 的问题

```
变更类型：bug 修复

变更步骤：
  步骤 1（修改 compiler.py）：
    文件：engine/scripts/compiler.py
    操作：修改 verdict 提取正则表达式，增加 max_executions 捕获组
    依赖：无
    
  步骤 2（修改 compiler.py）：
    文件：engine/scripts/compiler.py
    操作：在 edge 构建逻辑中写入 max_executions 字段
    依赖：步骤 1

影响接口：
  compiler.py compile() 内部逻辑变化
  兼容性判定：向后兼容（新增字段，旧代码忽略）

向后兼容性分析：
  - ROUTER.json 格式：新增 max_executions 字段，旧引擎读取时忽略
  - registry.json 格式：无变化
  - 数据格式兼容性：向后兼容

回滚方案：
  回退 compiler.py 到修改前版本
  回滚后验证：编译现有 APP，确认 max_executions 行为恢复原状

SDK_SPEC 演进需求：无需演进（编译产物新增字段，不破坏现有格式）

风险评估：低风险（小范围 × 低复杂度）
```
