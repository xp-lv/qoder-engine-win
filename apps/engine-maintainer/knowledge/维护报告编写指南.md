# 维护报告编写指南 — 标准结构与追溯链构建

> **本文档用途**：支撑知识管理者产出人类可读维护报告（AC-14/AC-15/AC-16）。定义维护报告的标准结构（7 章节）、变更追溯链构建方法、SDK_SPEC 演进需求标注规范、findings 整合方法。

---

## 1. 目的

本文档定义维护报告的标准编写方法。知识管理者按本指南的 7 章节结构编写维护报告，确保报告完整、可追溯、可作为 backward carries 载体传递审阅详情。

## 2. 适用角色

- **知识管理者**：本指南是知识管理者编写维护报告的核心依据

## 3. 维护报告 7 章节结构模板（AC-16）

### 3.1 总体结构

维护报告必须包含以下 7 个章节，缺一不可：

```
① 变更摘要
② 影响分析
③ 验证结果
④ 已知风险
⑤ 回滚指南
⑥ SDK_SPEC 演进需求
⑦ 变更追踪表
```

### 3.2 章节 ①：变更摘要

**内容要求**：列出本次维护的每个变更项。

**模板**：
```markdown
## 变更摘要

| 变更项 ID | 文件路径 | 修改类型 | 修改内容 |
|---|---|---|---|
| CHG-001 | engine/scripts/compiler.py | 修改 | 修复 verdict 提取正则 |
| CHG-002 | engine/scripts/router.py | 修改 | 新增 edge_counts 检查 |

**变更统计**：共 N 项变更，涉及 M 个文件
```

**数据来源**：变更摘要（代码实施者产出）

### 3.3 章节 ②：影响分析

**内容要求**：列出受影响组件和影响范围。

**模板**：
```markdown
## 影响分析

### 受影响组件
| 组件 | 文件 | 影响类型 | 影响描述 |
|---|---|---|---|
| compile() | compiler.py | 直接修改 | verdict 提取逻辑变更 |
| 路由决策 | router.py | 间接影响 | max_executions 检查行为变化 |

### 影响范围
- 直接影响：N 个文件
- 间接影响：M 个文件
- 接口变更：K 个 API
```

**数据来源**：引擎诊断报告 + 变更设计文档

### 3.4 章节 ③：验证结果

**内容要求**：汇总回归验证报告和对抗审阅报告的验证结论。

**模板**：
```markdown
## 验证结果

### 回归验证
| 验证维度 | 结果 | 详情 |
|---|---|---|
| 编译正确性 | PASS | 编译链路无报错 |
| DAG 可达性 | PASS | 所有角色可达 |
| 行为兼容性 | PASS | 无不兼容行为 |
| Gate 兼容性 | PASS | 产出物格式通过 |

### 对抗审阅
| 对抗维度 | 结果 | 严重级别 | 详情 |
|---|---|---|---|
| 边界条件 | PASS | — | 边界测试全部通过 |
| 向后兼容性 | PASS | — | 现有 APP 正常运行 |
```

**数据来源**：回归验证报告 + 对抗审阅报告

### 3.5 章节 ④：已知风险

**内容要求**：列出残余风险和缓解措施。

**模板**：
```markdown
## 已知风险

| 风险项 | 严重级别 | 缓解措施 |
|---|---|---|
| 大量 APP 同时编译时性能下降 | minor | 可后续优化为增量编译 |
```

**数据来源**：变更设计文档的风险评估 + 对抗审阅报告的 findings

### 3.6 章节 ⑤：回滚指南

**内容要求**：为每个变更项提供回滚步骤。

**模板**：
```markdown
## 回滚指南

### CHG-001 回滚步骤
1. git revert <commit-hash>
2. 验证编译：python engine/scripts/compiler.py --check
3. 验证 DAG：确认所有角色可达

### CHG-002 回滚步骤
1. 恢复 router.py 原始版本
2. 验证路由：确认 verdict 边正确
```

**数据来源**：变更设计文档的回滚方案

### 3.7 章节 ⑥：SDK_SPEC 演进需求（AC-15）

**内容要求**：标注代码变更是否涉及引擎行为变化。

**模板（有行为变化）**：
```markdown
## SDK_SPEC 演进需求

| 变更类型 | 影响接口 | 建议演进内容 | 优先级 |
|---|---|---|---|
| 接口签名变化 | compiler.py compile() | 新增 type 参数 | 高 |

衔接：由 sdk-maintainer APP 后续处理
```

**模板（无行为变化）**：
```markdown
## SDK_SPEC 演进需求

无需演进（本次变更仅修复内部实现，不涉及引擎行为变化）
```

**数据来源**：变更设计文档的 SDK_SPEC 演进需求标注

### 3.8 章节 ⑦：变更追踪表（AC-14）

**内容要求**：构建从需求到报告的完整追溯链。

**模板**：
```markdown
## 变更追踪表

| 需求 ID | 诊断条目 | 设计变更项 | 代码变更 | 验证结论 | 对抗 finding | 报告章节 |
|---|---|---|---|---|---|---|
| AC-2 | compiler.py verdict 提取 | CHG-001 | compiler.py L234 | PASS | 无 | ③验证结果 |
| AC-10 | router.py edge_counts | CHG-002 | router.py L156 | PASS | 无 | ③验证结果 |
```

## 4. 变更追溯链构建方法（AC-14）

### 4.1 追溯链结构

```
维护需求 ID
  → 诊断报告条目（受影响组件 + 根因）
    → 设计文档变更项（变更步骤 + 影响接口）
      → 代码变更 diff（文件路径 + 修改内容）
        → 验证报告结论（编译/DAG/行为/Gate）
          → 对抗报告 finding（6 维对抗结果）
            → 维护报告章节（对应章节编号）
```

### 4.2 构建步骤

1. 从需求文档提取验收标准 ID 列表
2. 对每个 ID，在诊断报告中找到对应的受影响组件
3. 对每个受影响组件，在设计文档中找到对应的变更项
4. 对每个变更项，在变更摘要中找到对应的代码变更
5. 对每个代码变更，在验证报告中找到对应的验证结论
6. 对每个验证结论，在对抗报告中找到对应的 finding
7. 对每个 finding，在维护报告中标注对应的章节

### 4.3 追溯链完整性判据

- 追溯链不得断裂（任一环节缺失对应条目 → 不通过）
- 每个变更项必须有完整的追溯链
- 验收标准中的每条标准都能在追溯链中找到对应

## 5. findings 整合方法

### 5.1 两路 findings 来源

| 来源 | 内容 | 角色 |
|---|---|---|
| 回归验证报告 | 编译/DAG/行为/Gate 验证结论 | 回归验证者 |
| 对抗审阅报告 | 6 维对抗 findings | 架构红队 |

### 5.2 findings 整合到维护报告

| findings 类型 | 整合到章节 | 整合方式 |
|---|---|---|
| critical findings | ④已知风险 + verdict 裁决 | 触发 loop/major_rework |
| major findings | ④已知风险 | 标注残余风险 |
| minor findings | ④已知风险 | 标注残余风险 |
| info findings | ③验证结果 | 记录为信息项 |

### 5.3 严重级别在报告中的呈现

- **critical**：报告中用 ⚠️ 标注，必须触发回退
- **major**：报告中用 ● 标注，建议修复
- **minor**：报告中用 ○ 标注，记录备案
- **info**：报告中正常文字，信息记录

## 6. 维护报告作为 backward carries 载体

### 6.1 要求

维护报告必须包含完整的验证/对抗 findings 摘要，使得：
- 代码实施者在 loop 回退时能从维护报告获取质量问题详情
- 变更设计师在 major_rework 回退时能从维护报告获取结构问题详情

### 6.2 findings 摘要要求

- 回归验证报告的所有 critical/major findings 必须在维护报告 ④已知风险 章节中摘要呈现
- 对抗审阅报告的所有 critical findings 必须在维护报告 ④已知风险 章节中摘要呈现
- 每个 finding 摘要包含：维度、严重级别、问题描述、修复建议

## 7. 检查清单

知识管理者在产出维护报告前自检：

- [ ] 7 个章节全部齐全（AC-16）
- [ ] 变更摘要中每个变更项有文件路径和修改内容
- [ ] 影响分析覆盖受影响组件和影响范围
- [ ] 验证结果包含回归验证 + 对抗审阅两路结论
- [ ] 已知风险包含残余风险和缓解措施
- [ ] 回滚指南为每个变更项提供回滚步骤
- [ ] SDK_SPEC 演进需求已标注（AC-15）
- [ ] 变更追踪表形成完整追溯链（AC-14）
- [ ] 验证/对抗 findings 已整合到报告
- [ ] 维护报告包含完整的 findings 摘要（作为 carries 载体）
